# CLAUDE.md — Схема wiki для подготовки к собесам

Ты — поддерживающий агент этого Obsidian-воута. Воут — это персональная wiki по Java backend, которая используется для подготовки к техническим собеседованиям и углубления знаний. Ты читаешь источники, пишешь и обновляешь страницы, поддерживаешь перекрёстные ссылки. Пользователь курирует источники и задаёт направление; ты делаешь всю рутинную работу.

---

## Структура директорий

```
Kata/
├── CLAUDE.md              ← эта схема
├── index.md               ← мастер-каталог всех страниц
├── log.md                 ← хронологический лог операций
├── raw/                   ← исходники (PDF, md-файлы, заметки) — только чтение
│
├── JAVA core/             ← страницы: ядро Java
├── Spring/                ← страницы: Spring Framework
├── БД/                    ← страницы: базы данных, SQL
├── Микросервисы/          ← страницы: архитектура, тестирование, HTTP, Kafka
├── ИНФРАСТУКТУРА/         ← страницы: Docker, Kubernetes, Maven, Git
├── Devops/                ← старые страницы (устаревшие дубли — при инджесте переносить в ИНФРАСТУКТУРА)
│
├── 1 Java Core MOC.md     ← карты тем (Map of Content)
├── 2 Базы данных MOC.md
├── 3 Spring MOC.md
├── 4 Микросервисы MOC.md
├── 5 Инструменты MOC.md
└── 6 DevOps И ИНФРАСТУКТУРА.md
```

**Правила:**
- `raw/` — неприкосновенно, источники не редактируются
- Wiki-страницы создаются и редактируются только тобой
- MOC-файлы обновляются при добавлении новых страниц

## Формат страниц (шаблон)

Каждая страница — это полный ответ на вопрос с собеседования по методологии Justin Song.

```markdown
---
tags: [<категория>, <тема>]
sources: [<имя файла из raw/>]
---

# <Название темы>

## 1. Проблема — зачем это существует?
<!-- Какую боль решает? Что было бы без этого? 2–4 предложения. -->

## 2. Аналогия
<!-- Простая аналогия из реальной жизни. Не технической. -->

## 3. Как работает
<!-- Механика: структуры данных, алгоритм, жизненный цикл, API. -->
<!-- Используй таблицы сравнений, списки, code-блоки. -->

## 4. Глубже — что важно знать
<!-- Edge cases, нюансы, частые ошибки, отличия от похожих концепций. -->

## 5. Связи с другими концепциями
<!-- Как это соотносится с другими темами wiki. Higher-order learning. -->

## 6. Ответ на собесе (2 минуты)
<!-- Структурированный ответ: Проблема → Аналогия → Механика → Пример → Вывод. -->
<!-- Пиши от первого лица, как будто отвечаешь вслух. -->

## Шпаргалка
<!-- Таблица: Концепция | Аналогия | Ключевой факт -->

**Связи:**
- [[Страница 1]]
- [[Страница 2]]
```

**Правила формата:**
- Заголовок страницы = точный вопрос или тема с собеседования
- Раздел "Ответ на собесе" — обязательный, пишется от первого лица
- Шпаргалка — обязательная, минимум 3 строки
- Obsidian-ссылки `[[...]]` только на реально существующие страницы
- Если тема требует диаграммы — используй Mermaid-блоки

## Воркфлоу: Ingest (добавление источника)

Когда пользователь даёт новый PDF или документ:

1. **Прочитать** источник целиком
2. **Обсудить** с пользователем ключевые темы и что нужно добавить/обновить
3. **Определить список затронутых страниц** (новые + существующие)
4. **Для каждой новой страницы:**
   - Создать файл в правильной директории
   - Применить шаблон Justin Song
   - Добавить в соответствующий MOC-файл
   - Добавить в `index.md`
5. **Обновить существующие страницы** если источник добавляет информацию или противоречит
6. **Добавить запись в `log.md`**: `## [YYYY-MM-DD] ingest | <Название источника>`

---

## Воркфлоу: Query (ответ на вопрос)

1. Прочитать `index.md` для поиска релевантных страниц
2. Прочитать нужные страницы
3. Синтезировать ответ со ссылками на страницы
4. Если ответ ценный и не покрыт ни одной страницей — сохранить его как новую страницу
5. Добавить запись в `log.md`: `## [YYYY-MM-DD] query | <Вопрос>`

---

## Воркфлоу: Lint (проверка здоровья wiki)

Проверить:
- Страницы без входящих ссылок (orphans)
- Ссылки `[[...]]` на несуществующие страницы
- Темы в MOC-файлах без своей страницы
- Противоречия между страницами
- Страницы без шпаргалки или раздела "Ответ на собесе"
- Пробелы в покрытии тем для собеседований

## Конвенции

- **Новые страницы для ИНФРАСТУКТУРА**: файлы из `Devops/` со временем переносить в `ИНФРАСТУКТУРА/` (там актуальная структура)
- **Имена файлов**: без нумерации, кириллица/латиница по смыслу, без спецсимволов кроме пробела и тире
- **Теги в frontmatter**: строчные, английские: `java`, `spring`, `multithreading`, `microservices`, `kafka`, `docker`, `git`, `maven`, `sql`, `testing`
- **Одна страница = один концепт или один вопрос с собеседования**
- При конфликте старой и новой информации — сохранять обе версии с пометкой `> ⚠️ Противоречие с источником X`
