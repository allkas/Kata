# Stack vs Heap память в JVM

## 1. Проблема — зачем это существует?

Объединить всю память программы в одну кучу — просто, но медленно и небезопасно: каждый поток мешал бы другим, а сборщик мусора работал бы над всем сразу. JVM разделяет память: Stack — быстрый личный буфер каждого потока для локального контекста вызовов, Heap — общее хранилище объектов с автоматическим управлением жизненным циклом.

## 2. Аналогия

**Stack** — блокнот на рабочем столе каждого сотрудника (потока). Записи ведутся стопкой: добавил новую задачу — положил лист сверху, завершил — убрал верхний лист. Стол быстро и автоматически очищается.

**Heap** — общий склад компании. Здесь хранятся крупные объекты, доступные разным сотрудникам. Уборщик (GC) периодически обходит склад и выбрасывает то, на что никто не ссылается.

## 3. Как работает

### Сравнительная таблица:

| Характеристика | Stack | Heap |
|----------------|-------|------|
| **Хранит** | Примитивы, ссылки, локальные переменные | Объекты (`new`) |
| **Область** | Per-thread (свой у каждого потока) | Общий для всех потоков |
| **Структура** | LIFO — фреймы вызовов | Young/Old Generation + Metaspace |
| **Скорость** | Очень быстрый (указатель + смещение) | Медленнее (поиск свободного места) |
| **Размер** | Маленький, фиксированный (-Xss, ~512KB–1MB) | Большой, настраиваемый (-Xms / -Xmx) |
| **Управление** | Автоматически (pop при выходе из метода) | GC |
| **Ошибка** | `StackOverflowError` | `OutOfMemoryError` |
| **Потокобезопасность** | Изолирован, безопасен | Разделяется → нужна синхронизация |

### Структура Stack Frame:

Каждый вызов метода создаёт **фрейм** в Stack с:
- Локальными переменными
- Параметрами метода
- Операндным стеком (промежуточные результаты вычислений)
- Ссылкой на Constant Pool (метаданные класса)
- Адресом возврата (куда вернуться после завершения)

```java
void a() {
    int x = 10;         // в Stack Frame для a()
    b(x);
}
void b(int param) {
    String s = "hi";    // ссылка в Stack, объект "hi" в Heap (String Pool)
    Object obj = new Object(); // ссылка в Stack, объект в Heap
}
// При выходе из b() — фрейм b() удаляется, obj и s недостижимы → GC
```

```
Stack (Thread 1):
┌─────────────────┐ ← вершина
│ Frame: b()       │ param=10, s→Heap, obj→Heap
│ Frame: a()       │ x=10
│ Frame: main()    │ ...
└─────────────────┘

Heap:
┌─────────────────────────────┐
│ String "hi" (String Pool)   │ ←─ ссылка из b().s
│ Object instance             │ ←─ ссылка из b().obj
│ ...                         │
└─────────────────────────────┘
```

### Ошибки:

```java
// StackOverflowError — фреймы переполняют Stack (бесконечная рекурсия)
void infinite() {
    infinite(); // каждый вызов добавляет фрейм
}               // java.lang.StackOverflowError

// OutOfMemoryError — Heap переполнен
List<byte[]> leak = new ArrayList<>();
while (true) {
    leak.add(new byte[1024 * 1024]); // java.lang.OutOfMemoryError: Java heap space
}

// OutOfMemoryError: Metaspace — динамически генерируется слишком много классов
// (ClassLoader'ы, библиотеки проксирования)
```

### Где что хранится:

```java
class Example {
    static int STATIC_INT = 5;         // значение в Metaspace (static-поле)
    int instanceInt = 10;              // в объекте на Heap

    void method() {
        int localInt = 20;             // в Stack (примитив)
        String localStr = "hello";     // ссылка в Stack, объект в Heap/String Pool
        Object obj = new Object();     // ссылка в Stack, объект в Heap
        localStr = null;               // ссылка обнулена; объект → кандидат для GC
    }
}
```

## 4. Глубже — нюансы

### Escape Analysis (JIT-оптимизация):

JIT-компилятор анализирует: «выходит ли объект за пределы метода?» Если нет — выделяет его **на Stack** вместо Heap. Это называется **stack allocation** или **scalar replacement**.

```java
void calculate() {
    Point p = new Point(1, 2); // p не выходит за пределы метода
    return p.x + p.y;          // JIT может разместить p на Stack
}
```

### ThreadLocal — изоляция данных на Heap:

`ThreadLocal` хранит данные на Heap, но логически привязывает к потоку — другие потоки не видят значение.

```java
ThreadLocal<SimpleDateFormat> formatter =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("dd.MM.yyyy"));
// Каждый поток имеет свой экземпляр SimpleDateFormat
```

### Настройка памяти JVM:

```
-Xss512k         — размер Stack на поток
-Xms256m         — начальный размер Heap
-Xmx1g           — максимальный размер Heap
-XX:MaxMetaspaceSize=256m — ограничить Metaspace
```

## 5. Связи с другими концепциями

- [[Garbage Collector JVM]] — GC управляет Heap; Stack очищается автоматически
- [[Многопоточность основы]] — каждый поток имеет свой Stack; Heap разделяется → race conditions
- [[JVM устройство ClassLoader]] — Metaspace хранит метаданные классов, загруженных ClassLoader

## 6. Ответ на собесе (2 минуты)

> "JVM делит память на несколько областей. **Stack** — персональный для каждого потока, хранит фреймы вызовов: локальные переменные, параметры метода, ссылки на объекты. При выходе из метода фрейм автоматически удаляется — GC не нужен. Переполнение Stack — `StackOverflowError`, обычно от бесконечной рекурсии.
>
> **Heap** — общее хранилище для всех объектов, созданных через `new`. Разделяется между потоками, поэтому нужна синхронизация. Им управляет GC: молодые объекты в Eden, выжившие уходят в Old Generation. `OutOfMemoryError: Java heap space` — Heap переполнен.
>
> **Metaspace** (с Java 8 вместо PermGen) хранит метаданные классов: байткод, методы, константы. Растёт динамически в native memory. `OutOfMemoryError: Metaspace` — слишком много загруженных классов (типично при интенсивном использовании reflection/proxy).
>
> **Ключевой момент:** ссылка хранится на Stack, объект — на Heap. Примитив в локальной переменной — на Stack, примитив внутри объекта — на Heap вместе с объектом."

## Шпаргалка

| Область | Что хранит | Очистка | Ошибка |
|---------|-----------|---------|--------|
| **Stack** | Локальные переменные, ссылки, фреймы | Автоматически (pop) | `StackOverflowError` |
| **Heap (Young)** | Новые объекты (Eden + Survivor) | Minor GC | — |
| **Heap (Old)** | Долгоживущие объекты | Major/Full GC | `OutOfMemoryError: Java heap space` |
| **Metaspace** | Метаданные классов | GC при unload | `OutOfMemoryError: Metaspace` |
| **Code Cache** | JIT-скомпилированный нативный код | JVM управляет | — |
