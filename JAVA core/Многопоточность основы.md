# Многопоточность основы

## 1. Проблема — зачем это существует?

Современные процессоры многоядерные. Однопоточная программа использует одно ядро — остальные простаивают. Многопоточность позволяет выполнять задачи параллельно и не блокировать поток на ожидании (IO, сеть, БД). Цена — **разделяемое изменяемое состояние**: когда несколько потоков читают и пишут одни данные без координации, результат непредсказуем.

## 2. Аналогия

Представь офис с общим документом на столе. Один сотрудник (поток) может работать с ним без проблем. Двое одновременно — и данные начнут затирать друг друга. Решения:
- **volatile** — говорит всем «смотрите на оригинал, не на свою копию»
- **synchronized** — «один работает, остальные ждут за дверью»
- **Atomic** — «специальный стол, замена страницы происходит мгновенно и неразрывно»

---

## 3. volatile

### Что гарантирует:
1. **Видимость (visibility):** запись в volatile-переменную сразу видна всем потокам. Без volatile каждый поток может иметь копию в CPU-кэше и не видеть изменений других.
2. **Happens-before:** запись в volatile *happens-before* любого последующего чтения этой переменной другим потоком. Упорядочивание инструкций не пересекает барьер.

### Что НЕ гарантирует:
- **Атомарность составных операций.** `i++` — это три шага: read → increment → write. volatile не делает их неразрывными.

```java
volatile int counter = 0;

// Поток A и Поток B выполняют одновременно:
counter++; // read(0) → +1 → write(1) — оба могут прочитать 0 и записать 1
// Итог: 1 вместо 2 — ГОНКА, несмотря на volatile
```

### Когда применять volatile:
```java
// Паттерн «флаг остановки» — классическое применение
volatile boolean running = true;

// Поток A:
while (running) { doWork(); }

// Поток B:
running = false; // сразу видно потоку A без volatile не гарантировано
```

**Правило:** volatile подходит если операция — **простое чтение или запись** одного поля, не составная.

## 4. volatile vs synchronized

| | `volatile` | `synchronized` |
|---|---|---|
| **Гарантирует** | Видимость | Видимость + атомарность |
| **Блокировка** | Нет | Да (mutex) |
| **Область** | Одно поле | Блок / метод |
| **Производительность** | Легче | Тяжелее |
| **Применение** | Флаги, одиночные чтения/записи | Составные операции, инварианты |

```java
// volatile: простой флаг — достаточно
private volatile boolean stopped = false;

// synchronized: составная операция — нужна блокировка
private int count = 0;
public synchronized void increment() { count++; } // read+write атомарно
```

**Double-checked locking — volatile обязателен:**
```java
private volatile Singleton instance; // без volatile — сломан в многопоточности

public Singleton getInstance() {
    if (instance == null) {
        synchronized (this) {
            if (instance == null) {
                instance = new Singleton(); // без volatile другой поток может увидеть неполностью сконструированный объект
            }
        }
    }
    return instance;
}
```

## 5. Deadlock, Livelock, Race Condition

### Race Condition (гонка):
Несколько потоков одновременно читают и пишут общие данные, результат зависит от порядка выполнения.
```java
// Классика: counter++ без синхронизации
// Поток A: read(5) → Поток B: read(5) → A: write(6) → B: write(6)
// Ожидали 7, получили 6
```
**Как избежать:** синхронизировать доступ (`synchronized`, `Lock`, `Atomic`-типы).

### Deadlock (взаимная блокировка):
Поток A держит ресурс 1, ждёт ресурс 2. Поток B держит ресурс 2, ждёт ресурс 1. Оба ждут вечно.
```java
// Классика:
synchronized(lockA) {
    synchronized(lockB) { ... } // Поток A
}
synchronized(lockB) {
    synchronized(lockA) { ... } // Поток B — DEADLOCK
}
```
**Как избежать:**
- Всегда захватывать блокировки в **одном и том же порядке**
- Использовать `tryLock(timeout)` из `java.util.concurrent.locks`
- Не держать блокировку дольше необходимого

### Livelock:
Потоки активны, меняют состояние, но прогресса нет — бесконечно реагируют друг на друга.
```
Два человека в узком коридоре: оба шагают вправо, видят друг друга, оба шагают влево — повторяется бесконечно.
```
**Как избежать:** добавить случайную задержку или механизм приоритета.

## 6. Атомарные типы (java.util.concurrent.atomic)

### Как работает CAS:
CAS (Compare-And-Swap) — одна аппаратная инструкция: «если текущее значение == expected, установить new; иначе — не менять». Не нужна блокировка — **оптимистичный подход**.

```java
AtomicInteger counter = new AtomicInteger(0);

// Эквивалент synchronized counter++, но без блокировки:
counter.incrementAndGet(); // потокобезопасно

// Явный CAS:
int current = counter.get();
boolean success = counter.compareAndSet(current, current + 1);
// Если другой поток изменил значение — success=false, нужно повторить
```

### Основные классы:
| Класс | Для чего |
|-------|----------|
| `AtomicInteger` / `AtomicLong` | Счётчики, ID-генераторы |
| `AtomicBoolean` | Флаги (альтернатива volatile+synchronized) |
| `AtomicReference<T>` | Атомарная замена объектов |
| `LongAdder` | Высококонкурентные счётчики (быстрее AtomicLong под нагрузкой) |

### Atomic vs synchronized:
| | Atomic (CAS) | synchronized |
|---|---|---|
| **При низкой конкуренции** | Быстрее | Медленнее |
| **При высокой конкуренции** | Много retry, CPU spinning | Стабильнее |
| **Сложность** | Проще | Сложнее |
| **Применение** | Счётчики, одна переменная | Сложная логика, инварианты |

## 7. Runnable vs Callable

```java
// Runnable — нет результата, нет checked exception
Runnable r = () -> System.out.println("done");
new Thread(r).start();

// Callable<T> — возвращает результат, может бросать checked
Callable<Integer> c = () -> compute(); // compute() бросает Exception
ExecutorService executor = Executors.newFixedThreadPool(4);
Future<Integer> future = executor.submit(c);
Integer result = future.get(); // блокирует до результата
```

| | `Runnable` | `Callable<T>` |
|---|---|---|
| Метод | `void run()` | `T call() throws Exception` |
| Результат | Нет | `T` (через `Future<T>`) |
| Исключения | Только unchecked | Checked + unchecked |
| Используется с | `Thread`, `ExecutorService` | `ExecutorService` |

**`Future<T>`** — контейнер для отложенного результата:
```java
future.get();              // ждёт результат (блокирует)
future.get(5, SECONDS);    // с таймаутом
future.isDone();           // готов ли результат
future.cancel(true);       // отменить задачу
```

## 8. ThreadLocal

### Как работает:
Каждый поток имеет свою копию переменной. Хранится в `Thread.threadLocalMap` — при завершении потока автоматически очищается.

```java
ThreadLocal<SimpleDateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

// В каждом потоке — своя копия SimpleDateFormat (не потокобезопасен сам по себе)
String formatted = dateFormat.get().format(new Date());
```

### Реальные применения:
- **`SecurityContext`** в Spring Security — пользователь текущего запроса
- **Транзакционный `Connection`** — один connection на поток
- **`SimpleDateFormat`** — не потокобезопасен, ThreadLocal даёт по копии на поток
- **Трассировка (MDC)** — correlation ID в логах

### Важно: утечка памяти в пуле потоков:
```java
// В ThreadPoolExecutor потоки переиспользуются!
threadLocal.set(value);
// ... обработка запроса ...
threadLocal.remove(); // ОБЯЗАТЕЛЬНО! Иначе следующий запрос в этом потоке увидит старое значение
```
Если не вызвать `remove()` — старое значение живёт в потоке бесконечно → утечка памяти и баги.

## 10. Ответ на собесе (2 минуты)

> "Многопоточность — большая тема, попробую структурировано.
>
> **volatile** гарантирует видимость: запись сразу видна всем потокам, без кэширования в регистрах. Но НЕ гарантирует атомарность — `i++` под volatile всё равно не потокобезопасен, потому что это три операции: read, increment, write. Использую volatile для простых флагов и double-checked locking.
>
> **synchronized** даёт и видимость, и атомарность, и взаимное исключение. Дороже volatile — блокирует поток. Для составных операций и инвариантов, которые охватывают несколько переменных.
>
> **Atomic-типы** — оптимистичный подход через CAS (Compare-And-Swap). Не блокируют, но при высокой конкуренции много retry. Для счётчиков и одиночных переменных — быстрее synchronized. Под высокой нагрузкой — `LongAdder`.
>
> **Deadlock** — классика: два потока держат блокировки и ждут друг друга. Избегаю захватом блокировок в одном порядке или `tryLock` с таймаутом.
>
> **ThreadLocal** — каждый поток получает свою копию переменной. В Spring Security так хранится SecurityContext. Главная ловушка: в пуле потоков после обработки обязательно вызывать `remove()`, иначе старые данные живут в следующем запросе."

## Шпаргалка

| Концепция | Гарантия | Когда |
|-----------|----------|-------|
| **volatile** | Видимость | Простые флаги, double-checked locking |
| **synchronized** | Видимость + атомарность | Составные операции, инварианты |
| **AtomicInteger** | Атомарность без блокировки (CAS) | Счётчики, низкая/средняя конкуренция |
| **LongAdder** | Атомарность, лучше под нагрузкой | Высококонкурентные счётчики |
| **Race condition** | — | Нет синхронизации → непредсказуемый результат |
| **Deadlock** | — | Круговое ожидание блокировок |
| **Livelock** | — | Активны, но без прогресса |
| **ThreadLocal** | Изоляция по потоку | SecurityContext, Connection, MDC |
| **Callable vs Runnable** | Callable возвращает результат | Future<T> для асинхронных задач |
