# Основы ООП

## 1. Проблема — зачем это существует?

До ООП был процедурный подход: программа — набор функций, работающих с глобальными данными. При росте кода любое изменение разрасталось в цепочку поломок: данные доступны всем, логика размазана по функциям. ООП решает это — **группирует данные и поведение в объекты** с чёткими границами ответственности.

## 2. Аналогия

Программа — это предприятие. Каждый отдел (объект) хранит свои данные и умеет делать своё дело. Бухгалтерия не лезет в базу кадров — это инкапсуляция. Новый отдел наследует процессы существующего — это наследование. Директор даёт одну команду «подготовить отчёт» разным отделам, и каждый делает её по-своему — это полиморфизм.

## 3. Четыре принципа

### Инкапсуляция
Скрытие внутреннего состояния. Доступ — только через публичный интерфейс с валидацией.
```java
class BankAccount {
    private double balance; // скрыто снаружи

    public void deposit(double amount) {
        if (amount > 0) balance += amount; // валидация внутри
    }

    public double getBalance() { return balance; }
}
```

### Наследование
Дочерний класс получает поля и методы родителя, расширяет или переопределяет их.
```java
class Animal {
    void speak() { System.out.println("..."); }
}
class Dog extends Animal {
    @Override void speak() { System.out.println("Woof"); }
}
```

### Полиморфизм
Один и тот же вызов — разное поведение в зависимости от **реального типа** объекта.

**Перегрузка (Overload) — compile-time полиморфизм:**
- Методы с одинаковым именем, но разными параметрами
- Компилятор выбирает нужный вариант по типам аргументов **на этапе компиляции**
```java
void print(int x) { ... }
void print(String s) { ... }
```

**Переопределение (Override) — runtime полиморфизм:**
- Дочерний класс меняет реализацию метода родителя
- JVM выбирает нужный вариант по реальному типу объекта **во время выполнения**
- Требует: `@Override`, те же параметры, возвращаемый тип — тот же или ковариантный
```java
Animal a = new Dog();
a.speak(); // "Woof" — вызов по реальному типу (Dog), не по типу ссылки (Animal)
```

### Абстракция
Скрытие деталей реализации за публичным интерфейсом. Клиент знает «что делает», не «как».
```java
Collections.sort(list, comparator); // не знаем алгоритм, знаем контракт
```

## 4. Глубже — статические методы: override vs hide

**Статические методы НЕ переопределяются — они скрываются (method hiding).**

```java
class Parent {
    static void greet() { System.out.println("Parent"); }
    void hello() { System.out.println("Parent hello"); }
}
class Child extends Parent {
    static void greet() { System.out.println("Child"); } // HIDING — не override
    @Override void hello() { System.out.println("Child hello"); } // настоящий override
}

Parent p = new Child();
p.greet();  // "Parent" — по типу ССЫЛКИ (compile-time)
p.hello();  // "Child hello" — по типу ОБЪЕКТА (runtime)
```

**Почему:** переопределение работает через vtable (таблицу виртуальных методов). Статические методы принадлежат классу, не объекту — у них нет vtable-записи, JVM вызывает их по типу ссылки.

## 6. Ответ на собесе (2 минуты)

> "ООП — способ организовать код через объекты, которые объединяют данные и поведение. Четыре принципа — не абстракция, каждый решает реальную проблему.
>
> **Инкапсуляция** защищает состояние. Поле `balance` приватное — изменить его можно только через `deposit()` с валидацией. Иначе любой код мог бы записать `account.balance = -1000`.
>
> **Наследование** убирает дублирование. `Dog` и `Cat` наследуют `Animal` — общая логика в одном месте. Но я стараюсь не злоупотреблять: глубокие иерархии хрупкие, в сложных случаях предпочитаю композицию.
>
> **Полиморфизм** — самый мощный принцип. Пишу `List<Animal> animals` и вызываю `animal.speak()` — JVM сама разбирается, чей `speak()` вызвать. Это позволяет добавлять новых животных без изменения клиентского кода.
>
> **Абстракция** скрывает детали. `Collections.sort()` принимает `Comparator` — меня не волнует алгоритм сортировки, я задаю только критерий.
>
> Важный нюанс про статические методы: они **не переопределяются**, а скрываются. `Parent p = new Child(); p.greet()` вызовет `Parent.greet()` — по типу ссылки, не объекта."

## Шпаргалка

| Принцип | Суть | Практика |
|---------|------|---------|
| **Инкапсуляция** | Скрытие состояния | `private` поля + геттеры/сеттеры |
| **Наследование** | Переиспользование кода | `extends`, один родитель |
| **Полиморфизм** | Разное поведение — один вызов | `animal.speak()` → Dog/Cat |
| **Абстракция** | Скрыть детали | Интерфейс / абстрактный класс |
| **Overload** | Compile-time, по параметрам | `add(int)` vs `add(double)` |
| **Override** | Runtime, по типу объекта | `@Override speak()` |
| **Hide (static)** | По типу ссылки | `Parent p = new Child(); p.static()` → Parent |
