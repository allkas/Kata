# Паттерны проектирования (Design Patterns)

## 1. Проблема — зачем это существует?

При разработке программ одни и те же архитектурные задачи возникают снова и снова: как создать объект без жёсткой связи с конкретным классом, как добавить поведение не меняя исходный класс, как организовать взаимодействие множества объектов. Без шаблонных решений каждая команда изобретает велосипед. Паттерны GoF (Gang of Four, книга 1994 года) — 23 проверенных решения типовых задач ООП, которые стали общим языком разработчиков.

## 2. Аналогия

Паттерны — как архитектурные чертежи в строительстве. Чертёж «стандартная дверная коробка» не описывает конкретную дверь — но любой строитель понимает, что имеется в виду, и не изобретает крепления каждый раз заново. Паттерн — это шаблон решения задачи, а не готовый код.

## 3. Как работает

GoF делит 23 паттерна на 3 группы:

| Группа | Задача | Ключевые паттерны |
|--------|--------|-------------------|
| **Creational** (порождающие) | Как создавать объекты | Singleton, Builder, Factory Method, Abstract Factory, Prototype |
| **Structural** (структурные) | Как компоновать объекты | Decorator, Proxy, Facade, Adapter, Composite |
| **Behavioral** (поведенческие) | Как объекты взаимодействуют | Strategy, Observer, Template Method, Command, Iterator |

### Singleton — один экземпляр на всё приложение

**Проблема:** нужен ровно один объект (конфигурация, пул соединений, логгер).

```java
// Thread-safe Singleton через double-checked locking
public class AppConfig {
    private static volatile AppConfig instance; // volatile обязателен!

    private AppConfig() {}

    public static AppConfig getInstance() {
        if (instance == null) {                   // 1-я проверка — без lock (быстрый путь)
            synchronized (AppConfig.class) {
                if (instance == null) {           // 2-я проверка — с lock (безопасный путь)
                    instance = new AppConfig();
                }
            }
        }
        return instance;
    }
}

// Лучший вариант — enum Singleton (thread-safe «из коробки», защищён от reflection и десериализации)
public enum DatabasePool {
    INSTANCE;
    public Connection getConnection() { ... }
}
// Использование: DatabasePool.INSTANCE.getConnection()
```

**Почему `volatile` обязателен?** Без него JVM может опубликовать частично инициализированный объект из-за переупорядочивания инструкций. `volatile` запрещает это. Подробнее — [[Многопоточность основы]].

**В Spring:** все бины по умолчанию `scope=singleton` в рамках контекста. Это не GoF Singleton — Spring управляет жизненным циклом через контейнер, а не через статическое поле.

### Builder — пошаговое создание сложного объекта

**Проблема:** объект с 5+ параметрами, часть опциональна. Telescoping constructor anti-pattern: `new User("Alice", "alice@mail.com", null, 25, true, false)` — невозможно понять, что есть что.

```java
// Builder pattern — читаемо и безопасно
User user = User.builder()
    .name("Alice")
    .email("alice@mail.com")
    .age(25)
    .active(true)
    .build();

// Реализация (Lombok @Builder генерирует это автоматически)
public class User {
    private final String name;
    private final String email;
    private final Integer age;
    private final boolean active;

    private User(Builder b) {
        this.name = b.name;
        this.email = b.email;
        this.age = b.age;
        this.active = b.active;
    }

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        private String name;
        private String email;
        private Integer age;
        private boolean active;

        public Builder name(String v)    { this.name = v;   return this; }
        public Builder email(String v)   { this.email = v;  return this; }
        public Builder age(Integer v)    { this.age = v;    return this; }
        public Builder active(boolean v) { this.active = v; return this; }
        public User build()              { return new User(this); }
    }
}
```

Builder использует static nested class — из него нельзя случайно обратиться к полям внешнего объекта до его создания. Подробнее — [[final static внутренние классы]].

### Factory Method — создание без знания конкретного класса

**Проблема:** клиент не должен знать, какой конкретный класс создаётся. Нужно расширять систему новыми типами без изменения клиентского кода (OCP из SOLID).

```java
// Абстрактный класс с factory method
public abstract class NotificationSender {
    // Factory Method — подкласс решает, что создать
    protected abstract Notification createNotification(String message);

    // Шаблонный метод использует factory method
    public void send(String message) {
        Notification n = createNotification(message);
        n.validate();
        n.deliver();
        n.logDelivery();
    }
}

public class EmailSender extends NotificationSender {
    @Override
    protected Notification createNotification(String message) {
        return new EmailNotification(message);
    }
}

public class SmsSender extends NotificationSender {
    @Override
    protected Notification createNotification(String message) {
        return new SmsNotification(message);
    }
}
```

### Abstract Factory — семейства согласованных объектов

**Проблема:** нужно создавать семейства взаимосвязанных объектов, которые должны использоваться вместе (UI: тёмная/светлая тема — кнопки, поля, диалоги должны соответствовать друг другу).

```java
public interface UIFactory {
    Button createButton();
    TextField createTextField();
    Dialog createDialog();
}

public class DarkThemeFactory implements UIFactory {
    public Button createButton()       { return new DarkButton(); }
    public TextField createTextField() { return new DarkTextField(); }
    public Dialog createDialog()       { return new DarkDialog(); }
}

public class LightThemeFactory implements UIFactory {
    public Button createButton()       { return new LightButton(); }
    public TextField createTextField() { return new LightTextField(); }
    public Dialog createDialog()       { return new LightDialog(); }
}

// Клиентский код работает с UIFactory — не знает про Dark/Light
public class App {
    private final UIFactory factory;
    public App(UIFactory factory) { this.factory = factory; }
    public void render() {
        Button btn = factory.createButton(); // всегда согласованный стиль
        btn.render();
    }
}
```

**Factory Method vs Abstract Factory:**

| | Factory Method | Abstract Factory |
|---|---|---|
| **Создаёт** | Один тип продукта | Семейство связанных продуктов |
| **Расширение** | Новый подкласс переопределяет метод | Новая конкретная фабрика |
| **Связь объектов** | Не гарантирует | Гарантирует согласованность |
| **Применение** | Один вариант объекта | Несколько взаимозависимых объектов |

### Decorator — добавление поведения без изменения класса

**Проблема:** нужно динамически добавлять ответственность объекту, не меняя его класс. Наследование не подходит — N комбинаций поведений → N² подклассов.

```java
// Java I/O — классический пример декоратора (цепочка обёрток)
InputStream in = new BufferedInputStream(   // добавляет буферизацию
    new GZIPInputStream(                    // добавляет распаковку
        new FileInputStream("data.gz")      // базовый источник
    )
);

// Собственный пример
public interface TextProcessor {
    String process(String text);
}

public class TrimProcessor implements TextProcessor {
    public String process(String text) { return text.trim(); }
}

// Декоратор: реализует тот же интерфейс + держит ссылку на обёрнутый объект
public class UpperCaseDecorator implements TextProcessor {
    private final TextProcessor wrapped;
    public UpperCaseDecorator(TextProcessor wrapped) { this.wrapped = wrapped; }

    public String process(String text) {
        return wrapped.process(text).toUpperCase(); // делегируем + добавляем
    }
}

public class LoggingDecorator implements TextProcessor {
    private final TextProcessor wrapped;
    public LoggingDecorator(TextProcessor wrapped) { this.wrapped = wrapped; }

    public String process(String text) {
        System.out.println("Input: " + text);
        String result = wrapped.process(text);
        System.out.println("Output: " + result);
        return result;
    }
}

// Использование — комбинируем произвольно
TextProcessor p = new LoggingDecorator(new UpperCaseDecorator(new TrimProcessor()));
p.process("  hello  "); // → "HELLO"
```

### Proxy — контроль доступа к объекту

**Проблема:** нужно добавить логику вокруг вызовов (авторизацию, кэширование, логирование, транзакции) без изменения оригинального класса.

```java
// Spring AOP создаёт прокси автоматически:
// @Transactional  → прокси открывает/закрывает транзакцию
// @Cacheable      → прокси проверяет кэш перед вызовом
// @Async          → прокси запускает метод в отдельном потоке

// JDK Dynamic Proxy (только для классов, реализующих интерфейс)
UserService proxy = (UserService) Proxy.newProxyInstance(
    UserService.class.getClassLoader(),
    new Class[]{UserService.class},
    (proxyObj, method, args) -> {
        System.out.println("Before: " + method.getName());
        Object result = method.invoke(realService, args);
        System.out.println("After: " + method.getName());
        return result;
    }
);

// CGLIB Proxy (Spring использует для классов без интерфейса) — создаёт подкласс
```

**Decorator vs Proxy — структура одинакова, намерение разное:**

| | Decorator | Proxy |
|---|---|---|
| **Цель** | Добавить поведение | Контролировать доступ |
| **Создание** | Клиент явно оборачивает | Создаётся прозрачно (фреймворк) |
| **Знает ли клиент** | Да | Нет |
| **Пример** | `BufferedInputStream` | Spring `@Transactional` |

### Strategy — взаимозаменяемые алгоритмы

**Проблема:** поведение объекта нужно менять во время выполнения. Большой `if/switch` по типу алгоритма — нарушение OCP.

```java
@FunctionalInterface
public interface DiscountStrategy {
    double apply(double price);
}

// Стратегии — отдельные классы (или лямбды)
public class NoDiscount       implements DiscountStrategy { public double apply(double p) { return p; } }
public class PercentDiscount  implements DiscountStrategy {
    private final double pct;
    public PercentDiscount(double pct) { this.pct = pct; }
    public double apply(double p)      { return p * (1 - pct / 100); }
}

// Контекст — держит ссылку на стратегию
public class PricingService {
    private DiscountStrategy strategy = new NoDiscount(); // дефолт

    public void setStrategy(DiscountStrategy s) { this.strategy = s; }
    public double calculate(double price)        { return strategy.apply(price); }
}

// Использование
PricingService svc = new PricingService();
svc.setStrategy(new PercentDiscount(20));  // 20% скидка
svc.setStrategy(price -> price * 0.5);    // лямбда — тоже Strategy
```

### Adapter — адаптация несовместимого интерфейса

**Проблема:** есть готовый класс с нужной функциональностью, но его интерфейс несовместим с тем, что ожидает клиент. Переписывать готовый класс нельзя (чужая библиотека, legacy-код).

```java
// Устаревший платёжный сервис с несовместимым интерфейсом
public class LegacyPaymentSystem {
    public void makePayment(String cardNumber, int amountInCents) { ... }
}

// Интерфейс, который ожидает наш код
public interface PaymentProcessor {
    void pay(PaymentRequest request);
}

// Adapter — переводит вызовы из одного интерфейса в другой
public class LegacyPaymentAdapter implements PaymentProcessor {
    private final LegacyPaymentSystem legacy;

    public LegacyPaymentAdapter(LegacyPaymentSystem legacy) {
        this.legacy = legacy;
    }

    @Override
    public void pay(PaymentRequest request) {
        // адаптируем: конвертируем BigDecimal в центы, извлекаем номер карты
        int cents = request.getAmount().multiply(BigDecimal.valueOf(100)).intValue();
        legacy.makePayment(request.getCardNumber(), cents);
    }
}

// Клиент работает через PaymentProcessor — не знает про legacy
PaymentProcessor processor = new LegacyPaymentAdapter(new LegacyPaymentSystem());
processor.pay(new PaymentRequest("4111111111111111", new BigDecimal("99.99")));
```

**В Java:** `Arrays.asList()` — адаптер массива к `List`; `InputStreamReader` — адаптер `InputStream` к `Reader`.

### Facade — единая точка входа к сложной подсистеме

**Проблема:** подсистема состоит из множества классов с запутанными взаимосвязями. Клиентский код вынужден знать детали реализации и порядок вызовов. Нужно упростить интерфейс.

```java
// Сложная подсистема (много классов с зависимостями)
class VideoDecoder { ... }
class AudioMixer { ... }
class SubtitleRenderer { ... }
class BufferManager { ... }

// Facade — простой API, скрывающий сложность
public class VideoPlayerFacade {
    private final VideoDecoder decoder = new VideoDecoder();
    private final AudioMixer mixer = new AudioMixer();
    private final SubtitleRenderer subtitles = new SubtitleRenderer();
    private final BufferManager buffer = new BufferManager();

    public void play(String filePath) {
        buffer.allocate(1024);
        decoder.decode(filePath);
        mixer.setVolume(80);
        subtitles.load(filePath + ".srt");
        decoder.startPlayback();
    }

    public void stop() {
        decoder.stop();
        buffer.release();
    }
}

// Клиент — один простой вызов вместо 5 классов
VideoPlayerFacade player = new VideoPlayerFacade();
player.play("movie.mp4");
```

**В Spring:** `JdbcTemplate` — Facade над JDBC API (Connection, PreparedStatement, ResultSet, Exception handling). `RestTemplate` — Facade над HTTP-клиентом.

**Facade vs Adapter:** Facade упрощает сложный интерфейс; Adapter адаптирует несовместимый интерфейс.

### Template Method — алгоритм с переопределяемыми шагами

**Проблема:** несколько классов реализуют один и тот же алгоритм, отличаясь лишь в деталях отдельных шагов. Дублируется каркас алгоритма.

```java
// Абстрактный класс задаёт скелет алгоритма
public abstract class DataExporter {
    // Template Method — final, нельзя переопределить
    public final void export(List<?> data) {
        List<String> validated = validate(data);   // шаг 1
        String formatted = format(validated);       // шаг 2 — переопределяется
        save(formatted);                            // шаг 3 — переопределяется
        sendNotification();                         // шаг 4 — общий (hook, по умолчанию ничего)
    }

    private List<String> validate(List<?> data) {
        // общая валидация для всех форматов
        return data.stream().map(Object::toString).toList();
    }

    protected abstract String format(List<String> data);  // шаг алгоритма — задаётся подклассом
    protected abstract void save(String content);          // шаг алгоритма — задаётся подклассом

    protected void sendNotification() {}  // hook — подкласс может переопределить, может нет
}

public class CsvExporter extends DataExporter {
    @Override
    protected String format(List<String> data) { return String.join(",", data); }
    @Override
    protected void save(String content)         { Files.writeString(Path.of("output.csv"), content); }
}

public class JsonExporter extends DataExporter {
    @Override
    protected String format(List<String> data) { return new ObjectMapper().writeValueAsString(data); }
    @Override
    protected void save(String content)         { Files.writeString(Path.of("output.json"), content); }
}
```

**В Spring:** `JdbcTemplate` (`execute()`, `query()` — template методы с callback-хуками); `AbstractController`; `AbstractTransactionalJUnit4SpringContextTests`.

### Chain of Responsibility — цепочка обработчиков

**Проблема:** запрос должен пройти через несколько обработчиков (валидация, авторизация, логирование, ...), при этом каждый обработчик сам решает, обрабатывать ли запрос дальше.

```java
public abstract class RequestHandler {
    private RequestHandler next;

    public RequestHandler setNext(RequestHandler next) {
        this.next = next;
        return next;  // возвращаем следующий — fluent builder цепочки
    }

    public void handle(HttpRequest request) {
        if (next != null) next.handle(request);  // передаём дальше по умолчанию
    }
}

public class AuthHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        if (!request.hasToken()) {
            throw new UnauthorizedException();   // прерываем цепочку
        }
        super.handle(request);  // передаём дальше
    }
}

public class RateLimitHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        if (isLimitExceeded(request.getIp())) {
            throw new TooManyRequestsException();
        }
        super.handle(request);
    }
}

public class LoggingHandler extends RequestHandler {
    @Override
    public void handle(HttpRequest request) {
        log.info("Request: {}", request.getPath());
        super.handle(request);  // всегда передаём
    }
}

// Сборка цепочки
RequestHandler chain = new LoggingHandler();
chain.setNext(new AuthHandler())
     .setNext(new RateLimitHandler());
chain.handle(request);
```

**В Spring Security:** цепочка `SecurityFilterChain` (FilterChainProxy) — это Chain of Responsibility. Каждый `Filter` обрабатывает запрос и передаёт его через `chain.doFilter()`.

### Observer — уведомление подписчиков об изменениях

**Проблема:** объект изменился → нужно уведомить всех заинтересованных, не зная заранее, кто они и сколько их.

```java
// Spring ApplicationEventPublisher — Observer в Spring
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher publisher;

    public Order createOrder(OrderRequest request) {
        Order order = saveOrder(request);
        publisher.publishEvent(new OrderCreatedEvent(this, order)); // публикуем событие
        return order;
    }
}

// Подписчики — независимые компоненты, не знают друг о друге
@Component
public class EmailNotifier {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        sendEmail(event.getOrder().getCustomerEmail());
    }
}

@Component
public class InventoryService {
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        reserveItems(event.getOrder().getItems());
    }
}
```

## 4. Глубже — что важно знать

**Паттерны в Spring Framework:**

| Паттерн | Где используется в Spring |
|---------|--------------------------|
| Singleton | Все бины (`scope=singleton` по умолчанию) |
| Factory Method | `BeanFactory`, `ApplicationContext.getBean()` |
| Proxy | AOP — `@Transactional`, `@Cacheable`, `@Async` |
| Template Method | `JdbcTemplate`, `RestTemplate`, `AbstractController` |
| Observer | `ApplicationEventPublisher` / `@EventListener` |
| Decorator | `HttpServletRequestWrapper`, фильтры |
| Strategy | `HandlerMapping`, `MessageConverter`, `Comparator` |
| Chain of Responsibility | `SecurityFilterChain` (Spring Security) |
| Facade | `JdbcTemplate`, `RestTemplate` |
| Adapter | `HandlerAdapter`, `MessageConverter` |

**Частые ошибки на собесе:**
- Путать Decorator и Proxy — структура одинакова, но намерение разное
- Путать Adapter и Facade — Adapter адаптирует интерфейс, Facade упрощает
- Не знать, что Singleton в Spring ≠ GoF Singleton (Spring не использует статическое поле)
- Не объяснять, зачем `volatile` в double-checked locking
- Думать, что Factory Method и Abstract Factory — одно и то же
- Путать Template Method и Strategy — оба задают алгоритм, но Template Method через наследование, Strategy через делегирование

**Template Method vs Strategy:**

| | Template Method | Strategy |
|---|---|---|
| **Механизм** | Абстрактные методы в родителе | Делегирование в отдельный объект |
| **Смена алгоритма** | Только при наследовании | Во время выполнения (setStrategy) |
| **Связанность** | Сильная (наследование) | Слабая (через интерфейс) |
| **Когда** | Каркас алгоритма стабилен | Алгоритм меняется динамически |

**Adapter vs Facade vs Decorator vs Proxy — легко запутаться:**

| Паттерн | Зачем | Ключевое |
|---------|-------|----------|
| Adapter | Преобразовать интерфейс | Несовместимые интерфейсы |
| Facade | Упростить интерфейс | Скрыть сложность подсистемы |
| Decorator | Добавить поведение | Тот же интерфейс, расширяет функционал |
| Proxy | Контролировать доступ | Тот же интерфейс, не расширяет |

## 6. Ответ на собесе (2 минуты)

Паттерны проектирования — это проверенные шаблоны решения типовых задач ООП. Их 23, разбиты на три группы: порождающие, структурные и поведенческие.

Из порождающих чаще всего спрашивают Singleton и Builder. Singleton — ровно один экземпляр. Thread-safe реализация: double-checked locking с `volatile` полем. `volatile` обязателен — без него JVM может опубликовать частично инициализированный объект. Лучший вариант — enum Singleton, он потокобезопасен автоматически. Builder решает проблему telescoping constructor — вместо конструктора с 7 параметрами цепочка вызовов, которую легко читать. В реальности использую `@Builder` от Lombok.

Factory Method и Abstract Factory часто путают. Factory Method — один factory-метод в родительском классе, подкласс решает, что создать. Abstract Factory — целая фабрика для создания семейства связанных объектов, чтобы они всегда были согласованы.

Из поведенческих — Strategy: выносим алгоритм в интерфейс, в контекст подаём нужную реализацию через setter. Лямбды в Java — это Strategy по сути. Observer — Spring `@EventListener`, публикуем событие, подписчики реагируют независимо.

Decorator и Proxy структурно похожи — оба оборачивают объект. Разница в намерении: Decorator добавляет поведение, Proxy контролирует доступ. Spring `@Transactional` — это Proxy.

## Шпаргалка

| Паттерн | Группа | Аналогия | Ключевой факт |
|---------|--------|----------|---------------|
| Singleton | Creational | Один директор школы | `volatile` + DCL; лучший вариант — `enum` |
| Builder | Creational | Конструктор LEGO | `@Builder` (Lombok); static nested class |
| Factory Method | Creational | Штамп на заводе | Подкласс решает, что создать |
| Abstract Factory | Creational | Фабрика мебели одного стиля | Семейство объектов — всегда согласованы |
| Adapter | Structural | Переходник для розетки | `Arrays.asList()`, `InputStreamReader` |
| Facade | Structural | Стойка ресепшен | `JdbcTemplate`; скрывает сложность подсистемы |
| Decorator | Structural | Обёртка для подарка | Java I/O; добавляет поведение без наследования |
| Proxy | Structural | Секретарь директора | Spring AOP: `@Transactional`, `@Cacheable` |
| Template Method | Behavioral | Рецепт с заменяемыми ингредиентами | `JdbcTemplate`; `final` метод-каркас |
| Strategy | Behavioral | Сменный двигатель | Интерфейс + setter; лямбды = стратегии |
| Chain of Responsibility | Behavioral | Пропускной пункт с охранниками | Spring Security `SecurityFilterChain` |
| Observer | Behavioral | Email-рассылка | Spring: `@EventListener` / `publishEvent()` |
