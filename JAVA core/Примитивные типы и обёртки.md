# Примитивные типы и обёртки

## 1. Проблема — зачем это существует?

Java — объектно-ориентированный язык, но арифметика на объектах была бы катастрофически медленной. Примитивные типы хранятся прямо на Stack (или встроены в объект), без накладных расходов на заголовок объекта и GC. Обёртки (`Integer`, `Double`) нужны там, где требуется объект: дженерики, коллекции, nullable-значения.

## 2. Аналогия

Примитив — это наличные деньги в кармане (быстро, прямо, без посредника). Обёртка — та же сумма на банковской карте: можно передать кому угодно, но для каждой операции нужно обращаться в банк (дополнительная работа).

## 3. Как работает

### 8 примитивных типов:

| Тип | Размер | Диапазон | Обёртка |
|-----|--------|----------|---------|
| `byte` | 1 байт | -128 … 127 | `Byte` |
| `short` | 2 байта | -32 768 … 32 767 | `Short` |
| `char` | 2 байта | 0 … 65 535 (Unicode) | `Character` |
| `int` | 4 байта | -2³¹ … 2³¹−1 | `Integer` |
| `long` | 8 байт | -2⁶³ … 2⁶³−1 | `Long` |
| `float` | 4 байта | ±3.4×10³⁸ (7 цифр) | `Float` |
| `double` | 8 байт | ±1.8×10³⁰⁸ (15 цифр) | `Double` |
| `boolean` | 1 бит (4 байта в JVM) | true/false | `Boolean` |

### Autoboxing / Unboxing (Java 5+):

```java
// Autoboxing — компилятор превращает в Integer.valueOf(42)
Integer i = 42;

// Unboxing — компилятор превращает в i.intValue()
int x = i;

// Опасность: NullPointerException при unboxing null
Integer nullValue = null;
int y = nullValue; // NullPointerException!

// Опасность в коллекциях
Map<String, Integer> map = new HashMap<>();
int count = map.get("key"); // NPE если ключ отсутствует
```

### Integer Pool (IntegerCache):

```java
Integer a = 127;
Integer b = 127;
System.out.println(a == b);   // true  — один объект из кэша!

Integer c = 128;
Integer d = 128;
System.out.println(c == d);   // false — разные объекты

System.out.println(c.equals(d)); // true  — всегда используй equals
```

JVM кэширует `Integer` от **-128 до 127** через `IntegerCache`. То же самое для `Byte`, `Short`, `Character` (0..127), `Long` (-128..127). `Double` и `Float` — не кэшируются.

**Правило:** никогда не сравнивай обёртки через `==` — только `equals()`.

### Приведение типов (Type Casting):

```java
// Widening (расширяющее) — неявное, без потерь:
int i = 100;
long l = i;      // int → long, автоматически
double d = i;    // int → double, автоматически
// byte → short → int → long → float → double

// Narrowing (сужающее) — явное, возможна потеря данных:
double pi = 3.14;
int x = (int) pi;   // 3 — дробная часть отброшена
long big = 300L;
byte b = (byte) big; // 44 — переполнение (300 % 256 = 44)
```

## 4. Глубже — нюансы

### Производительность autoboxing в циклах:

```java
// Плохо — тысячи ненужных объектов Integer:
Long sum = 0L;
for (long i = 0; i < 1_000_000; i++) {
    sum += i; // unboxing + boxing на каждой итерации
}

// Хорошо — примитивы:
long sum = 0L;
for (long i = 0; i < 1_000_000; i++) {
    sum += i;
}
```

### Работа с float/double — потеря точности:

```java
System.out.println(0.1 + 0.2);  // 0.30000000000000004 — не 0.3!
// Для точных вычислений (деньги) — используй BigDecimal
BigDecimal result = new BigDecimal("0.1").add(new BigDecimal("0.2")); // 0.3
```

### String.valueOf vs .toString():

```java
Integer i = null;
String s1 = String.valueOf(i);  // "null" — безопасно
String s2 = i.toString();       // NullPointerException!
```

## 5. Связи с другими концепциями

- [[Иммутабельность String]] — String Pool аналогичен Integer Pool, но для строк
- [[Устройство HashMap]] — Integer, String — идеальные ключи, immutable с корректным hashCode
- [[Generics]] — дженерики работают только с обёртками (`List<Integer>`, не `List<int>`)

## 6. Ответ на собесе (2 минуты)

> "В Java 8 примитивных типов: `byte`, `short`, `char`, `int`, `long`, `float`, `double`, `boolean`. Каждый имеет обёртку — `Integer`, `Double` и т.д. Примитивы хранятся на Stack, объекты — на Heap. Примитивы быстрее, обёртки нужны для дженериков и коллекций.
>
> **Autoboxing** — автоматическое преобразование, компилятор подставляет `Integer.valueOf(42)`. Главная ловушка — NPE при unboxing: если `Integer` равен `null`, преобразование в `int` выбросит `NullPointerException`.
>
> **Integer Pool:** JVM кэширует `Integer` от -128 до 127. Поэтому `Integer a = 127; Integer b = 127; a == b` → `true`, а `Integer c = 128; Integer d = 128; c == d` → `false`. **Вывод:** обёртки сравниваем только через `equals()`, никогда через `==`.
>
> **Widening** — расширяющее приведение (int → long → double) — неявное, без потерь. **Narrowing** — сужающее (double → int) — явное, возможна потеря данных.
>
> Для финансовых расчётов не использую `double` — теряется точность (0.1 + 0.2 = 0.30000...4). Использую `BigDecimal`."

## Шпаргалка

| Тип | Размер | Обёртка | Кэш |
|-----|--------|---------|-----|
| `int` | 4 байта | `Integer` | -128..127 |
| `long` | 8 байт | `Long` | -128..127 |
| `double` | 8 байт | `Double` | нет |
| `boolean` | 4 байта (JVM) | `Boolean` | true/false |
| `char` | 2 байта (Unicode) | `Character` | 0..127 |

| Концепция | Суть | Ловушка |
|-----------|------|---------|
| **Autoboxing** | `int` → `Integer` автоматически | NPE при unboxing null |
| **Integer Pool** | Кэш -128..127 | `==` не работает вне диапазона |
| **Widening** | int → long → double | Неявное, без потерь |
| **Narrowing** | double → int | Явное, теряет данные |
| **float/double** | Приближённое | Не для денег, используй `BigDecimal` |
