# Хранение паролей — хеширование и шифрование

## 1. Проблема — зачем это существует?

Хранить пароли в открытом виде (plaintext) — катастрофа: утечка базы = все пароли компрометированы. Шифрование обратимо — если украден ключ, все пароли расшифровываются. **Хеширование односторонне** — из хеша пароль не восстановить. Правильная схема: хранить не пароль, а его хеш + соль, используя медленные алгоритмы, специально designed для паролей.

## 2. Аналогия

**Шифрование** — сейф с ключом. Кто знает ключ — может открыть и достать содержимое.

**Хеширование** — мясорубка. Можно пропустить мясо → получить фарш. Из фарша мясо не восстановить. Но если дать другому куску — получишь такой же фарш, и можно сравнить.

**Соль** — добавить уникальную приправу к каждому куску мяса. Теперь одинаковые куски дают разный фарш — rainbow tables бесполезны.

## 3. Как работает

### Разница: хеширование vs шифрование vs кодирование

| | Обратимо? | Ключ? | Применение |
|--|:---------:|:-----:|------------|
| **Кодирование** (Base64) | Да | Нет | Передача бинарных данных, не безопасность |
| **Шифрование** (AES, RSA) | Да | Да | Секретные данные, которые нужно расшифровать |
| **Хеширование** (BCrypt) | Нет | Нет | Пароли — нужно только проверить, а не восстановить |

---

### Алгоритмы для хеширования паролей

**НЕЛЬЗЯ использовать для паролей:**
- MD5, SHA-1, SHA-256 — быстрые. «Быстро» плохо: GPU перебирает миллиарды хешей/сек.

**Нужно использовать медленные adaptive algorithms:**

| Алгоритм | Особенность | Применение |
|----------|------------|------------|
| **BCrypt** | Настраиваемый cost factor | Текущий стандарт де-факто для паролей |
| **Argon2** | Победитель Password Hashing Competition 2015 | Современный выбор, устойчив к GPU |
| **PBKDF2** | NIST-рекомендованный | Требования compliance, FIPS |
| **SCrypt** | Устойчив к GPU + memory-hard | Альтернатива Argon2 |

### BCrypt в Spring Security

```java
// Конфигурация
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);   // cost factor 12 (2^12 итераций)
    // cost = 10 (default, ~100ms), 12 (~300ms), 14 (~1200ms)
}

// Регистрация: сохранить хеш
@Service
public class UserService {
    @Autowired
    private PasswordEncoder passwordEncoder;

    public void register(String username, String rawPassword) {
        String hashedPassword = passwordEncoder.encode(rawPassword);
        // "$2a$12$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy"
        userRepository.save(new User(username, hashedPassword));
    }

    public boolean login(String username, String rawPassword) {
        User user = userRepository.findByUsername(username);
        return passwordEncoder.matches(rawPassword, user.getPasswordHash());
        // matches() повторяет хеширование и сравнивает
    }
}
```

**Как работает BCrypt:**
```
$2a$12$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy
 |  |  |                                                      |
алг cost соль (22 символа) ──────────────────────────────── хеш
```

Соль **встроена в хеш** — каждый пользователь имеет свою уникальную соль, генерируемую BCrypt автоматически.

### Атаки и защита

| Атака | Описание | Защита |
|-------|----------|--------|
| **Rainbow table** | Предвычисленные хеши для словарных паролей | Соль: одинаковые пароли → разные хеши |
| **Brute force** | Перебор всех вариантов | Медленный алгоритм (BCrypt cost) |
| **Dictionary attack** | Перебор словарных паролей | Соль + медленный алгоритм |
| **Timing attack** | Измерение времени сравнения | Constant-time comparison |

---

### Шифрование — когда и чем

Если данные нужно **восстановить** — шифрование (не хеш):

```java
// Симметричное — AES (один ключ)
// Применение: шифрование данных в БД, файлов

// Асимметричное — RSA (публичный + приватный ключ)
// Применение: HTTPS, подпись JWT, шифрование ключей

// В Java: javax.crypto.Cipher, или библиотека Bouncy Castle
// В Spring: Spring Security Crypto module
```

**Данные, которые НЕ должны быть в открытом виде в БД:**
- Пароли → BCrypt (нехешируемое)
- Номера карт, SSN → AES (шифрование, нужно восстанавливать)
- Токены API → SHA-256 (быстрый хеш достаточен — не пароль)

## 4. Глубже — что важно знать

**Перец (Pepper) — дополнительная защита:** секретное значение, добавляемое к паролю перед хешированием, хранится не в БД, а в конфиге/vault. Даже утечка БД не поможет без pepper.

```java
String passwordWithPepper = rawPassword + pepper;
return passwordEncoder.encode(passwordWithPepper);
```

**Обновление cost factor:** с ростом мощности железа нужно увеличивать cost. BCrypt поддерживает upgrade: при следующем логине пользователя перехешируй с новым cost.

**Что проверять при code review:**
- Нет ли `MD5` или `SHA-1` для паролей
- Нет ли `equals()` для сравнения хешей (должен быть `matches()` или constant-time)
- Нет ли логирования паролей

## 6. Ответ на собесе (2 минуты)

Пароли нельзя хранить в открытом виде — при утечке БД все пароли скомпрометированы. Шифрование тоже плохо: обратимо, при утечке ключа — то же самое.

Правильный подход — **медленное хеширование с солью**. Соль: уникальное случайное значение на каждого пользователя. Одинаковые пароли → разные хеши → rainbow tables бесполезны.

Медленность критична: BCrypt с cost=12 считается ~300ms. Злоумышленник с GPU вместо миллиардов хешей/сек получает 3 попытки/сек — брутфорс становится нецелесообразным.

В Spring — `BCryptPasswordEncoder`. При регистрации: `encode(rawPassword)`. При логине: `matches(rawPassword, storedHash)`. Никогда MD5 или SHA-1 для паролей — они быстрые.

## Шпаргалка

| Нужно | Метод | Пример |
|-------|-------|--------|
| Хранить пароль | BCrypt/Argon2 | `BCryptPasswordEncoder(12)` |
| Хранить номер карты | AES шифрование | Нужно восстанавливать |
| Хранить API токен | SHA-256 | Быстрый хеш достаточен |
| Передать данные | Base64 | Это НЕ безопасность |

| Атака | Защита |
|-------|--------|
| Rainbow table | Соль (встроена в BCrypt) |
| Brute force | Медленный алгоритм (cost factor) |
| Утечка ключа шифрования | Не шифровать пароли — хешировать |
