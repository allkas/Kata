# Git cherry-pick

## 1. Проблема — зачем это существует?

Иногда нужно перенести один конкретный коммит из одной ветки в другую — без слияния всех изменений. Типичный сценарий: критический баг исправлен в `feature`-ветке, его нужно срочно выкатить в `main` или `release`, не дожидаясь завершения всей фичи. `merge` перенёс бы все изменения, `cherry-pick` берёт только один конкретный коммит.

## 2. Аналогия

Ты печёшь разные торты (ветки). В одном рецепте — идеальный крем. Хочешь взять только этот крем и добавить в другой торт, не перемешивая всё остальное.

`cherry-pick` — скопировать один шаг из другого рецепта. Оригинал не меняется, в текущем торте появляется копия этого шага с новым номером в истории.

## 3. Как работает

```bash
# Перенести один коммит в текущую ветку
git cherry-pick abc123def

# Перенести диапазон коммитов
git cherry-pick abc123..def456   # все коммиты между ними (abc123 не включается)
git cherry-pick abc123^..def456  # включая abc123

# Перенести без автоматического коммита (можно отредактировать)
git cherry-pick -n abc123

# Перенести с сохранением оригинального автора
git cherry-pick -x abc123   # добавит "(cherry picked from commit abc123)" в сообщение
```

---

### Типичный workflow

```bash
# Сценарий: фикс сделан в feature-ветке, нужен в main

git log feature --oneline
# a1b2c3d Fix: NPE in UserService.findById
# e4f5g6h WIP: feature in progress
# ...

# Переходим в main
git checkout main

# Берём только коммит с фиксом
git cherry-pick a1b2c3d

# Результат: main теперь содержит этот фикс
# feature-ветка не изменилась
```

---

### При конфликтах

```bash
# Конфликт во время cherry-pick
git cherry-pick abc123
# CONFLICT (content): Merge conflict in UserService.java

# Разрешить конфликты вручную, затем:
git add UserService.java
git cherry-pick --continue

# Или отменить cherry-pick
git cherry-pick --abort
```

## 4. Глубже — что важно знать

**Новый хэш:** `cherry-pick` создаёт новый коммит с новым хэшем. Это не тот же коммит, а копия изменений. В `git log` обе ветки будут иметь разные коммиты с одинаковым содержимым.

**Дублирование при последующем merge:** если потом сделать `merge` ветки, где был исходный коммит, в историю могут попасть две записи об одном изменении. Git обычно справляется (определяет, что изменение уже применено), но иногда возникают конфликты.

**Cherry-pick vs merge vs rebase:**

| Операция | Что переносит | Когда |
|----------|--------------|-------|
| `merge` | Все изменения ветки | Завершённая фича |
| `rebase` | Все изменения ветки, перебазируя | Актуализация ветки |
| `cherry-pick` | Один конкретный коммит | Срочный хотфикс, выборочный перенос |

**Когда НЕ использовать:**
- Когда нужно перенести всю ветку — использовать `merge` или `rebase`
- Для регулярного переноса изменений между ветками — это признак плохой структуры веток

**Типичный use case в командах:** hotfix-workflow — фикс коммитится в основную ветку, потом cherry-pick в release-ветки поддерживаемых версий.

## 6. Ответ на собесе (2 минуты)

`cherry-pick` переносит один конкретный коммит из одной ветки в другую. В отличие от `merge`, который объединяет все изменения ветки, `cherry-pick` берёт только один выбранный коммит.

Главный use case: критический баг исправлен в feature-ветке, нужно срочно выкатить в `main` или `release`, не дожидаясь завершения фичи. Делаю `git log feature --oneline`, беру хэш нужного коммита, переключаюсь на `main`, делаю `git cherry-pick <hash>`.

Важный нюанс: cherry-pick создаёт **новый коммит с новым хэшем** — это копия изменений, а не тот же коммит. Поэтому если потом делать полный merge этой ветки, в истории могут появиться дублирующиеся записи об одном изменении.

## Шпаргалка

| Команда | Что делает |
|---------|-----------|
| `git cherry-pick abc123` | Перенести один коммит |
| `git cherry-pick abc123..def456` | Диапазон коммитов |
| `git cherry-pick -n abc123` | Перенести без коммита |
| `git cherry-pick -x abc123` | Добавить ссылку на источник в commit message |
| `git cherry-pick --continue` | После разрешения конфликта |
| `git cherry-pick --abort` | Отменить cherry-pick |
