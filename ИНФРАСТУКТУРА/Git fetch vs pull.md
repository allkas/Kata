# Git fetch vs pull

## 1. Проблема — зачем это существует?

Коллеги запушили новые коммиты. Нужно обновить локальный репозиторий. `fetch` и `pull` делают это по-разному: один загружает, но не трогает твои файлы; другой загружает и сразу сливает — что может сломать работу, если у тебя есть незакоммиченные изменения.

## 2. Аналогия

**fetch** — заглянуть в почтовый ящик, переписать что пришло, но письма оставить в ящике. Ты знаешь что есть, но к тебе ничего не попало.

**pull** — вынуть все письма из ящика и сразу положить к своим бумагам. Быстро, но если у тебя беспорядок на столе — стопка разрастётся непредсказуемо.

## 3. Как работает

```
Удалённый репозиторий (origin): A---B---C---D
Локальный репозиторий:          A---B (origin/main = B)
```

### git fetch
```bash
git fetch origin
# Загружает D и обновляет origin/main → D
# Локальная ветка main всё ещё на B
# Ничего не слито
```

После fetch:
```
origin/main: A---B---C---D
main (local): A---B
```

Можно посмотреть, что изменилось, и решить — merge или rebase:
```bash
git log main..origin/main    # коммиты которых нет локально
git diff main origin/main    # изменения в коде
```

### git pull
```bash
git pull origin main
# = git fetch origin + git merge origin/main
```

По умолчанию `pull` делает merge:
```
main (local): A---B---M  (M — merge commit)
                    \ /
             C---D (из origin)
```

С флагом `--rebase`:
```bash
git pull --rebase origin main
# = git fetch + git rebase
# Локальные коммиты перенесутся поверх D
```

## 4. Сравнение

| Характеристика | `fetch` | `pull` |
|----------------|---------|--------|
| **Загружает изменения** | Да | Да |
| **Меняет рабочую ветку** | Нет | Да |
| **Создаёт merge commit** | Нет | Может |
| **Риск конфликтов** | Нет | Есть |
| **Безопасность** | Высокая | Средняя |
| **Контроль** | Полный | Меньше |

## 5. Рекомендуемый подход

```bash
# ❌ Антипаттерн — слепой pull
git pull origin main   # не знаю, что произойдёт

# ✅ Безопасный подход:
git fetch origin
git log main..origin/main  # смотрю что появилось
git merge origin/main      # или rebase — осознанно

# ✅ Настроить pull на rebase (глобально):
git config --global pull.rebase true
git pull origin main  # теперь = fetch + rebase
```

## 7. Ответ на собесе (2 минуты)

> "`fetch` загружает изменения из удалённого репозитория и обновляет `origin/main`, но **не трогает** локальную ветку. Ты сам потом решаешь — merge или rebase.
>
> `pull` = `fetch` + `merge` (по умолчанию) или `fetch` + `rebase` (с флагом `--rebase`). Автоматически сливает — удобно, но менее контролируемо.
>
> Я предпочитаю `fetch`, потому что могу сначала посмотреть через `git log main..origin/main` что появилось, и только потом осознанно слить. Это особенно важно если у меня есть незакоммиченные изменения."

## Шпаргалка

| Концепция | Суть | Деталь |
|-----------|------|--------|
| **fetch** | Скачать без слияния | Обновляет origin/main, не main |
| **pull** | fetch + merge/rebase | Сразу меняет рабочую ветку |
| **pull --rebase** | fetch + rebase | Линейная история |
| **Безопасный workflow** | fetch → inspect → merge | Полный контроль |
