# Git merge vs rebase

## 1. Проблема — зачем это существует?

В команде несколько разработчиков работают параллельно. Ветки расходятся. Нужно объединить изменения. Два инструмента — `merge` и `rebase` — делают одно и то же (интегрируют изменения), но по-разному формируют историю: `merge` сохраняет реальную хронологию, `rebase` создаёт линейную.

## 2. Аналогия

**Merge** — склеить две верёвки узлом. Видно, что верёвки были разными, видно где соединились. История честная.

**Rebase** — перемотать катушку. Берёшь все свои коммиты и наматываешь их заново поверх чужих, как будто работал поверх актуальной базы с самого начала. История красивая, но переписана.

## 3. Как работает

### Git merge
```
main:    A---B---C
                  \
feature:           D---E
```
После `git merge feature`:
```
main:    A---B---C---F  (F — merge commit)
                  \ /
feature:           D---E
```
Создаётся **merge commit F**, у которого два родителя. История показывает, что ветки существовали параллельно.

### Git rebase
```
main:    A---B---C
                  
feature:     D'--E'  (перенесены поверх C)
```
После `git rebase main` (на ветке feature):
```
main:    A---B---C
                  \
feature:           D'--E'  (новые коммиты, другие хэши)
```
Коммиты D и E становятся D' и E' — **новые SHA-хэши**, пересозданные поверх C.

### Быстрое сравнение

| Характеристика | `merge` | `rebase` |
|----------------|---------|----------|
| **История** | Сохраняет реальную хронологию | Создаёт линейную |
| **Merge commit** | Создаётся | Не создаётся |
| **Хэши коммитов** | Не меняются | Меняются (переписываются) |
| **Конфликты** | Один раз при merge | Возможны на каждом коммите |
| **Читаемость** | Хуже при большой команде | Лучше — линейная история |
| **Риск** | Низкий | Нельзя применять к публичным веткам |

## 4. Золотое правило rebase

**Никогда не делай rebase ветки, которую уже запушил и на которой работают другие.**

Почему: `rebase` переписывает хэши коммитов. Если коллега взял твои коммиты D и E, а ты сделал rebase и создал D' и E' — у него теперь дублирующиеся изменения с другими хэшами. `git pull` превратится в хаос.

```bash
# ОПАСНО — если ветка feature уже опубликована:
git checkout feature
git rebase main   # переписывает историю feature

# БЕЗОПАСНО — только для локальных веток перед PR:
git checkout my-local-feature
git rebase main   # приводим в порядок до публикации
```

## 5. Когда что использовать

| Сценарий | Команда | Почему |
|----------|---------|--------|
| Влить фичу в `main` | `merge` | Сохраняем историю, не переписываем |
| Обновить локальную ветку перед PR | `rebase` | Линейная история, чистый PR |
| Публичная ветка (`main`, `develop`) | `merge` | Нельзя переписывать историю |
| Убрать merge-коммиты перед PR | `rebase -i` (interactive) | Squash, reorder commits |
| Перенести коммиты с одной ветки | `cherry-pick` | Только нужные коммиты |

## 6. Merge vs rebase — стратегии команд

**Feature Branch Workflow (merge):**
```bash
git checkout main
git merge feature/order-service  # merge commit виден в истории
```

**Rebase Workflow (линейная история):**
```bash
git checkout feature/order-service
git rebase main         # актуализируем на main
git checkout main
git merge feature/order-service   # fast-forward, без merge commit
```

**Squash merge (один коммит вместо всей ветки):**
```bash
git merge --squash feature/order-service  # все изменения как staged
git commit -m "feat: add order service"   # один итоговый коммит
```

## 8. Ответ на собесе (2 минуты)

> "`merge` и `rebase` оба интегрируют изменения из одной ветки в другую, но по-разному формируют историю.
>
> **Merge** создаёт merge commit с двумя родителями — история честная, видно что ветки существовали параллельно. Хэши коммитов не меняются. Безопасно для публичных веток.
>
> **Rebase** берёт коммиты текущей ветки и пересоздаёт их поверх целевой. Результат — линейная история без merge commit. Но хэши коммитов меняются — это переписывание истории.
>
> **Золотое правило:** никогда не делать rebase публичной ветки, на которой работают коллеги. Это сломает им репозиторий — у них окажутся дублирующиеся коммиты с другими хэшами.
>
> **Мой подход:** локально обновляю feature-ветку через rebase (чистая история), вливаю в main через merge или fast-forward."

## Шпаргалка

| Концепция | Суть | Деталь |
|-----------|------|--------|
| **merge** | Объединяет ветки, создаёт merge commit | Хэши не меняются, безопасен |
| **rebase** | Переносит коммиты поверх другой ветки | Новые хэши, линейная история |
| **Золотое правило** | Rebase только локальных веток | Не трогай опубликованные ветки |
| **fast-forward** | Merge без commit если нет расхождения | Линейная история без лишнего commit |
| **squash** | `--squash` — все коммиты в один | Чистый PR с одним коммитом |
| **interactive rebase** | `rebase -i` — переупорядочить, squash | Чистка истории перед PR |
