#### 1. Открытие: "Три роли в тестировании" (15 секунд)

> "Эти три аннотации — фундамент для построения тестов в Mockito. Они решают три разные задачи: **создание** тестового двойника (@Mock, @Spy) и **автоматическое внедрение** этих двойников в тестируемый объект (@InjectMocks)."

---

#### 2. Аналогия: "Сборка автомобиля" (30 секунд)

> "Я объясняю через аналогию со **сборкой автомобиля на тестовом стенде**:

Представьте, что вы тестируете **двигатель автомобиля**.

**@Mock** — это как **создать муляж запчасти**:

> - Вам нужно проверить двигатель, но вместо реального бензонасоса вы ставите муляж.
>     
> - Муляж делает ровно то, что вы скажете: 'Когда двигатель попросит бензин — верни давление 2 атмосферы'.
>     

**@Spy** — это как **взять реальную запчасть, но с возможностью подменить её поведение**:

> - Вы берете реальный бензонасос, который работает как обычно.
>     
> - Но вы говорите: 'В этом тесте пусть он ведет себя нештатно — не давай давление'.
>     

**@InjectMocks** — это как **автосборщик, который сам ставит все запчасти на место**:

> - Вы создали муляжи или реальные запчасти.
>     
> - Автосборщик (@InjectMocks) берет их и **автоматически монтирует** в двигатель.
>     
> - Вам не нужно вручную прописывать: `engine.setFuelPump(mockFuelPump)`."
>

#### 3. Детальный разбор каждой аннотации (1 минута)

**@Mock — полностью поддельный объект**

|Характеристика|Описание|
|---|---|
|**Что создает**|Полностью поддельный объект (mock), все методы — заглушки|
|**Как создается**|`Mockito.mock(Class.class)`|
|**Реальное поведение**|Нет — по умолчанию возвращает default values|
|**Когда использовать**|Для изоляции зависимостей, которые не должны влиять на тест|


```java

@Mock
private UserRepository userRepository;  // полностью поддельный репозиторий
@Mock
private EmailService emailService;      // полностью поддельный email-сервис
```
**@Spy — частично реальный объект**

|Характеристика|Описание|
|---|---|
|**Что создает**|Обертку над реальным объектом (spy)|
|**Как создается**|`Mockito.spy(realObject)` или `@Spy` с экземпляром|
|**Реальное поведение**|Да — по умолчанию вызывает реальные методы|
|**Когда использовать**|Когда нужно реальное поведение, но с подменой отдельных методов|

```java

@Spy
private UserService userService = new UserService();  // реальный сервис
@Spy
private EmailService emailService;  // требует инициализации
```
**@InjectMocks — автоматическое внедрение**

|Характеристика|Описание|
|---|---|
|**Что делает**|Создает реальный объект и **автоматически внедряет** в него все @Mock и @Spy|
|**Как внедряет**|По типу, по имени, через конструктор, сеттеры, поля|
|**Когда использовать**|Для тестируемого класса, в который нужно подставить моки и спаи|

```java

@InjectMocks
private OrderService orderService;  // тестируемый класс
// Mockito сам найдет userRepository и emailService и внедрит их в orderService
```

#### 4. Как работает @InjectMocks (30 секунд)

> "Это самая умная аннотация. Она ищет подходящие моки и спаи для внедрения в тестируемый объект.

**Порядок внедрения (приоритет):**

> 1. **Конструктор** — самый предпочтительный
>     
> 2. **Сеттеры** — если нет подходящего конструктора
>     
> 3. **Поля (field injection)** — если нет ни конструктора, ни сеттеров
>     

**Что внедряется:**

> - Все поля, аннотированные `@Mock` или `@Spy`
>     
> - По типу (если один мок нужного типа)
>     
> - По имени (если несколько моков одного типа, используется имя поля)
>     

**Пример:**

```java

public class OrderService {
    private final UserRepository userRepository;
    private final EmailService emailService;
    
    public OrderService(UserRepository userRepository, EmailService emailService) {
        this.userRepository = userRepository;
        this.emailService = emailService;
    }
}
// В тесте:
@Mock private UserRepository userRepository;
@Mock private EmailService emailService;
@InjectMocks private OrderService orderService;  // автоматом подставит оба мока
```

#### 5. Сравнение и компромиссы (30 секунд)
| Характеристика | @Mock | @Spy | @InjectMocks |
|----------------|-------|------|--------------|
| **Создает объект** | Да (полностью поддельный) | Да (обертка над реальным) | Да (реальный объект) |
| **Внедряет зависимости** | Нет | Нет | Да (автоматически) |
| **Реальное поведение** | Нет | Да (по умолчанию) | Да (реальный код тестируемого класса) |
| **Когда использовать** | Для зависимостей | Для частичного контроля | Для тестируемого класса |
| **Инициализация** | `MockitoAnnotations.openMocks(this)` | Требует реального объекта | Автоматически после инициализации моков |

#### 6. Связь с прошлыми темами (30 секунд)
> "Теперь свяжу с более широким контекстом.
**Связь с Mock vs Spy:**
> - `@Mock` — это сокращение для `Mockito.mock()`
> - `@Spy` — это сокращение для `Mockito.spy()`
> - `@InjectMocks` автоматизирует то, что раньше делалось вручную: `new OrderService(userRepository, emailService)`
**Связь с тестированием микросервисов:**
> - При тестировании сервиса мы используем `@Mock` для его зависимостей (репозитории, http-клиенты)
> - `@InjectMocks` создает сам тестируемый сервис с этими моками
> - `@Spy` редко используется — чаще `@Mock` для изоляции
**Связь с TDD:**
> - `@Mock` + `@InjectMocks` позволяют писать тесты **до** реализации класса. Вы описываете, как зависимости должны себя вести, и пишете тест, который проверяет логику еще не написанного кода.
**Важный нюанс:**
> - `@InjectMocks` работает только с **полями**, которые используются в тестируемом классе. Если зависимость внедряется через конструктор — работает отлично. Если через `new` внутри метода — не сможет её замокать."

#### 7. Закрытие: "Коротко о главном" (15 секунд)
> "Резюмирую: 
> - **@Mock** создает поддельные зависимости (изоляция)
> - **@Spy** создает обертку над реальным объектом (частичный контроль)
> - **@InjectMocks** создает тестируемый объект и автоматически внедряет в него все моки и спаи
>
> Стандартный паттерн в тестах: `@Mock` для всех зависимостей, `@InjectMocks` для тестируемого класса, `@Spy` — только когда реально нужно реальное поведение."

## Шпаргалка для запоминания (одна страница)
### Сравнение аннотаций
| Аннотация | Что создает | Реальное поведение | Внедряет зависимости | Когда использовать |
|-----------|-------------|-------------------|---------------------|-------------------|
| **@Mock** | Полностью поддельный объект | Нет | Нет | Для зависимостей (изоляция) |
| **@Spy** | Обертка над реальным объектом | Да (по умолчанию) | Нет | Когда нужно реальное поведение, но с подменой |
| **@InjectMocks** | Реальный объект | Да (код тестируемого класса) | Да (автоматически) | Для тестируемого класса |
### Как они работают вместе
```java
@ExtendWith(MockitoExtension.class)
class OrderServiceTest {
    
    @Mock
    private UserRepository userRepository;  // зависимость 1 — мок
    
    @Mock
    private EmailService emailService;      // зависимость 2 — мок
    
    @InjectMocks
    private OrderService orderService;      // тестируемый класс — автоматом получит оба мока
    
    @Test
    void testCreateOrder() {
        // настройка моков
        when(userRepository.findById(1L)).thenReturn(Optional.of(user));
        
        // вызов тестируемого метода
        orderService.createOrder(order);
        
        // проверка
        verify(userRepository).save(any());
        verify(emailService).send(any());
    }
}
```
### Связь с прошлыми темами

|Тема|Связь|
|---|---|
|**Mock vs Spy**|@Mock = mock(), @Spy = spy()|
|**Unit-тестирование**|@Mock для изоляции, @InjectMocks для тестируемого класса|
|**Микросервисы**|@Mock для внешних сервисов (HTTP-клиенты)|
|**TDD**|Позволяет писать тесты до реализации|
