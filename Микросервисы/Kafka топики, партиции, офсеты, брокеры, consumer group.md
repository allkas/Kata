# Kafka: топики, партиции, офсеты, брокеры, consumer group

## 1. Проблема — зачем это существует?

Традиционные очереди (RabbitMQ) удаляют сообщение после обработки и не масштабируются горизонтально по чтению. При highload (миллионы событий в секунду) это узкое место. Kafka переосмыслила брокер как **распределённый лог**: сообщения хранятся на диске, несколько consumer group читают один поток независимо, скорость чтения не влияет на скорость записи.

## 2. Аналогия

Kafka — как супер-почтовая служба для предприятий.

| Термин Kafka | Аналогия |
|--------------|---------|
| **Топик (Topic)** | Улица / почтовый адрес (категория писем) |
| **Партиция (Partition)** | Почтовый ящик на этой улице |
| **Офсет (Offset)** | Номер письма в ящике (0, 1, 2, 3...) |
| **Брокер (Broker)** | Почтовое отделение (физический сервер) |
| **Consumer Group** | Бригада курьеров, разбирающих ящики |
| **Кластер (Cluster)** | Сеть почтовых отделений |

Ключевое правило: одно письмо из одного ящика забирает только один курьер из одной бригады. Разные бригады могут читать одни и те же письма независимо.

## 3. Как работает

### Топик (Topic)

Логическая категория данных. Примеры: `payments`, `user-events`, `order-status`. Физически — набор партиций.

### Партиция (Partition)

Топик делится на партиции — неизменяемые упорядоченные логи:

```
Топик "payments":
  Партиция 0: [offset:0 alice:100] [offset:1 bob:250] [offset:2 alice:50]
  Партиция 1: [offset:0 carol:75]  [offset:1 dave:300]
  Партиция 2: [offset:0 eve:500]   [offset:1 frank:25]
```

- Запись идёт только в **конец** партиции (append-only)
- Порядок гарантируется **внутри партиции**, но не между партициями
- Kafka **не удаляет** сообщения после чтения — хранит согласно retention policy

### Офсет (Offset)

Уникальный индекс сообщения **внутри партиции**. Глобального офсета по всему топику нет — это позволяет параллельную запись без глобальных блокировок.

Consumer group фиксирует (коммитит) свой офсет. При перезапуске чтение возобновляется с последнего закоммиченного офсета.

### Брокер (Broker)

Один сервер (нода) в кластере Kafka. Хранит несколько партиций разных топиков. Один брокер — лидер для части партиций, реплика для других.

### Consumer Group

Группа потребителей, работающих совместно над одним топиком:

```
Топик: 3 партиции
Consumer Group A (2 consumer'а):
  Consumer 1 → Партиции 0, 2
  Consumer 2 → Партиция 1

Consumer Group B (1 consumer):
  Consumer 1 → Партиции 0, 1, 2  (читает всё независимо от группы A)
```

**Ключевые правила Consumer Group:**
- Одна партиция → только один consumer внутри группы (нет дублирования)
- Consumer'ов в группе не должно быть больше, чем партиций (лишние простаивают)
- Для масштабирования: увеличить партиции → добавить consumer'ов

### Почему Kafka быстрая?

| Механизм | Эффект |
|----------|--------|
| Sequential I/O | Запись в конец файла — самая быстрая дисковая операция |
| Zero-copy | Передача данных из OS page cache прямо в сеть без копирования |
| Batching | Producer накапливает сообщения в батч перед отправкой |
| Компрессия | Батчи хорошо сжимаются (snappy, lz4) |

## 4. Глубже — что важно знать

**Что происходит при перевесе consumer'ов:**
- 5 consumer'ов, 3 партиции → 2 consumer'а простаивают (это нормально при scaling down)
- Решение: количество партиций задаётся при создании топика и не уменьшается легко — планировать заранее

**Rebalancing:** когда consumer входит в группу или выходит, Kafka перераспределяет партиции между consumer'ами. Во время rebalancing потребление останавливается (stop-the-world rebalance). В новых версиях — incremental cooperative rebalancing.

**Retention policy:**
- По времени: `retention.ms=604800000` (7 дней)
- По размеру: `retention.bytes=1073741824` (1 GB)
- Compact: хранит только последнее значение для каждого ключа (для state)

**Kafka — не очередь, а лог:** сообщения не удаляются после чтения. Можно перечитать историю с любого офсета, запустить новую consumer group на старых данных, восстановить состояние после сбоя.

## 6. Ответ на собесе (2 минуты)

Kafka — это распределённый лог. Главное отличие от обычной очереди: сообщения не удаляются после чтения, хранятся на диске согласно retention policy.

Топик — логическая категория данных. Физически он разбит на партиции — упорядоченные неизменяемые логи. Каждое сообщение в партиции имеет уникальный офсет — индекс, с которого consumer продолжит чтение после перезапуска. Партиции распределены по брокерам — физическим серверам кластера.

Consumer group — группа потребителей, читающих топик совместно. Каждая партиция достаётся ровно одному consumer'у в группе — это гарантирует порядок и исключает дубли внутри группы. Но разные consumer group читают топик независимо: у каждой свой офсет. Это позволяет, например, аналитике и бизнес-логике потреблять один поток событий независимо.

Для масштабирования: увеличиваем партиции → добавляем consumer'ов в группу. Но consumer'ов не должно быть больше, чем партиций — лишние будут простаивать.

## Шпаргалка

| Концепция | Аналогия | Ключевой факт |
|-----------|----------|---------------|
| Топик | Улица с почтовыми ящиками | Логическая категория; физически = партиции |
| Партиция | Один почтовый ящик | Append-only лог; порядок гарантирован внутри |
| Офсет | Номер письма в ящике | Уникален внутри партиции; consumer коммитит свой |
| Брокер | Почтовое отделение | Физический сервер; хранит партиции |
| Consumer Group | Бригада курьеров | 1 партиция → 1 consumer; разные группы независимы |
