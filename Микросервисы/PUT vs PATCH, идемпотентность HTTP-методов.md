# PUT vs PATCH, идемпотентность HTTP-методов

## 1. Проблема — зачем это существует?

При обновлении ресурса через API возникает вопрос: отправлять весь объект или только изменённые поля? Если отправлять всё — безопасно, но дорого по трафику. Если только изменения — экономно, но сложнее. HTTP формализует этот выбор через PUT (полная замена) и PATCH (частичное обновление). Дополнительно вводится понятие идемпотентности — свойство, критичное для надёжных систем с ретраями.

## 2. Аналогия

**PUT** — как заменить весь лист в папке. Достаёшь старый, кладёшь новый. Поля, которые не вложил, — пропали.

**PATCH** — как исправить конкретные строки корректором. Остальное не трогаешь.

**Идемпотентность** — как кнопка лифта. Нажать три раза не вызовет лифт трижды — результат тот же, что и от одного нажатия.

## 3. Как работает

### PUT — полная замена ресурса

```http
PUT /users/123
Content-Type: application/json
{"name": "Alice Smith", "email": "alice@mail.com", "age": 30}
```

- Заменяет ресурс целиком: поля не переданные в теле **удаляются**
- **Всегда идемпотентен**: повторный PUT с теми же данными → тот же результат
- Если ресурс не существует — создаёт (`201`); если существует — заменяет (`200`/`204`)

```
До PUT:         После PUT (без phone):
name: Alice     name: Alice Smith  ✅
email: old@     email: alice@      ✅
age: 28         age: 30            ✅
phone: +123     phone: [удалён]   ❌
```

---

### PATCH — частичное обновление

```http
PATCH /users/123
Content-Type: application/json
{"age": 31}    ← только изменённые поля
```

- Незатронутые поля **сохраняются**
- Идемпотентность **опциональна** — зависит от реализации

**Два формата PATCH:**

**JSON Merge Patch (RFC 7396)** — простой, интуитивный:
```http
PATCH /users/123
Content-Type: application/merge-patch+json
{"age": 31, "phone": null}
```
- Отсутствующие поля → не меняются
- `null` → удалить поле

**JSON Patch (RFC 6902)** — точные операции, атомарность:
```http
PATCH /users/123
Content-Type: application/json-patch+json
[
  {"op": "replace", "path": "/age",  "value": 31},
  {"op": "add",     "path": "/tags/-", "value": "vip"},
  {"op": "remove",  "path": "/phone"}
]
```
Операции: `add`, `remove`, `replace`, `copy`, `move`, `test` (атомарная проверка перед изменением).

---

### Идемпотентность

**Идемпотентность** — повторный вызов с теми же параметрами даёт тот же результат, что и первый.

```
x = 5           → идемпотентна: x всегда = 5
x = x + 1       → не идемпотентна: каждый вызов увеличивает x
DELETE /user/1  → идемпотентна: ресурса всё равно нет (204 → 404, но состояние одно)
```

**Идемпотентный PATCH:**
```http
PATCH /users/123
{"status": "active"}    ← повтор не меняет результат
```

**Неидемпотентный PATCH:**
```http
PATCH /counter/123
{"op": "increment", "value": 1}    ← каждый вызов увеличивает счётчик
```

**Таблица идемпотентности HTTP-методов:**

| Метод | Идемпотентный | Безопасный | Пример использования |
|-------|:---:|:---:|---|
| GET | ✅ | ✅ | Получение данных |
| POST | ❌ | ❌ | Создание ресурса (сервер назначает ID) |
| PUT | ✅ | ❌ | Полная замена / идемпотентное создание |
| PATCH | ⚠️ | ❌ | Частичное обновление |
| DELETE | ✅ | ❌ | Удаление ресурса |

## 4. Глубже — что важно знать

**Зачем идемпотентность важна на практике:**
Сеть ненадёжна. Клиент отправил PUT — таймаут. Повторить безопасно? PUT — да, POST — нет. Идемпотентные методы можно ретраить автоматически в HTTP-клиентах, API Gateway, load balancer.

**PUT для создания с известным ID (идемпотентное создание):**
```http
PUT /users/alice@mail.com   ← клиент задаёт ID
{"name": "Alice"}           ← создаст если нет, заменит если есть
```
Повторный PUT не создаст дубль — в отличие от POST.

**Когда что выбирать:**

| Сценарий | Метод | Причина |
|----------|-------|---------|
| Обновить одно поле профиля | PATCH | Передаём только нужное поле |
| Обновить весь профиль | PUT | Полная замена, нет риска ошибиться в полях |
| Инкрементировать счётчик | PATCH | PUT не знает текущего значения |
| Идемпотентное создание | PUT | POST не идемпотентен, создаст дубль при ретрае |
| Сложная атомарная операция | PATCH (JSON Patch) | `test` операция для атомарной проверки |

## 6. Ответ на собесе (2 минуты)

PUT и PATCH оба обновляют ресурс, но по-разному. PUT заменяет ресурс целиком — поля, не переданные в теле, удаляются. Поэтому для обновления одного поля нужно передать весь объект. PATCH обновляет только то, что указал — остальное сохраняется.

Идемпотентность — это свойство операции, при котором повторный вызов с теми же данными даёт тот же результат. PUT всегда идемпотентен: повторный PUT с теми же данными — тот же результат. PATCH — зависит от реализации: `{"age": 31}` идемпотентно, а `{"op": "increment"}` — нет.

На практике идемпотентность важна для ретраев: если запрос ушёл и пришёл таймаут — идемпотентный запрос можно повторить без риска. Поэтому API Gateway и клиенты ретраят GET/PUT/DELETE, но не POST.

Два формата PATCH: JSON Merge Patch — простой (absent = не трогать, null = удалить). JSON Patch — операции (add, remove, replace, test) с атомарной проверкой состояния.

## Шпаргалка

| | PUT | PATCH |
|---|---|---|
| **Действие** | Полная замена | Частичное обновление |
| **Неуказанные поля** | Удаляются | Сохраняются |
| **Идемпотентность** | ✅ Всегда | ⚠️ Зависит от реализации |
| **Размер запроса** | Весь ресурс | Только изменения |
| **Создание** | ✅ Если не существует | ❌ Обычно нет |
| **Стандарт** | RFC 7231 | RFC 5789 |
