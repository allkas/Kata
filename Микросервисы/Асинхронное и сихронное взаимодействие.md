# Синхронное и асинхронное взаимодействие в микросервисах

## 1. Проблема — зачем это существует?

В монолите компоненты общаются через вызовы функций в памяти. В микросервисах всё идёт по сети. Синхронный вызов (HTTP) требует, чтобы оба сервиса были живы одновременно, создаёт цепочку зависимостей и не справляется с пиковой нагрузкой. Асинхронный подход через брокер сообщений решает эти проблемы: отправитель не ждёт получателя, сервисы не зависят друг от друга по времени.

## 2. Аналогия

**Синхронное взаимодействие** — телефонный звонок: нужно, чтобы собеседник поднял трубку прямо сейчас. Если он занят — ждёшь. Если недоступен — не получится.

**Асинхронное взаимодействие** — email: написал и отправил. Получатель прочитает, когда будет готов. Можно отправить тысячу писем сразу, не ожидая каждого ответа.

## 3. Как работает

### Синхронное взаимодействие (HTTP/gRPC)

```
OrderService ──HTTP POST──► PaymentService
             ◄──200 OK──────
             ──HTTP POST──► InventoryService
             ◄──200 OK──────
```

- Вызывающий **ждёт ответа**
- Оба сервиса должны быть **доступны одновременно**
- При ошибке — немедленный ответ с кодом ошибки
- Удобно когда нужен **мгновенный результат**

---

### Асинхронное взаимодействие (Kafka/RabbitMQ)

```
OrderService ──publish──► [Kafka: orders topic]
                                    │
             ◄──200 OK────          │ (сразу отвечает клиенту)
                                    ▼
                         PaymentService (читает из топика)
                         InventoryService (читает из топика)
```

- Отправитель **не ждёт** — публикует событие и идёт дальше
- Получатель читает в **своём темпе**
- Сервисы **не зависят** друг от друга по времени

---

### Три проблемы, которые решает асинхронность

**1. Temporal Coupling (временная связанность)**

Без брокера: `order-service` вызывает `payment-service` по HTTP. Если `payment-service` упал на 5 минут — заказ не создаётся.

С брокером: `order-service` публикует событие в Kafka. `payment-service` оживает через 5 минут и обрабатывает всё, что накопилось. Сервисы не должны быть доступны одновременно.

**2. Буферизация пиков (Load Buffering)**

Без брокера: «чёрная пятница», 10 000 заказов/сек — `payment-service` рассчитан на 1 000/сек, падает.

С брокером: `order-service` быстро кидает в Kafka, клиенту отвечает «принято в обработку». `payment-service` читает со своей скоростью. Kafka — буфер между быстрым продюсером и медленным потребителем.

**3. Decoupling (слабое связывание)**

Без брокера: `order-service` явно вызывает Payment, Inventory, Analytics. Добавление CRM → изменение кода `order-service`.

С брокером: `order-service` публикует `OrderCreated`. Все заинтересованные подписываются. Добавление нового сервиса — без изменений в `order-service`.

---

### Сравнение подходов

| | Синхронное (HTTP) | Асинхронное (Kafka) |
|--|-------------------|---------------------|
| Ожидание | Да | Нет |
| Оба сервиса должны быть живы | Да | Нет |
| Результат | Мгновенный | Eventual |
| Пиковая нагрузка | Плохо (падение) | Хорошо (буфер) |
| Связанность | Сильная | Слабая |
| Отладка | Проще | Сложнее |
| Когда | Нужен немедленный ответ | Можно обработать позже |

### Dead Letter Queue (DLQ)

Если сервис не смог обработать сообщение (исключение, невалидные данные) — после N попыток оно уходит в **DLQ (Dead Letter Queue)**. Не теряется, ждёт ручного разбора или автоматической логики.

```yaml
# Конфигурация retry + DLQ в Spring Kafka
spring.kafka.consumer.max-poll-records=10
# После 3 попыток — в DLQ
```

## 4. Глубже — что важно знать

**Circuit Breaker в асинхронном мире нужен реже.** В синхронном HTTP Circuit Breaker разрывает цепь при каскадном отказе. В асинхронном мире `payment-service` может упасть — сообщения просто копятся в очереди. Когда оживает — «догоняет». Каскадного отказа нет.

**Идемпотентность потребителей.** Kafka гарантирует at-least-once delivery — одно сообщение может прийти дважды. Каждый consumer должен обрабатывать дубли корректно: проверять `transactionId` перед записью.

**Request-Reply pattern.** Иногда нужен асинхронный запрос с получением ответа. Реализуется через два топика: `requests` и `responses`, с correlation ID. Усложняет код, применяется редко.

**Когда оставить синхронность:**
- Пользователь ждёт прямо сейчас (поиск, авторизация, чтение данных)
- Нужна немедленная валидация (проверить, есть ли товар в наличии)
- Операция быстрая и некритичная к отказу

## 6. Ответ на собесе (2 минуты)

В микросервисах два способа коммуникации. **Синхронный (HTTP/gRPC):** вызывающий ждёт ответа, оба сервиса должны быть живы одновременно — удобно для операций, где нужен мгновенный результат (авторизация, чтение данных).

**Асинхронный (Kafka/RabbitMQ):** отправитель публикует событие и не ждёт — решает три проблемы. Первая: temporal coupling — получатель может быть временно недоступен, обработает позже. Вторая: буферизация пиков — Kafka накапливает сообщения при всплеске нагрузки, потребитель читает в своём темпе. Третья: decoupling — новые сервисы подписываются на события без изменения издателя.

Моё правило: если пользователь ждёт ответа прямо сейчас — синхронно. Если операцию можно обработать позже, она долгая или требует нескольких сервисов — асинхронно. Саги строятся поверх асинхронного взаимодействия.

## Шпаргалка

| | HTTP (sync) | Kafka (async) |
|--|-------------|---------------|
| **Ожидание** | Да | Нет |
| **Доступность получателя** | Обязательна | Не требуется |
| **Пиковая нагрузка** | Плохо | Буфер |
| **Связанность** | Сильная | Слабая |
| **Когда** | Нужен ответ сейчас | Можно позже |

| Проблема | Решение через брокер |
|----------|---------------------|
| Temporal coupling | Сообщения ждут в очереди |
| Пики нагрузки | Буферизация в топике |
| Tight coupling | Publisher не знает subscribers |
| Обработка дублей | Идемпотентный consumer |
