# Брокеры сообщений

## 1. Проблема — зачем это существует?

В синхронной архитектуре сервисы вызывают друг друга напрямую по HTTP. Если `email-service` тормозит, `order-service` ждёт и блокирует поток. Если `email-service` упал, заказ не оформится, хотя email — не критичная операция. Кроме того, один сервис не может отправить событие сразу в несколько получателей без жёсткой связи.

**Брокер сообщений решает три проблемы:**
- **Decoupling** — отправитель не знает и не ждёт получателей
- **Буферизация пиков** — брокер принимает сообщения быстрее, чем consumer их обрабатывает
- **Fan-out** — одно событие могут получить несколько независимых сервисов

## 2. Аналогия

Брокер сообщений — это **ящик для заявлений в госучреждении**. Гражданин (producer) кладёт заявление в ящик и уходит — он не ждёт у окошка. Чиновник (consumer) достаёт заявления в своём темпе и обрабатывает. Несколько отделов (consumer group) могут получать копии одного заявления независимо.

## 3. Как работает

**Основные компоненты:**

| Компонент | Kafka | RabbitMQ |
|-----------|-------|----------|
| Хранилище | Топик (append-only лог) | Очередь (Queue) |
| Модель | Pull — consumer сам тянет | Push — брокер толкает |
| Хранение | Долгое (retention days/weeks) | Короткое (до ACK) |
| Replay | ✅ Да (перемотка по offset) | ❌ Нет (удаляется после ACK) |
| Маршрутизация | Простая (топик → партиции) | Сложная (exchanges, bindings) |
| Throughput | 100k+ msg/sec | 10–50k msg/sec |
| Задержка | >10 мс (batch) | <1 мс |

**Гарантии доставки:**
- **At-most-once** — сообщение может потеряться, дубли исключены
- **At-least-once** — сообщение дойдёт, но возможны дубли (нужна идемпотентность)
- **Exactly-once** — сложно, требует транзакций; Kafka поддерживает через idempotent producer + транзакции

**Dead Letter Queue (DLQ):** сообщения, которые consumer не смог обработать N раз, отправляются в специальную очередь для ручного анализа.

## 4. Глубже — что важно знать

**Когда Kafka, когда RabbitMQ:**

| Критерий | Выбор |
|----------|-------|
| Нужен replay / история событий | Kafka |
| Много потребителей одного события | Kafka |
| Сложная маршрутизация (topic/headers/routing key) | RabbitMQ |
| Задержка <10 мс (RPC-style) | RabbitMQ |
| Миллионы событий/сек, аналитика | Kafka |
| Task queue (обработка изображений, email) | RabbitMQ |

**Частые ошибки:**
- Consumer не подтверждает сообщение (no ACK) → бесконечный redelivery
- Нет идемпотентности при `at-least-once` → дублированные заказы/платежи
- DLQ не мониторится → молча теряются сообщения

## 6. Ответ на собесе (2 минуты)

> "Брокер сообщений решает проблему жёсткой связи между сервисами. Когда `order-service` напрямую вызывает `email-service` по HTTP, мы получаем temporal coupling: `order-service` ждёт ответа и умирает вместе с `email-service`.
>
> Аналогия — ящик для заявлений в государственном учреждении. Гражданин оставляет заявление и уходит, чиновник обрабатывает в своём темпе. Никто никого не ждёт.
>
> Механически брокер — это посредник, который принимает сообщения от producer и хранит их до того момента, когда consumer готов их обработать. Это даёт три вещи: decoupling (отправитель не знает получателей), буферизацию пиков нагрузки и fan-out — одно событие могут получить несколько независимых сервисов.
>
> Есть две основные реализации: Kafka — это append-only лог с долгим хранением и возможностью replay, идеален для event sourcing и потоковой аналитики. RabbitMQ — очередь с push-моделью и гибкой маршрутизацией, идеален для task queues и RPC.
>
> В продакшене важно настроить Dead Letter Queue — куда попадают сообщения, которые consumer не смог обработать. Без мониторинга DLQ можно молча терять события неделями."

## Шпаргалка

| Концепция | Аналогия | Ключевой факт |
|-----------|----------|---------------|
| Брокер сообщений | Ящик для заявлений | Decoupling producer от consumer |
| Kafka | Библиотечный архив | Можно перечитать с любого момента (offset) |
| RabbitMQ | Курьерская служба | Push-модель, сложная маршрутизация, после ACK удаляет |
| At-least-once | Заказное письмо с уведомлением | Дойдёт, но нужна идемпотентность |
| Dead Letter Queue | Корзина для нераспознанных писем | Сообщения, которые consumer не смог обработать |
| Fan-out | Рассылка одного письма в несколько отделов | Одно событие → несколько consumer groups |
