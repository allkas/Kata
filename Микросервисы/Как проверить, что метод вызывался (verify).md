#### 1. Открытие: "Проверяем не результат, а взаимодействие" (15 секунд)

> "`verify()` — это ключевой метод Mockito для **поведенческого тестирования**. Если `when().thenReturn()` настраивает **что вернуть**, то `verify()` проверяет **был ли вызван метод и с какими параметрами**. Это особенно важно, когда тестируемый метод **ничего не возвращает** (void) или когда результат работы — это побочные эффекты (вызов другого сервиса, запись в БД)."

---

#### 2. Аналогия: "Наблюдатель в офисе" (30 секунд)

> "Я объясняю через аналогию с **офисным наблюдателем**:

Представьте, что вы тестируете работу **секретаря**. Вы даете ему задание: 'Отправь письмо клиенту'.

**Проверка результата (assert)** — это как **проверить, что письмо пришло клиенту**. Но если письмо не пришло мгновенно (асинхронно), вы не можете это проверить.

**Проверка вызова (verify)** — это как **поставить камеру над секретарем**:

> - Вы смотрите запись: 'Ага, он взял телефон, набрал номер, сказал: "Здравствуйте"'.
>     
> - Вы не проверяете, что клиент услышал — вы проверяете, что **секретарь сделал свою работу**.
>     

`verify()` — это та самая 'камера', которая фиксирует, что метод был вызван, с какими аргументами и сколько раз."

---

#### 3. Базовый синтаксис и варианты (45 секунд)

**Базовый синтаксис:**

java

// Проверить, что метод был вызван хотя бы один раз
verify(userService).save(user);
// Проверить, что метод вызывался с конкретными аргументами
verify(emailService).send("user@example.com", "Welcome!");

**Счетчики вызовов (times):**

java

// Ровно 1 раз (дефолт)
verify(userService, times(1)).save(user);
// Ни разу
verify(userService, never()).delete(any());
// Ровно 3 раза
verify(userService, times(3)).notify();
// Хотя бы 1 раз
verify(userService, atLeast(1)).process();
// Не более 2 раз
verify(userService, atMost(2)).log();

**Проверка порядка вызовов (InOrder):**

java

InOrder inOrder = inOrder(emailService, smsService);
inOrder.verify(emailService).send("order@example.com");
inOrder.verify(smsService).send("+79991234567");
// Гарантирует, что email был отправлен ДО sms

**Проверка с матчерами аргументов:**

java

// any() — любой аргумент
verify(logger).log(anyString());
// eq() — точное совпадение (нужен, если смешиваются any и конкретные)
verify(service).process(eq("user"), anyInt());
// capture() — захват аргумента для дальнейших проверок
ArgumentCaptor<Email> captor = ArgumentCaptor.forClass(Email.class);
verify(emailService).send(captor.capture());
assertEquals("welcome@example.com", captor.getValue().getRecipient());

#### 4. Два подхода: state verification vs behavior verification (30 секунд)

> "Это важный концептуальный момент.

|Тип проверки|Что проверяет|Когда использовать|
|---|---|---|
|**State verification**|Состояние объекта после вызова|Метод возвращает значение или меняет состояние|
|**Behavior verification**|Взаимодействие с зависимостями|Метод void, вызов внешних сервисов, асинхронные операции|

**Связь с прошлым:**

> - В **unit-тестах** мы часто используем **behavior verification** (verify), потому что тестируем изолированный класс и хотим убедиться, что он правильно взаимодействует со своими зависимостями (которые замоканы).
>     
> - В **интеграционных тестах** чаще используют **state verification** — проверяют, что данные действительно попали в БД."
>

#### 5. Где и когда использовать verify (45 секунд)

> "Теперь о конкретных сценариях.

**1. Методы void (нет возвращаемого значения)**

java

// Тестируем, что сервис отправил уведомление
orderService.createOrder(order);
verify(notificationService).sendOrderConfirmation(order.getId());
// Если sendOrderConfirmation не вызван — тест упадет

**2. Проверка взаимодействия с зависимостями**

java

// Проверяем, что сервис сохранил объект в репозиторий
verify(userRepository).save(user);
verify(userRepository, never()).delete(any());

**3. Проверка, что метод НЕ был вызван (never)**

java

// При ошибке валидации email не должен отправляться
assertThrows(ValidationException.class, () -> service.register(invalidUser));
verify(emailService, never()).send(any());

**4. Проверка количества вызовов (важно для циклов, батчей)**

java

// Проверяем, что для каждого из 10 пользователей вызван метод
verify(userService, times(10)).process(any(User.class));

**5. Проверка порядка (InOrder) — для последовательностей**

java

// Проверяем, что сначала списали деньги, потом зарезервировали товар
InOrder inOrder = inOrder(paymentService, inventoryService);
inOrder.verify(paymentService).charge(any());
inOrder.verify(inventoryService).reserve(any());

#### 6. Связь с прошлыми темами (30 секунд)

> "Теперь свяжу с более широким контекстом.

**Связь с сагами (Saga):**

> - При тестировании саги мы проверяем, что оркестратор **вызвал** нужные сервисы в правильном порядке.
>     
> - `verify()` с `InOrder` — идеальный инструмент для проверки последовательности вызовов в саге.
>     

**Связь с Kafka / асинхронностью:**

> - В асинхронных системах сложно проверить результат сразу. `verify()` позволяет проверить, что событие **было отправлено** в Kafka.
>     

**Связь с Circuit Breaker:**

> - При тестировании Circuit Breaker мы проверяем, что при открытом состоянии **не вызываются** методы зависимого сервиса: `verify(dependency, never()).call()`.
>     

**Связь с Mock vs Spy:**

> - `verify()` работает и с Mock, и с Spy. Но с Spy нужно быть осторожным: если метод реально вызывается, он может иметь побочные эффекты.
>     

**Важный нюанс:**

> - `verify()` проверяет **что было вызвано**, но не проверяет **что вернулось**. Для проверки результата нужен `assertEquals()` или `assertThat()`. Оба подхода дополняют друг друга."
>

#### 7. Закрытие: "Коротко о главном" (15 секунд)

> "Резюмирую: `verify()` — это инструмент **поведенческого тестирования**. Он проверяет, что метод был вызван с нужными аргументами, нужное количество раз и в нужном порядке. Особенно ценен для void-методов, проверки взаимодействий и тестирования саг. Главное правило: не злоупотреблять — если можно проверить состояние, проверяйте состояние. Если нужно проверить взаимодействие — используйте verify."

## Шпаргалка для запоминания (одна страница)

### Основные варианты verify

|Сценарий|Синтаксис|Когда использовать|
|---|---|---|
|**Один вызов**|`verify(mock).method()`|Дефолт, проверяет что вызван хотя бы раз|
|**Ровно N раз**|`verify(mock, times(N)).method()`|Циклы, батчи|
|**Ни разу**|`verify(mock, never()).method()`|Ошибки, условия|
|**Хотя бы раз**|`verify(mock, atLeast(1)).method()`|Асинхронные вызовы|
|**Не более N раз**|`verify(mock, atMost(2)).method()`|Ограничения|
|**Порядок**|`InOrder.verify(mock).method()`|Саги, последовательности|
|**Захват аргумента**|`ArgumentCaptor`|Проверка сложных аргументов|
