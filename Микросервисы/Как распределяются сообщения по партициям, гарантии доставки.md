# Как распределяются сообщения по партициям, гарантии доставки

## 1. Проблема — зачем это существует?

Kafka разбивает топик на партиции для параллелизма и масштабирования. Но при параллельной записи возникает вопрос: в какую партицию попадёт каждое сообщение? И что происходит при сбое: сообщение потеряется, будет обработано дважды или ровно один раз? Выбор стратегии партиционирования и гарантий доставки — это компромисс между производительностью и надёжностью.

## 2. Аналогия

Три способа распределить письма по сортировщикам на почте:
1. **По адресу** (partitioning key): все письма с одного района → один сортировщик — порядок гарантирован
2. **По очереди** (round-robin): равномерная нагрузка, но порядок не гарантирован
3. **Вручную** (explicit partition): директор сам решает — полный контроль

Гарантия доставки — как требование расписаться в получении:
- `acks=0`: отправил и забыл (письмо могло потеряться)
- `acks=1`: расписался получатель на почте (но ещё не доставил адресату)
- `acks=all`: расписались все курьеры-дублёры (надёжно, но медленно)

## 3. Как работает

### Три стратегии выбора партиции

```
Producer отправляет сообщение
    │
    ├─ Указана partition явно? → запись в эту партицию
    │
    ├─ Указан key? → partition = hash(key) % numPartitions
    │              → все сообщения с одним ключом → одна партиция → порядок!
    │
    └─ Нет ключа → Sticky Partitioner (рекомендуется)
                 → Round-Robin (устаревший)
```

**Partitioning Key — самый важный случай:**
```java
producer.send(new ProducerRecord<>("orders", userId, orderJson));
//                                              ↑ ключ
// Все заказы одного пользователя → одна партиция → гарантированный порядок
```

**Sticky Partitioner** (по умолчанию с Kafka 2.4):
- Накапливает батч для одной партиции → потом переходит к следующей
- Меньше сетевых запросов, лучше компрессия
- Лучше Round-Robin по throughput

---

### Гарантии доставки — уровни acks

| acks | Подтверждение | Риск потери | Производительность |
|------|--------------|:-----------:|:-----------------:|
| `0` | Не ждёт | Высокий | Максимальная |
| `1` | Только лидер партиции | Средний (при падении лидера) | Высокая |
| `all` / `-1` | Лидер + все ISR реплики | Минимальный | Снижена |

**ISR (In-Sync Replicas)** — реплики, успевающие синхронизироваться с лидером. `acks=all` ждёт подтверждения только от ISR, не от всех реплик.

```java
Properties props = new Properties();
props.put("acks", "all");              // максимальная надёжность
props.put("enable.idempotence", "true");
props.put("retries", Integer.MAX_VALUE);
```

---

### Семантики доставки

| Семантика | Гарантия | Как достичь |
|-----------|----------|-------------|
| **At-most-once** | Не более одного раза (может потеряться) | `acks=0`, нет ретраев |
| **At-least-once** | Не менее одного раза (возможны дубли) | `acks=all` + ретраи |
| **Exactly-once** | Ровно один раз | Идемпотентный producer + транзакции |

---

### Идемпотентный Producer (producer-side exactly-once)

```java
props.put("enable.idempotence", "true");  // включает автоматически acks=all
```

Как работает:
1. Producer получает уникальный **Producer ID (PID)** от брокера
2. Каждое сообщение получает порядковый номер **(sequence number)**
3. При повторной отправке брокер обнаруживает дубль по `(PID, sequence)` и игнорирует

---

### Транзакции (end-to-end exactly-once)

Для атомарной записи в несколько партиций + синхронизации consumer:

```java
producer.initTransactions();
producer.beginTransaction();
try {
    producer.send(recordA);  // партиция 0
    producer.send(recordB);  // партиция 1
    producer.commitTransaction();   // оба сообщения видны одновременно
} catch (Exception e) {
    producer.abortTransaction();    // оба откатываются
}
```

Consumer должен читать только закоммиченные транзакции:
```java
props.put("isolation.level", "read_committed");
```

## 4. Глубже — что важно знать

**Что сломает порядок сообщений:**
- Нет ключа → Sticky/Round-Robin → сообщения попадают в разные партиции → порядок не гарантирован между партициями
- Ключ есть, но количество партиций изменилось → `hash(key) % newCount` → сообщения пойдут в другую партицию

**At-least-once — самый частый выбор:** `acks=all` + ретраи + consumer обрабатывает идемпотентно. Дубли возможны, но consumer-сторона их обрабатывает (проверка по ID сообщения).

**Практические настройки для надёжной системы:**
```properties
# Producer
acks=all
enable.idempotence=true
max.in.flight.requests.per.connection=5   # для идемпотентности: не более 5
retries=2147483647
compression.type=snappy
linger.ms=10        # ждём 10ms для батча

# Consumer (exactly-once)
enable.auto.commit=false   # коммитим вручную после обработки
isolation.level=read_committed
```

**Когда что использовать:**

| Сценарий | Рекомендация |
|----------|-------------|
| Логи, метрики (потеря не критична) | `acks=0`, at-most-once |
| Стандартные события | `acks=1`, at-least-once |
| Платёжные данные, финансы | `acks=all` + idempotence, exactly-once |
| Атомарные операции через несколько топиков | Транзакции |

## 6. Ответ на собесе (2 минуты)

Распределение по партициям работает так: если producer явно указал партицию — туда и запишем. Если указан ключ — `hash(key) % numPartitions`, все сообщения с одинаковым ключом попадают в одну партицию и гарантируют порядок. Если ключа нет — Sticky Partitioner: накапливает батч для одной партиции, потом переходит к следующей. Это лучше Round-Robin по throughput.

Гарантии доставки определяются параметром `acks`. `acks=0` — producer не ждёт подтверждения, максимальная скорость, но сообщения могут теряться. `acks=1` — подтверждение только от лидера, средняя надёжность. `acks=all` — от лидера и всех ISR-реплик, минимальный риск потери, но медленнее.

Три семантики: at-most-once (может потеряться), at-least-once (могут быть дубли, самый частый выбор), exactly-once (ровно один раз). Для exactly-once включаем идемпотентный producer: брокер отслеживает `(ProducerID, sequenceNumber)` и игнорирует дубли при ретраях. Для атомарных операций через несколько партиций — транзакции.

## Шпаргалка

| Концепция | Аналогия | Ключевой факт |
|-----------|----------|---------------|
| Partitioning key | Индекс по районам | `hash(key) % N` → один consumer → порядок гарантирован |
| `acks=0` | Отправил, не смотрел | Максимальный throughput, потери возможны |
| `acks=1` | Расписался начальник | Стандарт; потеря при падении лидера до репликации |
| `acks=all` | Расписались все копии | Минимальный риск потери; используй с idempotence |
| Exactly-once | Банковский перевод | `enable.idempotence=true` + транзакции + `read_committed` |
