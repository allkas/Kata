# Распределённые транзакции, саги

## 1. Проблема — зачем это существует?

В монолите бизнес-операция, затрагивающая несколько таблиц, оборачивается в одну ACID-транзакцию: либо всё сохранилось, либо ничего. В микросервисах у каждого сервиса своя БД — `@Transactional` между ними не работает. Как провести операцию «создать заказ + списать деньги + зарезервировать товар», если это три разные базы? Saga — паттерн, заменяющий ACID в распределённых системах.

## 2. Аналогия

Планирование поездки: нужно купить билет на самолёт, забронировать отель, заказать трансфер. В турагентстве всё единым пакетом — атомарно (монолит). Самостоятельно — три отдельных бронирования. Если отель недоступен — нужно вручную сдать билет и отменить такси.

Saga — это план с компенсациями: у каждого шага есть «откат» на случай провала следующего.

## 3. Как работает

### Механика Saga

Сага = последовательность **локальных транзакций** в разных сервисах. Если шаг N терпит неудачу — выполняются **компенсирующие транзакции** для всех предыдущих шагов в обратном порядке.

```
Шаг 1: Order Service    → создать заказ (PENDING)
Шаг 2: Payment Service  → списать деньги
Шаг 3: Inventory Service→ зарезервировать товар
Шаг 4: Order Service    → обновить статус (CONFIRMED)

Если Шаг 3 упал:
  Компенсация 2: Payment Service  → вернуть деньги
  Компенсация 1: Order Service    → отменить заказ (CANCELLED)
```

---

### Хореография — событийный подход

Нет центрального координатора. Сервисы общаются через события (Kafka/RabbitMQ). Каждый сервис знает, на какие события реагировать и что публиковать.

```
OrderService  ──publish──► OrderCreated
                                    │
PaymentService ◄──consume──────────┘
PaymentService ──publish──► PaymentProcessed
                                    │
InventoryService ◄──consume────────┘
InventoryService ──publish──► InventoryReserved / InventoryFailed
                                    │
(если Failed)                       ▼
PaymentService ◄──consume── InventoryFailed → откат платежа
```

**Плюсы:** нет единой точки отказа, слабое связывание, легко добавить новый подписчик.

**Минусы:** сложно отследить состояние конкретной саги, риск циклических зависимостей, тяжело отлаживать.

---

### Оркестрация — дирижёр

Центральный **Saga Coordinator** знает весь сценарий и командует участниками.

```java
// Псевдокод оркестратора
class OrderSagaOrchestrator {

    void execute(Order order) {
        try {
            paymentService.charge(order);     // Шаг 1
            inventoryService.reserve(order);  // Шаг 2
            orderService.confirm(order);      // Шаг 3
        } catch (InventoryException e) {
            paymentService.refund(order);     // Компенсация
            orderService.cancel(order);
        }
    }
}
```

**Плюсы:** вся логика в одном месте, легко менять порядок, понятное состояние.

**Минусы:** единая точка отказа, оркестратор рискует превратиться в «монолит внутри микросервисов».

---

### Сравнение подходов

| | Хореография | Оркестрация |
|--|-------------|-------------|
| Аналогия | Народный танец (нет дирижёра) | Оркестр с дирижёром |
| Координация | Через события | Центральный координатор |
| Связанность | Слабая | Умеренная |
| Отладка | Сложна (цепочка в N сервисах) | Проще (состояние в одном месте) |
| SPOF | Нет | Да (оркестратор) |
| Когда | Мало шагов, простые связи | Сложный процесс, ветвления |

## 4. Глубже — что важно знать

**Идемпотентность обязательна.** В сагах сообщения могут прийти дважды (сетевые ретраи, перезапуск). Каждый шаг должен быть идемпотентным — повторный вызов не должен давать дополнительный эффект. Проверка: «это списание уже было проведено?» через уникальный `transactionId`.

**Eventual consistency — не баг, а фича.** Между шагами система может быть временно несогласована (деньги списаны, товар ещё не зарезервирован). Это нормально. Saga обеспечивает **итоговую согласованность** — в конце система придёт к консистентному состоянию.

**Kafka идеальна для хореографии:** хранит события, позволяет replay, гарантирует at-least-once доставку.

**Инструменты оркестрации:** Camunda, Axon Framework, Temporal — предоставляют готовые Saga-движки с state machine.

**Saga vs 2PC (двухфазный коммит):** 2PC блокирует ресурсы на время подготовки и подтверждения — плохо масштабируется, SPOF координатора. Saga не блокирует, компенсирует после факта — лучший выбор для микросервисов.

## 6. Ответ на собесе (2 минуты)

В монолите атомарность даёт ACID-транзакция. В микросервисах каждый сервис — отдельная БД, `@Transactional` не работает. Saga — паттерн, который решает это: цепочка локальных транзакций, у каждой из которых есть компенсирующая операция на случай провала следующих шагов.

Два подхода реализации. **Хореография**: сервисы общаются через события в Kafka, нет дирижёра — слабое связывание, но сложно отлаживать. **Оркестрация**: центральный координатор управляет всеми шагами — проще отслеживать состояние, но появляется единая точка отказа.

Важные нюансы. Каждый шаг саги должен быть **идемпотентным** — сообщения могут прийти дважды. Система живёт в **eventual consistency** — временная несогласованность данных между шагами это нормально.

Выбор: для простых процессов (3–4 шага) — хореография на Kafka. Для сложных с ветвлениями — оркестрация (Camunda, Axon).

## Шпаргалка

| Концепция | Аналогия | Ключевое |
|-----------|----------|---------|
| Сага | Поездка с пересадками | Локальные транзакции + компенсации |
| Компенсация | Возврат билета со штрафом | Откат предыдущих шагов при ошибке |
| Хореография | Народный танец | Сервисы → события, нет дирижёра |
| Оркестрация | Дирижёр | Координатор командует шагами |
| Eventual consistency | «В итоге всё хорошо» | Временная несогласованность допустима |
| Идемпотентность | Дважды нажатая кнопка | Повтор шага не ломает результат |
