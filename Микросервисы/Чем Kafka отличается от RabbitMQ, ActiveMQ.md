# Чем Kafka отличается от RabbitMQ, ActiveMQ

## 1. Проблема — зачем это существует?

Все три — брокеры сообщений, но созданы для разных задач. Выбор неправильного инструмента ведёт к серьёзным проблемам: использование RabbitMQ там, где нужен replay событий (например, аудит) — данные потеряются после ACK. Kafka там, где нужна маршрутизация по сложным правилам — код будет перегружен. Вопрос "чем отличаются" — это вопрос о том, когда что выбирать и почему.

## 2. Аналогия

**Kafka** — это **библиотечный архив с хронологическими записями**. Каждое событие записывается в журнал. Любой читатель может открыть журнал с любой страницы и прочитать всё с нужного момента. Записи не удаляются после прочтения — они хранятся по расписанию (retention).

**RabbitMQ** — это **курьерская служба**. Посылка передаётся конкретному получателю по правилам маршрутизации. После доставки и подтверждения (ACK) посылка уничтожается. Никакого архива — доставил и забыл.

## 3. Как работает

**Главное архитектурное различие:**

| Характеристика | Kafka | RabbitMQ / ActiveMQ |
|----------------|-------|---------------------|
| Модель хранения | Распределённый append-only лог | Очередь (удаляется после ACK) |
| Модель доставки | Pull — consumer сам тянет по offset | Push — брокер толкает consumer |
| Повторное чтение | ✅ Да (replay с любого offset) | ❌ Нет |
| Хранение | Retention: дни/недели, на диске | До ACK, обычно в RAM |
| Маршрутизация | Простая: топик → партиции по ключу | Сложная: exchanges (direct/topic/fanout/headers) + binding keys |
| Производительность | 100k–1M+ msg/sec | 10–50k msg/sec |
| Задержка | >10 мс (batch-ориентирован) | <1 мс (push-ориентирован) |
| Порядок сообщений | Гарантирован внутри партиции | Не гарантирован (если >1 consumer) |
| Протоколы | Kafka Protocol | AMQP, MQTT, STOMP, HTTP |

**Механика Kafka (Pull-based):**
```
Producer → [Топик: Partition 0] → Consumer A (читает с offset 42)
                [Partition 1] → Consumer B (читает с offset 18)
                [Partition 2] → Consumer C (читает с offset 5)
```
Consumer сам контролирует позицию. Можно перемотать назад — перечитать историю событий.

**Механика RabbitMQ (Push-based):**
```
Producer → Exchange → Binding (routing key) → Queue → Consumer
                                             → Queue → Consumer 2
```
После `ack` сообщение удалено. Сложная маршрутизация по topic-маскам (`order.#`), headers, direct.

## 4. Глубже — что важно знать

**Когда Kafka:**
- Нужен **replay** — перечитать историю (аудит, event sourcing, отладка)
- **Много независимых потребителей** одного события (аналитика + ML + бэкап)
- **Высокая пропускная способность** — миллионы событий/сек (логи, метрики, IoT)
- **Потоковая обработка** — Kafka Streams, Flink, Spark
- **Data pipeline** между микросервисами

**Когда RabbitMQ:**
- **Сложная маршрутизация** — "только заказы из РФ на сумму > 1000 руб."
- **Низкая задержка** — нужен ответ < 10 мс (RPC-style)
- **Task queues** — обработка изображений, отправка email, background jobs
- **Competing consumers** — несколько воркеров берут задачи из одной очереди
- **IoT-устройства** — MQTT-протокол

**Частые ошибки:**
- Kafka для RPC — задержка неприемлемо высокая, ответ на конкретный запрос сложно сопоставить
- RabbitMQ для event sourcing — после ACK данные теряются, нет истории
- Kafka без DLQ — если consumer падает на конкретном сообщении, он застревает навсегда (poison pill)

**Гибридные архитектуры:** часто Kafka + RabbitMQ в одной системе — Kafka для event log и аналитики, RabbitMQ для task queues и уведомлений.

## 6. Ответ на собесе (2 минуты)

> "Kafka и RabbitMQ — это брокеры сообщений, но созданы для разных задач, и выбор между ними — это архитектурное решение.
>
> Ключевое различие — в модели хранения. Kafka — это распределённый append-only лог. Каждое событие записывается и хранится по retention-политике: дни, недели. Любой consumer может перечитать историю с любого момента. Аналогия — библиотечный архив с хронологическими записями.
>
> RabbitMQ — это очередь. После того как consumer подтвердил получение (ACK), сообщение удаляется. Зато RabbitMQ даёт очень низкую задержку (< 1 мс), push-модель и сложную маршрутизацию через exchanges — можно доставлять сообщения по ключу, по маске, по заголовкам.
>
> На практике: если мне нужен аудит-лог, event sourcing, или много независимых потребителей одного события — выбираю Kafka. Если нужна task queue с background workers, RPC-стиль, или маршрутизация по сложным правилам — выбираю RabbitMQ. В крупных системах часто используют оба: Kafka для event-driven архитектуры, RabbitMQ для task queues.
>
> Важный нюанс: Kafka требует настройки DLQ вручную (через отдельный топик), иначе застрявшее сообщение ('poison pill') заблокирует consumer навсегда."

## Шпаргалка

| Концепция | Аналогия | Ключевой факт |
|-----------|----------|---------------|
| Kafka | Библиотечный архив, журнал событий | Pull-модель, replay, долгое хранение |
| RabbitMQ | Курьерская служба | Push-модель, после ACK удаляет, сложная маршрутизация |
| ActiveMQ | Старая почта (AMQP/JMS) | Как RabbitMQ, но старше; Spring использует JMS API |
| Pull vs Push | Читатель сам идёт в библиотеку vs курьер доставляет домой | Kafka — pull, RabbitMQ — push |
| Replay | Перечитать газету с начала | Только Kafka |
| Exchange (RabbitMQ) | Сортировочный центр почты | Маршрутизация по ключу/маске/заголовкам |
