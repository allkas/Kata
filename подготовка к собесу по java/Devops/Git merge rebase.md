# Git merge / rebase
## 1. merge vs rebase
- **merge** — создаёт merge commit, сохраняет историю.
- **rebase** — переписывает историю, линейная история.
## 2. fetch vs pull
- **fetch** — только загружает изменения.
- **pull** = fetch + merge.
## 3. cherry-pick
Применяет конкретный коммит из другой ветки.
## 6. Ответ на собесе (2 минуты)

> "Merge и rebase — два способа интегрировать изменения из одной ветки в другую.
>
> Merge создаёт merge commit, который объединяет две ветки. История сохраняется честно — видно, где была сделана ветка, где слита. Минус: история засоряется merge commits при активной командной разработке.
>
> Rebase переписывает историю: коммиты ветки 'переклеиваются' поверх целевой ветки так, как будто разработка шла линейно. История чище, но золотое правило — rebase только локальных веток. Rebase на уже запушенной ветке ломает историю у других разработчиков.
>
> fetch только скачивает изменения с remote, не изменяя рабочую директорию. pull = fetch + merge. Безопаснее делать fetch, посмотреть diff, затем merge или rebase.
>
> cherry-pick — перенос конкретного коммита в другую ветку. Создаёт новый коммит с тем же содержимым, но новым hash."
