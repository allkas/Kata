# Garbage Collector JVM

## 1. Проблема — зачем это существует?

В C/C++ разработчик вручную управляет памятью: `malloc` / `free`. Забыл освободить — утечка памяти, освободил дважды — crash. Java решает это автоматически: GC отслеживает, какие объекты больше не нужны (недостижимы), и освобождает память. Разработчик думает о логике, не о памяти.

## 2. Аналогия

JVM — это офис. **Stack** — это рабочий стол каждого сотрудника (потока): там лежат только текущие задачи, после окончания работы стол чистится сам. **Heap** — это общий склад: там хранятся большие объекты, доступные всем. **GC** — уборщик, который периодически обходит склад и выбрасывает то, на что никто больше не ссылается.

## 3. Как работает

### Память JVM

```
JVM Memory
├── Stack (per thread)     — примитивы, ссылки, локальные переменные, фреймы вызовов
├── Heap                   — все объекты, создаваемые через new
│   ├── Young Generation
│   │   ├── Eden           — новые объекты
│   │   ├── Survivor 0 (S0)
│   │   └── Survivor 1 (S1)
│   └── Old Generation     — долгоживущие объекты
├── Metaspace (Java 8+)    — метаданные классов (раньше PermGen)
└── Code Cache             — JIT-скомпилированный код
```

**Stack vs Heap:**
| | Stack | Heap |
|---|---|---|
| **Хранит** | Примитивы, ссылки, локальные переменные | Объекты (`new`) |
| **Область** | Per-thread (у каждого потока свой) | Общий для всех потоков |
| **Скорость** | Очень быстрый (LIFO) | Медленнее |
| **Управление** | Автоматически (pop при выходе из метода) | GC |
| **Ошибка** | `StackOverflowError` (бесконечная рекурсия) | `OutOfMemoryError` |

### Жизненный цикл объекта:
```
1. new Object() → попадает в Eden
2. Minor GC → выжившие переходят в S0/S1 (age++)
3. После N minor GC (обычно 15) → переходят в Old Generation
4. Major/Full GC → очищает Old Generation (дольше, Stop-The-World)
```

### Виды GC в Java:
| GC | Особенность | Когда использовать |
|---|---|---|
| **Serial GC** | Один поток, Stop-The-World | Маленькие приложения |
| **Parallel GC** | Несколько потоков, паузы | Batch-задачи (пропускная способность) |
| **G1 GC** (default Java 9+) | Регионы, предсказуемые паузы | Большинство приложений |
| **ZGC** | Паузы <1мс | Low-latency системы |

**Объект недостижим** когда нет ссылок от GC roots:
- Локальные переменные (stack)
- Статические поля
- JNI-ссылки
- Активные потоки

## 4. Глубже — типы ссылок и ошибки

### WeakReference / SoftReference / WeakHashMap:
```java
// SoftReference — удаляется при нехватке памяти (для кэшей)
SoftReference<byte[]> cache = new SoftReference<>(new byte[1024]);

// WeakReference — удаляется при ближайшем GC
WeakReference<Object> weak = new WeakReference<>(someObject);
someObject = null; // после GC weak.get() вернёт null

// WeakHashMap — ключи слабые ссылки: запись удаляется, когда ключ недостижим
WeakHashMap<Object, String> map = new WeakHashMap<>();
```
**Применение `WeakReference`:** кэши, где объект можно пересоздать; listener'ы, чтобы не удерживать подписчиков.

### OutOfMemoryError vs StackOverflowError:
```java
// OutOfMemoryError — heap переполнен
List<byte[]> leak = new ArrayList<>();
while (true) leak.add(new byte[1024 * 1024]); // java.lang.OutOfMemoryError: Java heap space

// StackOverflowError — бесконечная рекурсия
void infinite() { infinite(); } // java.lang.StackOverflowError
```

**Диагностика OOM:**
- JVM-флаг: `-XX:+HeapDumpOnOutOfMemoryError` — дамп при падении
- Анализ: Eclipse MAT, VisualVM
- Частые причины: утечка кэша, event listener'ы без отписки, статические коллекции

## 6. Ответ на собесе (2 минуты)

> "JVM-память делится на Stack и Heap. **Stack** — персональный для каждого потока: хранит локальные переменные, примитивы и ссылки. Очищается автоматически при выходе из метода. **Heap** — общий склад для всех объектов, созданных через `new`. Им управляет GC.
>
> **Heap делится на поколения.** Новые объекты идут в Eden. После Minor GC выжившие перемещаются в Survivor zones. Долгоживущие объекты (после ~15 GC) попадают в Old Generation. Это оптимизация: большинство объектов «умирают молодыми», Minor GC быстрый и частый.
>
> **GC по умолчанию с Java 9 — G1.** Он делит Heap на регионы, даёт предсказуемые паузы. Для low-latency систем есть ZGC с паузами менее миллисекунды.
>
> **`WeakReference`** я использую для кэшей: объект держится в памяти пока нужен, GC его удалит при давлении на память. `WeakHashMap` — ключи слабые, запись удаляется вместе с ключом — удобно для кэша вычислений.
>
> **OOM диагностика:** включаю `-XX:+HeapDumpOnOutOfMemoryError`, анализирую дамп в MAT. Чаще всего это утечка в кэше или в статической коллекции."

## Шпаргалка

| Концепция | Суть | Ошибка |
|-----------|------|--------|
| **Stack** | Per-thread, примитивы + ссылки | `StackOverflowError` |
| **Heap (Young)** | Eden + Survivors, новые объекты, Minor GC | — |
| **Heap (Old)** | Долгоживущие, Major/Full GC (медленно) | `OutOfMemoryError` |
| **SoftReference** | Умирает при нехватке памяти | Для кэшей |
| **WeakReference** | Умирает при ближайшем GC | Для listener'ов, кэшей |
| **WeakHashMap** | Ключи — weak refs | Кэш с автоочисткой |
| **G1 GC** | Default Java 9+, предсказуемые паузы | Большинство приложений |
