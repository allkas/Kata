# SQL индексы JOIN нормализация

## 1. Проблема — зачем это существует?

Таблица на миллион строк без индекса — это Seq Scan: база читает каждую строку одну за другой. O(n). Индекс — отдельная структура данных, которая позволяет найти нужные строки за O(log n). JOIN — единственный способ собрать данные из нескольких нормализованных таблиц в один результат. Нормализация избавляет от дублирования и аномалий обновления.

## 2. Аналогия

**Индекс** — как предметный указатель в конце книги. Без него: листать все 500 страниц (Seq Scan). С ним: открыть указатель, найти тему, перейти на страницу (Index Scan).

**JOIN** — как соединение строк из двух журналов по общему полю. INNER — только записи, которые есть в обоих. LEFT — все из первого журнала, из второго — если есть, иначе прочерк.

## 3. Индексы

### Как B-tree индекс работает механически:
```
Запрос: WHERE id = 42

БЕЗ индекса:
  [1][5][23][42][67][...] — читаем каждую строку: O(n)

С B-tree индексом:
         [30]
        /    \
     [10]    [50]
    /    \   /   \
  [5]  [20][40]  [60]
→ 42 > 30 → идём вправо → 42 < 50 → идём влево → нашли [40] → следующий узел → [42]: O(log n)
```

Индекс хранит **копию индексируемого столбца** + указатель на строку в основной таблице (heap).

### Виды индексов:
| Тип | Применение | Операции |
|-----|-----------|----------|
| **B-tree** (default) | Числа, строки, даты | `=`, `<`, `>`, `BETWEEN`, `LIKE 'abc%'` |
| **Hash** | Только точное равенство | `=` (быстрее B-tree для равенства) |
| **GIN** | Массивы, JSONB, полнотекстовый поиск | `@>`, `&&`, `@@` |
| **GiST** | Геометрия, диапазоны | Перекрытие, включение |

### Составной (composite) индекс — правило левого префикса:
```sql
CREATE INDEX idx ON orders (customer_id, status, created_at);

-- Используется:
WHERE customer_id = 1                          -- ✅ левый префикс
WHERE customer_id = 1 AND status = 'PAID'      -- ✅
WHERE customer_id = 1 AND status = 'PAID' AND created_at > '2024-01-01' -- ✅

-- НЕ используется:
WHERE status = 'PAID'                          -- ❌ не с левого
WHERE created_at > '2024-01-01'               -- ❌
```

### Covering index — поиск только в индексе:
```sql
-- Если SELECT выбирает только индексируемые столбцы:
CREATE INDEX idx ON users (email, name);
SELECT name FROM users WHERE email = 'a@b.com'; -- Index Only Scan — таблицу не трогает
```

## 4. Когда индекс вреден

| Ситуация | Почему |
|----------|--------|
| **Много INSERT/UPDATE/DELETE** | Каждое изменение строки обновляет все индексы на неё |
| **Низкая селективность** | `WHERE is_active = true` (90% строк) — оптимизатор выберет Seq Scan |
| **Малые таблицы** | Overhead индекса превышает выгоду |
| **`LIKE '%pattern'`** | Ведущий wildcard — B-tree не может использовать |
| **Функция на столбце** | `WHERE UPPER(name) = 'IVAN'` — индекс на `name` не используется. Нужен функциональный индекс: `CREATE INDEX ON users (UPPER(name))` |

## 5. JOIN'ы

```
Таблица A: [1,Alice] [2,Bob]  [3,Carol]
Таблица B: [1,HR]    [2,IT]   [4,Ops]

INNER JOIN: [1,Alice,HR] [2,Bob,IT]          — только совпадения
LEFT JOIN:  [1,Alice,HR] [2,Bob,IT] [3,Carol,NULL] — все из A
RIGHT JOIN: [1,Alice,HR] [2,Bob,IT] [NULL,NULL,Ops] — все из B
FULL JOIN:  все строки из обеих, NULL где нет пары
```

```sql
-- INNER JOIN: заказы только для существующих клиентов
SELECT o.id, c.name FROM orders o
INNER JOIN customers c ON o.customer_id = c.id;

-- LEFT JOIN: все клиенты, даже без заказов
SELECT c.name, o.id FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id;
-- Клиенты без заказов: o.id = NULL → WHERE o.id IS NULL → найти клиентов без заказов
```

## 6. WHERE vs HAVING

```sql
-- WHERE: фильтрует строки ДО группировки (работает с отдельными строками)
SELECT department, COUNT(*)
FROM employees
WHERE salary > 50000          -- сначала отбрасываем дешёвых сотрудников
GROUP BY department;

-- HAVING: фильтрует группы ПОСЛЕ GROUP BY (работает с агрегатами)
SELECT department, COUNT(*)
FROM employees
GROUP BY department
HAVING COUNT(*) > 5;          -- оставляем только крупные отделы

-- Оба вместе:
SELECT department, AVG(salary)
FROM employees
WHERE hire_date > '2020-01-01'     -- строки
GROUP BY department
HAVING AVG(salary) > 60000;        -- группы
```

**Правило:** если фильтр использует агрегатную функцию (`COUNT`, `SUM`, `AVG`) — это `HAVING`. Иначе — `WHERE`.

## 7. Нормализация

### 1NF — атомарность:
Каждая ячейка — одно значение. Нет повторяющихся групп.
```
❌ до: phones = "111, 222, 333"  
✅ после: отдельная строка на каждый телефон
```

### 2NF — нет частичных зависимостей:
Применимо к составным PK. Каждый non-key атрибут зависит от **всего** ключа, а не от его части.
```
❌ Таблица: (student_id, course_id, student_name, grade)
   student_name зависит только от student_id — частичная зависимость
✅ Разделить: Students(student_id, name) + Enrollments(student_id, course_id, grade)
```

### 3NF — нет транзитивных зависимостей:
Non-key атрибут не должен зависеть от другого non-key атрибута.
```
❌ Таблица: (order_id, customer_id, customer_city, customer_zip)
   customer_city зависит от customer_zip, а не от order_id напрямую
✅ Разделить: Orders(order_id, customer_id) + Customers(customer_id, city, zip)
```

### Денормализация:
Намеренное нарушение нормальных форм ради производительности чтения. Пример: хранить `total_amount` в заказе вместо пересчёта каждый раз через JOIN. Цена — сложность поддержания консистентности при обновлениях.

## 8. Оптимизация запросов (EXPLAIN ANALYZE)

```sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;
```

**Что искать в плане:**
| Узел | Что значит | Хорошо/Плохо |
|------|-----------|--------------|
| `Seq Scan` | Полный перебор таблицы | Плохо на больших таблицах |
| `Index Scan` | Поиск через индекс + обращение к таблице | Хорошо |
| `Index Only Scan` | Только индекс, таблицу не трогает | Отлично |
| `Nested Loop` | JOIN через вложенные циклы | OK для малых таблиц |
| `Hash Join` | JOIN через хеш-таблицу | Хорошо для больших |
| `rows=100 actual rows=10000` | Оценка плохая → устаревшая статистика | Запустить `ANALYZE` |

**Частые причины медленных запросов:**
- Нет индекса на колонку в WHERE/JOIN
- `LIKE '%prefix'` — ведущий wildcard
- Функция на индексируемой колонке
- Устаревшая статистика (запустить `ANALYZE` или `VACUUM ANALYZE`)
- SELECT * — тянуть ненужные колонки

## 10. Ответ на собесе (2 минуты)

> "Индекс — это отдельная структура данных (B-tree по умолчанию), которая хранит копию индексируемых столбцов в отсортированном виде с указателями на строки. Поиск — O(log n) вместо O(n) полного перебора.
>
> **Когда индекс вреден:** при высокой частоте записей — каждый INSERT/UPDATE обновляет все индексы. При низкой селективности — если 90% строк удовлетворяют условию, оптимизатор всё равно выберет Seq Scan. Функция на индексируемой колонке сломает использование индекса.
>
> **JOIN'ы:** INNER — только совпадения с обеих сторон. LEFT — все из левой таблицы, из правой NULL если нет пары. FULL — все строки из обеих. Частый паттерн: LEFT JOIN + WHERE right.id IS NULL — найти записи без связанной сущности.
>
> **WHERE vs HAVING:** WHERE фильтрует строки до группировки, HAVING — группы после. Если используешь агрегат в условии — это HAVING.
>
> **Нормализация:** 1NF — атомарные значения, 2NF — нет частичных зависимостей от составного ключа, 3NF — нет транзитивных зависимостей между non-key атрибутами. Денормализую осознанно ради производительности чтения."

## Шпаргалка

| Концепция | Суть | Ключевой факт |
|-----------|------|---------------|
| **B-tree индекс** | Отсортированное дерево | O(log n) поиск |
| **Составной индекс** | Несколько колонок | Правило левого префикса |
| **Вреден индекс** | Запись-интенсивные таблицы, низкая селективность | Функция на колонке тоже ломает |
| **INNER JOIN** | Только совпадения | NULL строки отбрасываются |
| **LEFT JOIN** | Все из левой | NULL если нет пары справа |
| **WHERE** | До GROUP BY | Строки |
| **HAVING** | После GROUP BY | Агрегаты |
| **1NF** | Атомарность | Нет массивов в ячейке |
| **2NF** | Нет частичных зависимостей | При составном PK |
| **3NF** | Нет транзитивных зависимостей | A→B→C → разделить |
| **EXPLAIN ANALYZE** | Seq Scan — плохо | Смотреть actual rows vs estimate |
