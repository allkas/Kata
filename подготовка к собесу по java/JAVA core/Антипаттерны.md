# Антипаттерны (Anti-patterns)

## 1. Проблема — зачем это существует?

Паттерны — проверенные решения. Антипаттерны — типичные ошибки проектирования, которые выглядят разумно, но на практике создают проблемы: трудный рефакторинг, хрупкий код, невозможность тестирования. Знать антипаттерны важно: на собесе могут спросить, что плохо в предложенном коде, а не только как написать хорошо.

## 2. Аналогия

Антипаттерн — как народный миф, который передаётся из поколения в поколение, потому что звучит логично: «пей больше воды при простуде», «чем больше код, тем серьёзнее программист». Звучит правдоподобно, делается с добрыми намерениями — но ухудшает результат.

## 3. Как работает

### God Object (Божественный объект)

Один класс, который знает всё и делает всё. Нарушает SRP и принцип разделения ответственностей.

```java
// ПЛОХО — God Object
public class UserManager {
    public User createUser(...) { ... }
    public void sendWelcomeEmail(...) { ... }
    public boolean validatePassword(...) { ... }
    public void saveToDatabase(...) { ... }
    public void logActivity(...) { ... }
    public String generateToken(...) { ... }
    public void sendPushNotification(...) { ... }
}

// ХОРОШО — разделяем по ответственностям
public class UserService      { public User createUser(...) { ... } }
public class EmailService     { public void sendWelcomeEmail(...) { ... } }
public class PasswordEncoder  { public boolean validate(...) { ... } }
public class UserRepository   { public void save(...) { ... } }
public class TokenService     { public String generate(...) { ... } }
```

**Признаки:** класс на 500+ строк, десятки методов без очевидной общей темы, все другие классы зависят от него.

---

### Big Ball of Mud (Большой комок грязи)

Система без архитектуры: все компоненты зависят друг от друга произвольно, нет модульности, нет четких границ. Возникает при хаотичном росте без рефакторинга.

**Признаки:**
- Невозможно изменить один класс, не сломав другой
- Нельзя тестировать компоненты изолированно
- Новый разработчик не может понять, с чего начать

**Решение:** постепенное выделение слоёв (Controller → Service → Repository), ограниченные контексты (DDD Bounded Context), модули с явными API.

---

### Poltergeists (Полтергейсты)

Классы, которые существуют только чтобы вызвать другой класс или передать данные. Избыточное делегирование без добавления ценности.

```java
// ПЛОХО — PoltergeistController только вызывает PoltergeistHelper
public class PoltergeistController {
    private PoltergeistHelper helper = new PoltergeistHelper();
    public void process() { helper.doActualWork(); }
}

public class PoltergeistHelper {
    public void doActualWork() { /* здесь реальная логика */ }
}

// ХОРОШО — убрать лишний уровень, логика прямо в нужном месте
```

**Признаки:** класс не имеет состояния, существует краткосрочно, делегирует 100% вызовов.

### Singleton как антипаттерн

GoF Singleton сам по себе — не всегда плохо. Но реализация через статическое поле создаёт проблемы:

```java
// ПРОБЛЕМА — статический Singleton
public class Config {
    private static Config instance = new Config();
    public static Config getInstance() { return instance; }
    public String getDbUrl() { return "jdbc:..."; }
}

// Это делает код:
// 1. Нетестируемым — нельзя подменить Config в тестах без reflection
// 2. Жёстко связанным — зависимость скрыта внутри метода
// 3. Нарушающим DIP — класс сам создаёт свою зависимость
public class UserService {
    public User getUser(int id) {
        String url = Config.getInstance().getDbUrl(); // скрытая зависимость!
        ...
    }
}

// ХОРОШО — DI через конструктор (Spring делает это автоматически)
public class UserService {
    private final Config config;
    public UserService(Config config) { this.config = config; } // явная зависимость
}
```

**Вывод:** Singleton — паттерн создания объекта. В Spring используй `@Component` + `scope=singleton` — контейнер управляет lifecycle, DI остаётся возможным.

### Introduced Complexity (Введённая сложность)

Добавление абстракций, фреймворков, паттернов там, где они не нужны.

```java
// ПЛОХО — AbstractSingletonProxyFactoryBeanFactoryBean для простого расчёта
public class TaxCalculatorAbstractFactory {
    public TaxCalculatorStrategy createStrategy(Region region) {
        return TaxCalculatorStrategyRegistry.getInstance()
            .lookupStrategy(region.getCode());
    }
}

// ХОРОШО — простой switch или Map
public double calculateTax(double amount, Region region) {
    return switch (region) {
        case US -> amount * 0.1;
        case EU -> amount * 0.2;
        default -> 0;
    };
}
```

**Признаки:** фреймворк для решения задачи в 5 строк; паттерн «на будущее»; интерфейс с единственной реализацией без очевидных планов расширения.

### Анемичная доменная модель (Anemic Domain Model)

Объекты домена — это DTO: только поля и геттеры/сеттеры. Вся бизнес-логика в сервисном слое. Противоречит принципам OOP.

```java
// ПЛОХО — анемичная модель
public class Order {
    private List<Item> items;
    private OrderStatus status;
    // только геттеры/сеттеры, нет логики
}

public class OrderService {
    public void complete(Order order) {
        if (order.getItems().isEmpty()) throw new IllegalStateException();
        order.setStatus(OrderStatus.COMPLETED); // логика в сервисе, не в домене
    }
}

// ХОРОШО — богатая доменная модель
public class Order {
    private List<Item> items;
    private OrderStatus status;

    public void complete() {
        if (items.isEmpty()) throw new IllegalStateException("Order has no items");
        this.status = OrderStatus.COMPLETED; // логика в объекте
    }

    public boolean canBeCompleted() { return !items.isEmpty(); }
}
```

**Где применимо:** анемичная модель оправдана в простых CRUD-приложениях; богатая доменная модель — в сложных доменах с нетривиальными правилами.

## 4. Глубже — что важно знать

**Как паттерны превращаются в антипаттерны:**
- Singleton → антипаттерн, если создаёт скрытые зависимости
- Factory → антипаттерн, если фабрика сложнее создаваемых объектов
- Decorator → антипаттерн, если цепочка из 10 декораторов нечитаема
- Observer → антипаттерн, если события создают неочевидные циклы зависимостей

**Практические правила:**
- YAGNI (You Ain't Gonna Need It) — не добавляй абстракции «на будущее»
- KISS (Keep It Simple, Stupid) — простейшее решение, которое работает
- DRY (Don't Repeat Yourself) — но не жертвуй читаемостью ради устранения дублирования

## 6. Ответ на собесе (2 минуты)

> "Антипаттерны — типичные ошибки проектирования, которые выглядят разумно, но создают проблемы.
>
> **God Object** — класс, который знает всё и делает всё. Признак — тысячи строк, все зависят от него. Решение — выделить ответственности в отдельные классы.
>
> **Singleton как антипаттерн** — не сам по себе, а реализация через статическое поле. Делает код нетестируемым: нельзя подменить в тестах. В Spring вместо статического Singleton использую `@Component` + DI.
>
> **Анемичная доменная модель** — объекты без логики, только геттеры. Вся бизнес-логика уходит в сервисы. Для простого CRUD — нормально, для сложного домена — нарушение OOP.
>
> Общее правило: YAGNI и KISS. Добавляю абстракции только когда они нужны сейчас, а не «на будущее»."

## Шпаргалка

| Антипаттерн | Симптом | Решение |
|-------------|---------|---------|
| God Object | Класс на 1000+ строк, все зависят от него | SRP, разделить по ответственностям |
| Big Ball of Mud | Изменение одного ломает другое | Слои, модули, Bounded Context |
| Poltergeists | Класс только делегирует без ценности | Удалить лишний уровень |
| Singleton-антипаттерн | Нельзя подменить в тестах | DI через контейнер (Spring) |
| Introduced Complexity | Фреймворк для задачи в 5 строк | YAGNI, KISS |
| Anemic Domain Model | Домен = DTO, логика в сервисах | Перенести логику в объект |
