# Интерфейс vs абстрактный класс

## 1. Проблема — зачем это существует?

Java поддерживает только одиночное наследование классов. При этом нужны два разных инструмента:
- Описать **контракт** (что объект умеет) без привязки к иерархии
- **Переиспользовать** общую логику и состояние в группе связанных классов

Для этого появились интерфейс и абстрактный класс — с разной философией и разными ограничениями.

## 2. Аналогия

**Абстрактный класс** — шаблон машины на заводе. Уже есть рама, двигатель, базовая электрика. Каждая конкретная модель (BMW, Toyota) наследует основу и дорабатывает под себя. Но наследоваться можно только от **одного** шаблона.

**Интерфейс** — сертификат соответствия стандарту: «Пожароустойчив», «Водонепроницаем», «Поддерживает USB-C». Одно устройство может иметь сколько угодно сертификатов. Каждый производитель реализует их по-своему.

## 3. Как работает

| | Интерфейс | Абстрактный класс |
|---|---|---|
| Наследование | Множественное (`implements A, B, C`) | Одиночное (`extends A`) |
| Поля | Только `public static final` | Любые |
| Конструктор | Нет | Есть |
| Методы (до Java 8) | Только `abstract` | Любые |
| Методы (Java 8+) | `abstract` + `default` + `static` | Любые |
| Состояние (state) | Нет | Есть |
| Модификаторы доступа | Методы — `public` | Любые |

**Java 8+ изменения:**
```java
interface Validator {
    boolean validate(String input); // abstract

    default void logError(String msg) { // реализация в интерфейсе
        System.err.println("Validation error: " + msg);
    }

    static Validator nonEmpty() { // фабричный метод
        return input -> input != null && !input.isEmpty();
    }
}
```
`default`-методы нужны для **эволюции API**: добавить метод в интерфейс без поломки всех реализаций.

## 4. Глубже — когда что выбирать

**Абстрактный класс — когда:**
- Подклассы имеют общее **состояние** (поля) и общую **логику**
- Классы связаны отношением IS-A и делят реализацию
- Примеры: `AbstractList`, `HttpServlet`, `AbstractController`

**Интерфейс — когда:**
- Нужен **контракт** без привязки к иерархии
- Объект должен играть несколько **ролей** одновременно
- Нет общего состояния
- Примеры: `Runnable`, `Comparable`, `Serializable`, `Closeable`

**Ловушка с `default`-методами:** если два интерфейса объявляют `default`-метод с одинаковой сигнатурой — компилятор заставит явно переопределить в классе:
```java
class MyClass implements A, B { // оба имеют default void hello()
    public void hello() { A.super.hello(); } // надо явно выбрать
}
```

**Правило выбора:** начинай с интерфейса. Переходи к абстрактному классу только если нужно переиспользовать реальный код или состояние.

## 5. Связи с другими концепциями

## 6. Ответ на собесе (2 минуты)

> "Это вопрос о двух разных инструментах с разной философией — и важно понимать эту разницу, а не просто зазубрить таблицу.
>
> **Абстрактный класс** — это частично реализованный класс. Он может иметь поля, конструктор, реализованные методы. Я использую его, когда есть группа связанных классов, которые делят общее состояние или логику. Классический пример — `AbstractList`: там реализованы `iterator()`, `indexOf()` и другие методы через `get()` и `size()`, которые подкласс обязан определить сам.
>
> **Интерфейс** — это чистый контракт. Никакого состояния, только обязательства. Главное преимущество — множественная реализация: класс может быть одновременно `Runnable`, `Comparable` и `Closeable`. В практике Spring я всегда завязываю зависимости на интерфейс — это позволяет подменять реализацию без изменения клиентского кода.
>
> **По Java 8:** `default`-методы размыли границу, но я их использую только для обратной совместимости при эволюции API — не для бизнес-логики.
>
> **Моё правило выбора:** по умолчанию — интерфейс. Абстрактный класс — только когда несколько классов реально делят код и состояние."

## Шпаргалка

| | Интерфейс | Абстрактный класс |
|---|---|---|
| **Аналогия** | Сертификат стандарта | Шаблон с готовыми частями |
| **Наследование** | Множественное | Одиночное |
| **Состояние** | Нет | Есть |
| **Конструктор** | Нет | Есть |
| **Когда** | Контракт / роль | Общая логика + состояние |
| **Примеры** | `Runnable`, `Comparable` | `AbstractList`, `HttpServlet` |
