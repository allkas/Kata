# Контракт equals и hashCode

## 1. Проблема — зачем это существует?

По умолчанию `equals()` сравнивает **адреса в памяти** (`==`). Это значит, что два объекта `new Person("Ivan", 25)` и `new Person("Ivan", 25)` — разные, хотя логически они одинаковые. Коллекции `HashMap` и `HashSet` полагаются на оба метода для корректной работы. Без переопределения — добавишь объект в `HashSet`, а `contains()` вернёт `false`.

## 2. Аналогия

**hashCode** — это номер стеллажа в библиотеке. **equals** — сравнение книги по содержимому.

Сначала библиотекарь идёт на **нужный стеллаж** (hashCode), потом среди книг там **сравнивает по содержимому** (equals). Если hashCode неверный — пойдёт не на тот стеллаж и книгу не найдёт, даже если она есть.

## 3. Как работает

### Контракт (три правила):
1. `equals() == true` → `hashCode()` **обязан** быть одинаковым
2. `equals() == false` → `hashCode()` **может** быть одинаковым (коллизия — допустима)
3. `hashCode() != hashCode()` → `equals()` **обязан** вернуть `false`

### Жизненный цикл в HashMap:
```
put(key, value):
  1. key.hashCode()  → номер бакета (корзины)
  2. key.equals()    → сравнение с существующими ключами в бакете
  3. если equals=true → перезаписать, иначе → добавить в цепочку
```

### Правильная реализация:
```java
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Person)) return false;
    Person p = (Person) o;
    return age == p.age && Objects.equals(name, p.name);
}

@Override
public int hashCode() {
    return Objects.hash(name, age); // те же поля, что и в equals
}
```

## 4. Глубже — важные случаи

### Почему в hashCode используется множитель 31:

```java
// Objects.hash() внутри делает примерно это:
int result = 1;
result = 31 * result + field1.hashCode();
result = 31 * result + field2.hashCode();
```

**Почему 31:** нечётное простое число. Умножение на нечётное не теряет биты (в отличие от чётных). JIT-компилятор оптимизирует `31 * x` → `(x << 5) - x` (битовый сдвиг быстрее умножения). Простое число уменьшает коллизии за счёт лучшего распределения.

### getClass() vs instanceof в equals:

```java
// Вариант 1: getClass() — строгое сравнение класса
@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (o == null || getClass() != o.getClass()) return false; // только точный класс
    Person p = (Person) o;
    return age == p.age && Objects.equals(name, p.name);
}

// Вариант 2: instanceof — допускает подклассы
@Override
public boolean equals(Object o) {
    if (!(o instanceof Person)) return false; // Person и его подклассы
    Person p = (Person) o;
    return age == p.age && Objects.equals(name, p.name);
}
```

| | `getClass()` | `instanceof` |
|--|--------------|-------------|
| **Симметричность** | Гарантирована | Нарушается при смешивании классов |
| **Подклассы** | Не равны родителю | Могут быть равны |
| **Рекомендация** | Для конечных классов | Только если класс final или subclass не переопределяет equals |

Джошуа Блох рекомендует `instanceof` + финальный класс или явная проверка на подкласс. **Симметричность:** `a.equals(b) == b.equals(a)` — обязательное свойство контракта `equals`.

### Если hashCode() всегда возвращает одно число:
```java
@Override public int hashCode() { return 42; } // антипаттерн!
```
- Все объекты попадут в **один бакет**
- HashMap превратится в связанный список
- Поиск: **O(n)** вместо O(1), в Java 8+ — O(log n) после 8 элементов (дерево)
- Корректно, но катастрофически медленно

### Если изменить поля ключа после вставки в HashMap:
```java
Person key = new Person("Ivan", 25);
map.put(key, "value");

key.setName("Petr"); // hashCode изменился!

map.get(key); // null — ищет в другом бакете, объект потерян
```
**Вывод:** ключи HashMap должны быть **immutable** (или хотя бы поля, участвующие в hashCode/equals). Именно поэтому `String` — идеальный ключ.

### Связь с HashSet:
`HashSet` внутри — это `HashMap<E, PRESENT>`. Вся та же логика. `add()` проверяет через `hashCode()` + `equals()`.

## 6. Ответ на собесе (2 минуты)

> "Это, пожалуй, самый частый вопрос по Java — и у него есть глубина.
>
> `equals()` и `hashCode()` — методы `Object`, которые по умолчанию работают по **адресу в памяти**. То есть два объекта `new Person("Ivan", 25)` и `new Person("Ivan", 25)` — разные. Для бизнес-логики это почти никогда не то, что нужно.
>
> **Контракт:** если `equals()` вернул `true` — `hashCode()` обязан быть одинаковым. Обратное необязательно: одинаковый `hashCode` у разных объектов — это коллизия, допустимо.
>
> **Почему нужно переопределять вместе:** `HashMap` использует сначала `hashCode()` для нахождения бакета, потом `equals()` для сравнения ключей внутри. Если переопределить только `equals()` — объекты с одинаковым содержимым попадут в разные бакеты и `get()` вернёт `null`.
>
> **Классический подвох:** если `hashCode()` всегда возвращает одно число — HashMap корректна, но деградирует до O(n). Я использую `Objects.hash(field1, field2)` — это быстро и безопасно.
>
> **Ещё один подвох:** нельзя изменять поля объекта-ключа после вставки в HashMap. `hashCode()` изменится, объект останется в старом бакете — потеря данных. Поэтому ключи должны быть immutable."

## Шпаргалка

| Концепция | Суть | Важно |
|-----------|------|-------|
| **equals()** | Сравнение по содержимому | По умолчанию — по ссылке (`==`) |
| **hashCode()** | Номер бакета в коллекции | Должен быть одинаков при equals=true |
| **Коллизия** | Разные объекты — один hashCode | Допустимо, но замедляет |
| **hashCode = константа** | Все в один бакет | O(n) → катастрофа производительности |
| **Мутабельный ключ** | hashCode меняется | Объект теряется в HashMap |
| **Лучший ключ** | `String`, `Integer` | Immutable, правильно реализованы |
