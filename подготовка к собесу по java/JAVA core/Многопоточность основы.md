# Многопоточность основы

## 1. Проблема — зачем это существует?

Современные процессоры многоядерные. Однопоточная программа использует одно ядро — остальные простаивают. Многопоточность позволяет выполнять задачи параллельно и не блокировать поток на ожидании (IO, сеть, БД). Цена — **разделяемое изменяемое состояние**: когда несколько потоков читают и пишут одни данные без координации, результат непредсказуем.

## 2. Аналогия

Представь офис с общим документом на столе. Один сотрудник (поток) может работать с ним без проблем. Двое одновременно — и данные начнут затирать друг друга. Решения:
- **volatile** — говорит всем «смотрите на оригинал, не на свою копию»
- **synchronized** — «один работает, остальные ждут за дверью»
- **Atomic** — «специальный стол, замена страницы происходит мгновенно и неразрывно»

## 3. Жизненный цикл потока — Thread States

```
new Thread()        thread.start()       scheduler
   NEW  ──────────────► RUNNABLE ◄──────────────────────┐
                           │                             │
                    blocked on I/O              yield() / time slice
                    or lock wait                         │
                           ▼                             │
                        BLOCKED ──► lock released ──────►┘
                        WAITING  ──► notify()/join()
                        TIMED_WAITING ──► timeout
                           │
                    run() завершён
                           ▼
                       TERMINATED
```

| Состояние | Когда | Выход |
|-----------|-------|-------|
| **NEW** | `new Thread()` — ещё не запущен | `thread.start()` |
| **RUNNABLE** | Выполняется или готов выполняться | sleep/wait/IO/блокировка |
| **BLOCKED** | Ждёт монитор (synchronized) | Монитор освобождён |
| **WAITING** | `wait()`, `join()`, `LockSupport.park()` | `notify()`, `notifyAll()`, `join()` завершён |
| **TIMED_WAITING** | `sleep(n)`, `wait(n)`, `join(n)` | Таймаут истёк |
| **TERMINATED** | `run()` завершился нормально или с исключением | — |

```java
Thread t = new Thread(() -> System.out.println("hi"));
System.out.println(t.getState()); // NEW
t.start();
System.out.println(t.getState()); // RUNNABLE (или TERMINATED если быстро)
```

**Process vs Thread:**
| | Process | Thread |
|--|---------|--------|
| Память | Своя | Разделяет Heap с другими потоками |
| Стоимость создания | Дорого | Дешевле |
| Изоляция | Полная | Нет (shared state → проблемы) |
| Коммуникация | IPC (медленно) | Общая память (быстро, опасно) |

## 4. volatile

### Что гарантирует:
1. **Видимость (visibility):** запись в volatile-переменную сразу видна всем потокам. Без volatile каждый поток может иметь копию в CPU-кэше и не видеть изменений других.
2. **Happens-before:** запись в volatile *happens-before* любого последующего чтения этой переменной другим потоком. Упорядочивание инструкций не пересекает барьер.

### Что НЕ гарантирует:
- **Атомарность составных операций.** `i++` — это три шага: read → increment → write. volatile не делает их неразрывными.

```java
volatile int counter = 0;

// Поток A и Поток B выполняют одновременно:
counter++; // read(0) → +1 → write(1) — оба могут прочитать 0 и записать 1
// Итог: 1 вместо 2 — ГОНКА, несмотря на volatile
```

### Нюанс: long и double без volatile неатомарны
На 32-битных JVM запись `long` (64 бит) и `double` — две отдельные 32-битных операции. Поток может прочитать half-written значение. `volatile long` делает запись атомарной.

### Когда применять volatile:
```java
// Паттерн «флаг остановки» — классическое применение
volatile boolean running = true;

// Поток A:
while (running) { doWork(); }

// Поток B:
running = false; // сразу видно потоку A — без volatile не гарантировано
```

**Правило:** volatile подходит если операция — **простое чтение или запись** одного поля, не составная.

## 5. volatile vs synchronized

| | `volatile` | `synchronized` |
|---|---|---|
| **Гарантирует** | Видимость | Видимость + атомарность |
| **Блокировка** | Нет | Да (mutex) |
| **Область** | Одно поле | Блок / метод |
| **Производительность** | Легче | Тяжелее |
| **Применение** | Флаги, одиночные чтения/записи | Составные операции, инварианты |

```java
// volatile: простой флаг — достаточно
private volatile boolean stopped = false;

// synchronized: составная операция — нужна блокировка
private int count = 0;
public synchronized void increment() { count++; } // read+write атомарно
```

**Double-checked locking — volatile обязателен:**
```java
private volatile Singleton instance; // без volatile — сломан в многопоточности

public Singleton getInstance() {
    if (instance == null) {
        synchronized (this) {
            if (instance == null) {
                instance = new Singleton(); // без volatile другой поток может увидеть неполностью сконструированный объект
            }
        }
    }
    return instance;
}
```

## 6. synchronized — Monitor (Mutex)

### Концепция монитора:
Каждый объект в Java — это **монитор** (mutex). `synchronized` захватывает монитор объекта: только один поток может владеть монитором в момент времени.

```java
// На экземпляре — монитор: this
public synchronized void instanceMethod() { ... }
// Эквивалентно:
public void instanceMethod() {
    synchronized (this) { ... }
}

// На классе — монитор: MyClass.class
public static synchronized void staticMethod() { ... }
// Эквивалентно:
public static void staticMethod() {
    synchronized (MyClass.class) { ... }
}
```

**Разные объекты — разные мониторы:**
```java
// НЕ блокируют друг друга:
synchronized (obj1) { ... } // поток A
synchronized (obj2) { ... } // поток B — не заблокирован
```

### Статический vs нестатический synchronized:
| | Нестатический | Статический |
|---|---|---|
| **Монитор** | Экземпляр (`this`) | Объект класса (`MyClass.class`) |
| **Область защиты** | Данные конкретного объекта | Статические данные (глобальные) |
| **Взаимоисключение** | Только между методами **одного экземпляра** | Между **всеми потоками всех экземпляров** |

```java
class Counter {
    private int count = 0;
    private static int staticCount = 0;

    // Два потока с разными экземплярами Counter — НЕ блокируются
    public synchronized void increment() { count++; }

    // Два потока с любыми экземплярами Counter — блокируются
    public static synchronized void incrementStatic() { staticCount++; }
}
```

## 7. wait / notify / notifyAll

`wait()`, `notify()`, `notifyAll()` — методы **Object**, вызываются только внутри `synchronized`-блока.

```
Поток A: synchronized(lock) → lock.wait() → освобождает монитор → WAITING
Поток B: synchronized(lock) → lock.notify() → A переходит в BLOCKED
Поток A: снова захватывает монитор → продолжает работу
```

### Паттерн Producer-Consumer:
```java
class Buffer {
    private final Queue<Integer> queue = new LinkedList<>();
    private final int MAX = 10;

    public synchronized void produce(int item) throws InterruptedException {
        while (queue.size() == MAX) {
            wait(); // буфер полон — ждать
        }
        queue.add(item);
        notifyAll(); // разбудить потребителей
    }

    public synchronized int consume() throws InterruptedException {
        while (queue.isEmpty()) {
            wait(); // буфер пуст — ждать
        }
        int item = queue.poll();
        notifyAll(); // разбудить производителей
        return item;
    }
}
```

**Почему `while`, не `if`?** После `notifyAll()` несколько потоков просыпаются и конкурируют. Когда поток захватит монитор, условие уже может быть снова нарушено — нужно перепроверить.

| | `notify()` | `notifyAll()` |
|---|---|---|
| **Будит** | Один произвольный поток | Все ожидающие |
| **Когда использовать** | Все потоки взаимозаменяемы | Разные условия ожидания |
| **Риск** | Пропустить нужный поток | Лишние пробуждения |

## 8. Deadlock, Livelock, Starvation, Race Condition

### Race Condition (гонка):
Несколько потоков одновременно читают и пишут общие данные, результат зависит от порядка выполнения.
```java
// Классика: counter++ без синхронизации
// Поток A: read(5) → Поток B: read(5) → A: write(6) → B: write(6)
// Ожидали 7, получили 6
```
**Data Race vs Race Condition:** Data Race — конкретная техническая проблема (два потока читают/пишут переменную без синхронизации). Race Condition — более широкое понятие: некорректное поведение из-за порядка выполнения (может быть и без data race, например при неправильном использовании атомиков).

### Deadlock (взаимная блокировка):
Поток A держит ресурс 1, ждёт ресурс 2. Поток B держит ресурс 2, ждёт ресурс 1. Оба ждут вечно.

**4 условия Coffman (все должны выполняться одновременно):**
1. Взаимное исключение (mutex)
2. Удержание и ожидание (hold and wait)
3. Отсутствие вытеснения (no preemption)
4. Циклическое ожидание (circular wait)

```java
synchronized(lockA) {
    synchronized(lockB) { ... } // Поток A
}
synchronized(lockB) {
    synchronized(lockA) { ... } // Поток B — DEADLOCK
}
```
**Как избежать:**
- Всегда захватывать блокировки в **одном и том же порядке**
- Использовать `tryLock(timeout)` из `java.util.concurrent.locks`
- Не держать блокировку дольше необходимого

### Livelock:
Потоки активны, меняют состояние, но прогресса нет — бесконечно реагируют друг на друга.
```
Два человека в узком коридоре: оба шагают вправо, видят друг друга, оба шагают влево — повторяется бесконечно.
```
**Как избежать:** добавить случайную задержку или механизм приоритета.

### Starvation (голодание):
Поток постоянно откладывается в пользу других с более высоким приоритетом — никогда не получает ресурс.
```
Сервер постоянно обрабатывает высокоприоритетные запросы → низкоприоритетный поток ждёт вечно.
```
**Как избежать:** справедливые блокировки (`new ReentrantLock(true)` — fair lock), ограничение приоритетов.

## 9. Атомарные типы (java.util.concurrent.atomic)

### Как работает CAS:
CAS (Compare-And-Swap) — одна аппаратная инструкция: «если текущее значение == expected, установить new; иначе — не менять». Не нужна блокировка — **оптимистичный подход**.

```java
AtomicInteger counter = new AtomicInteger(0);

// Эквивалент synchronized counter++, но без блокировки:
counter.incrementAndGet(); // потокобезопасно

// Явный CAS:
int current = counter.get();
boolean success = counter.compareAndSet(current, current + 1);
// Если другой поток изменил значение — success=false, нужно повторить
```

### Основные классы:
| Класс | Для чего |
|-------|----------|
| `AtomicInteger` / `AtomicLong` | Счётчики, ID-генераторы |
| `AtomicBoolean` | Флаги (альтернатива volatile+synchronized) |
| `AtomicReference<T>` | Атомарная замена объектов |
| `LongAdder` | Высококонкурентные счётчики (быстрее AtomicLong под нагрузкой) |

### Atomic vs synchronized:
| | Atomic (CAS) | synchronized |
|---|---|---|
| **При низкой конкуренции** | Быстрее | Медленнее |
| **При высокой конкуренции** | Много retry, CPU spinning | Стабильнее |
| **Сложность** | Проще | Сложнее |
| **Применение** | Счётчики, одна переменная | Сложная логика, инварианты |

## 10. Semaphore

Семафор — счётчик разрешений. Ограничивает количество потоков, одновременно выполняющих участок кода.

```java
// Разрешить не более 3 потоков одновременно
Semaphore semaphore = new Semaphore(3);

void accessResource() throws InterruptedException {
    semaphore.acquire(); // захватить разрешение (блокирует если 0)
    try {
        useResource();   // защищённая секция
    } finally {
        semaphore.release(); // вернуть разрешение
    }
}
```

**Semaphore vs synchronized:**
| | `synchronized` / `Lock` | `Semaphore` |
|---|---|---|
| **Количество** | 1 поток | N потоков |
| **Ownership** | Тот же поток должен отпустить | Любой поток может отпустить |
| **Применение** | Взаимное исключение | Пул ресурсов (DB connections, rate limiting) |

`Semaphore(1)` — аналог mutex, но без ownership: другой поток может вызвать `release()`.

## 11. Daemon-потоки, приоритеты, join/sleep/yield/interrupt

### Daemon vs User-потоки:
```java
Thread daemon = new Thread(() -> {
    while (true) { runBackgroundTask(); }
});
daemon.setDaemon(true); // ОБЯЗАТЕЛЬНО до start()
daemon.start();
```
- **User-поток** (default): JVM не завершается, пока есть хотя бы один активный user-поток
- **Daemon-поток**: JVM завершается, когда остались только daemon-потоки. Без `finally`, без гарантий — поток просто убивается

**Типичные daemon-потоки:** GC, JIT-компилятор, таймеры фреймворков

### Приоритеты:
```java
thread.setPriority(Thread.MIN_PRIORITY);  // 1
thread.setPriority(Thread.NORM_PRIORITY); // 5 (default)
thread.setPriority(Thread.MAX_PRIORITY);  // 10
```
**Важно:** приоритеты — **подсказка** планировщику ОС, не гарантия. На Linux (JVM) приоритеты потоков часто игнорируются. Не использовать как механизм синхронизации.

### join / sleep / yield:
```java
Thread worker = new Thread(() -> compute());
worker.start();
worker.join();     // текущий поток ждёт завершения worker
worker.join(5000); // ждать максимум 5 секунд

Thread.sleep(1000); // пауза 1 сек, поток остаётся в TIMED_WAITING
                    // НЕ освобождает монитор!

Thread.yield();     // подсказка планировщику: «я готов уступить CPU»
                    // планировщик может проигнорировать
```

| Метод | Состояние | Монитор | Прерываем |
|-------|-----------|---------|-----------|
| `sleep(n)` | TIMED_WAITING | Не отпускает | Да (InterruptedException) |
| `wait()` | WAITING | **Отпускает** | Да (InterruptedException) |
| `join()` | WAITING/TIMED_WAITING | Не отпускает | Да |
| `yield()` | RUNNABLE | Не отпускает | Нет |

### interrupt() vs stop():
```java
// stop() — УСТАРЕЛ. Принудительно убивает поток, оставляет объекты в inconsistent state.
// НЕ использовать!

// interrupt() — вежливый запрос на остановку
Thread worker = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        doWork();
    }
});
worker.start();
// ...
worker.interrupt(); // устанавливает флаг interrupted = true

// Если поток в WAITING/SLEEPING — бросает InterruptedException:
Thread sleeper = new Thread(() -> {
    try {
        Thread.sleep(10_000);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt(); // восстановить флаг!
        cleanup();
    }
});
```
**Правило:** при перехвате `InterruptedException` — либо перебросить выше, либо вызвать `Thread.currentThread().interrupt()` для восстановления флага.

## 12. Runnable vs Callable

```java
// Runnable — нет результата, нет checked exception
Runnable r = () -> System.out.println("done");
new Thread(r).start();

// Callable<T> — возвращает результат, может бросать checked
Callable<Integer> c = () -> compute(); // compute() бросает Exception
ExecutorService executor = Executors.newFixedThreadPool(4);
Future<Integer> future = executor.submit(c);
Integer result = future.get(); // блокирует до результата
```

| | `Runnable` | `Callable<T>` |
|---|---|---|
| Метод | `void run()` | `T call() throws Exception` |
| Результат | Нет | `T` (через `Future<T>`) |
| Исключения | Только unchecked | Checked + unchecked |
| Используется с | `Thread`, `ExecutorService` | `ExecutorService` |

**`FutureTask<T>`** — реализует одновременно `Runnable` и `Future<T>`:
```java
FutureTask<Integer> task = new FutureTask<>(() -> compute());
new Thread(task).start();    // можно запустить через Thread
Integer result = task.get(); // или через ExecutorService

// FutureTask кэширует результат:
task.get(); // второй вызов — мгновенно, без повторного вычисления
```

**`Future<T>`** — контейнер для отложенного результата:
```java
future.get();              // ждёт результат (блокирует)
future.get(5, SECONDS);    // с таймаутом
future.isDone();           // готов ли результат
future.cancel(true);       // отменить задачу (true = interrupt если выполняется)
```

## 13. ThreadLocal

### Как работает:
Каждый поток имеет свою копию переменной. Хранится в `Thread.threadLocalMap` — при завершении потока автоматически очищается.

```java
ThreadLocal<SimpleDateFormat> dateFormat =
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

// В каждом потоке — своя копия SimpleDateFormat (не потокобезопасен сам по себе)
String formatted = dateFormat.get().format(new Date());
```

### Реальные применения:
- **`SecurityContext`** в Spring Security — пользователь текущего запроса
- **Транзакционный `Connection`** — один connection на поток
- **`SimpleDateFormat`** — не потокобезопасен, ThreadLocal даёт по копии на поток
- **Трассировка (MDC)** — correlation ID в логах

### Важно: утечка памяти в пуле потоков:
```java
// В ThreadPoolExecutor потоки переиспользуются!
threadLocal.set(value);
// ... обработка запроса ...
threadLocal.remove(); // ОБЯЗАТЕЛЬНО! Иначе следующий запрос в этом потоке увидит старое значение
```
Если не вызвать `remove()` — старое значение живёт в потоке бесконечно → утечка памяти и баги.

## 15. Ответ на собесе (2 минуты)

> "Многопоточность — большая тема, попробую структурировано.
>
> **Thread lifecycle:** поток проходит состояния NEW → RUNNABLE → BLOCKED/WAITING/TIMED_WAITING → TERMINATED. BLOCKED — ждёт монитор (synchronized). WAITING — вызвал wait() или join() без таймаута.
>
> **volatile** гарантирует видимость: запись сразу видна всем потокам. НЕ гарантирует атомарность — `i++` под volatile всё равно не потокобезопасен. Для флагов и double-checked locking.
>
> **synchronized** захватывает монитор объекта. Нестатический — монитор `this`, статический — монитор `MyClass.class`. Внутри synchronized можно вызывать `wait()` — поток освобождает монитор и уходит в WAITING, `notify()` будит один поток.
>
> **Deadlock** — классика: два потока держат блокировки и ждут друг друга. Четыре условия Coffman. Избегаю захватом блокировок в одном порядке.
>
> **Starvation** — поток никогда не получает ресурс из-за высокоприоритетных конкурентов. Решение — справедливые блокировки (`ReentrantLock(true)`).
>
> **Atomic-типы** — CAS без блокировки. `LongAdder` для высококонкурентных счётчиков.
>
> **ThreadLocal** — каждый поток получает свою копию. В пуле потоков ОБЯЗАТЕЛЬНО вызывать `remove()`, иначе утечка памяти.
>
> **interrupt()** — вежливый запрос на остановку через флаг. `stop()` устарел — убивает поток грубо."

| Концепция | Гарантия | Когда |
|-----------|----------|-------|
| **volatile** | Видимость | Простые флаги, double-checked locking |
| **synchronized** | Видимость + атомарность | Составные операции, инварианты |
| **AtomicInteger** | Атомарность без блокировки (CAS) | Счётчики, низкая/средняя конкуренция |
| **LongAdder** | Атомарность, лучше под нагрузкой | Высококонкурентные счётчики |
| **Race condition** | — | Нет синхронизации → непредсказуемый результат |
| **Deadlock** | — | Круговое ожидание блокировок (4 условия Coffman) |
| **Livelock** | — | Активны, но без прогресса |
| **Starvation** | — | Никогда не получает ресурс из-за приоритетов |
| **ThreadLocal** | Изоляция по потоку | SecurityContext, Connection, MDC |
| **Callable vs Runnable** | Callable возвращает результат | Future<T> для асинхронных задач |
| **FutureTask** | Runnable + Future | Запуск через Thread или ExecutorService |
| **Daemon-поток** | JVM завершается при отсутствии user-потоков | GC, фоновые задачи |
| **sleep()** | TIMED_WAITING, монитор не отпускает | Пауза |
| **wait()** | WAITING, монитор **отпускает** | Producer-Consumer |
| **interrupt()** | Флаг прерывания | Вежливая остановка потока |
| **Semaphore(N)** | N потоков одновременно | Пул ресурсов, rate limiting |

**Thread States:**
| Состояние | Причина |
|-----------|---------|
| NEW | `new Thread()` |
| RUNNABLE | `start()`, выполняется |
| BLOCKED | Ждёт монитор |
| WAITING | `wait()`, `join()` |
| TIMED_WAITING | `sleep(n)`, `wait(n)` |
| TERMINATED | `run()` завершён |
