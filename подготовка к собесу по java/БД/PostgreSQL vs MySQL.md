# PostgreSQL vs MySQL — что лучше для больших данных

## 1. Проблема — зачем это существует?

При выборе СУБД разработчики сталкиваются с MySQL и PostgreSQL как двумя главными open-source вариантами. У каждой — своя история, сильные стороны, поведение при сложных запросах и работе с данными. На собесе спрашивают, чтобы понять: понимаешь ли ты разницу или берёшь что попало.

## 2. Аналогия

**MySQL** — шустрый курьер: быстро доставляет простые посылки (read-heavy workloads, веб-приложения). Иногда срезает углы (менее строгий SQL в старых режимах).

**PostgreSQL** — надёжный грузовик: может везти сложный груз (JSON, аналитика, сложные запросы, расширения), строго соблюдает правила (полный SQL стандарт), медленнее на простых задачах.

## 3. Как работает

### Ключевые отличия

| | PostgreSQL | MySQL |
|--|------------|-------|
| **Тип** | Object-Relational DBMS | Relational DBMS |
| **Лицензия** | PostgreSQL License (свободнее) | GPL v2 (ограничения) |
| **ACID** | Полный, всегда | Полный (InnoDB), MyISAM — нет транзакций |
| **MVCC** | Собственная реализация | InnoDB MVCC |
| **JSON** | Нативный JSONB (индексируемый) | JSON (с MySQL 5.7), хуже PostgreSQL |
| **Полнотекстовый поиск** | Встроенный, мощный | Есть, слабее |
| **Расширения** | PostGIS, TimescaleDB, pgvector | Ограниченные |
| **Стандарт SQL** | Очень близко к стандарту | Исторически отклонения |
| **Репликация** | Streaming, Logical | Master-Slave, GTID |
| **Производительность** | Лучше на сложных запросах | Лучше на простых read |

---

### PostgreSQL: сильные стороны

**JSONB — лучший выбор для document-like данных:**
```sql
-- Нативное хранение и индексирование JSON
CREATE TABLE products (
    id SERIAL PRIMARY KEY,
    data JSONB
);

CREATE INDEX idx_products_category ON products USING GIN (data);

SELECT * FROM products
WHERE data @> '{"category": "electronics"}'  -- очень быстро с GIN индексом
  AND (data->>'price')::numeric > 100;
```

**PostGIS — геопространственные данные:**
```sql
-- Найти все точки в радиусе 10км
SELECT * FROM stores
WHERE ST_Distance(location, ST_MakePoint(37.6, 55.7)) < 10000;
```

**Window Functions (стандарт SQL):**
```sql
SELECT name, salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC),
    LAG(salary) OVER (PARTITION BY department ORDER BY salary) as prev_salary
FROM employees;
```

---

### MySQL: сильные стороны

**Read-heavy веб-приложения:** MySQL исторически быстрее на простых SELECT при высоком concurrency (оптимизированный буфер InnoDB, меньше overhead).

**Экосистема:** более широкое распространение в legacy web-стеке (LAMP), хостинги, cPanel. Amazon Aurora — MySQL-совместимая managed БД с автомасштабированием.

---

### Что лучше для больших данных

Для **OLTP (транзакции, обычные приложения):**
- Современные проекты → PostgreSQL (лучше соответствие стандартам, JSONB, расширения)
- Legacy веб-стек → MySQL (если уже есть и работает)

Для **аналитики (OLAP):**
- Оба плохо подходят для петабайтов. Нужны: ClickHouse, Redshift, BigQuery, Snowflake
- **TimescaleDB** (расширение PostgreSQL) — временные ряды, метрики

Для **смешанного JSON + реляционного:**
- PostgreSQL + JSONB явно лучше MySQL JSON

### Oracle — когда упоминается

| | PostgreSQL | MySQL | Oracle |
|--|------------|-------|--------|
| **Стоимость** | Бесплатно | Бесплатно (Community) | Очень дорого |
| **Масштаб** | Средний → Крупный | Малый → Средний | Enterprise |
| **Где** | Стартапы, tech-компании | Веб, legacy | Банки, корпорации |
| **Фичи** | Богатые | Базовые | Максимальные |

## 4. Глубже — что важно знать

**VACUUM в PostgreSQL:** из-за MVCC мёртвые строки (удалённые/обновлённые) не удаляются сразу. `VACUUM ANALYZE` очищает их и обновляет статистику. `AUTOVACUUM` делает это автоматически, но при высокой нагрузке может не успевать.

**MySQL storage engines:** InnoDB (default, с транзакциями) vs MyISAM (без транзакций, legacy). Всегда InnoDB в современных приложениях.

**Выбор для нового проекта:** в 2024 году PostgreSQL — выбор по умолчанию для нового Java backend-проекта. MySQL — только при явном требовании или legacy.

## 6. Ответ на собесе (2 минуты)

PostgreSQL и MySQL — оба отличные open-source СУБД. Для нового проекта я выбираю PostgreSQL по нескольким причинам: полное соответствие SQL-стандарту, нативный JSONB с индексированием, богатая экосистема расширений (PostGIS для гео, TimescaleDB для временных рядов, pgvector для ML), лучшая производительность на сложных аналитических запросах.

MySQL исторически быстрее на простых read-heavy workloads и имеет более широкое распространение в legacy web-стеке. Amazon Aurora MySQL — популярная managed БД в облаке.

Для «больших данных» в смысле аналитики — оба не лучший выбор. Там нужны колоночные СУБД: ClickHouse, Redshift, BigQuery.

На практике важнее правильные индексы, правильный дизайн схемы и правильные запросы, чем выбор между двумя СУБД — обе справятся с большинством задач.

## Шпаргалка

| | PostgreSQL | MySQL |
|--|------------|-------|
| **JSONB** | Нативный, индексируемый | Есть, хуже |
| **Расширения** | PostGIS, TimescaleDB... | Ограниченные |
| **SQL стандарт** | Строгое следование | Исторические отклонения |
| **Read-heavy** | Хорошо | Немного быстрее |
| **Default изоляция** | Read Committed | Repeatable Read |
| **Новый проект** | Выбор по умолчанию | Legacy/специфика |
