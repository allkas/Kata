# Проблема N+1 запросов

## 1. Проблема — зачем это знать?

Hibernate — ORM: он скрывает SQL за Java-объектами. Но это скрытие опасно: одна строка `order.getItems()` может породить сотни SQL-запросов. Это называется N+1: вместо одного запроса с JOIN получаем 1 запрос за родителями + N отдельных запросов за дочерними сущностями. На продакшне с тысячами заказов — это катастрофа производительности.

## 2. Аналогия

Нужно узнать зарплату сотрудников каждого отдела. Плохой подход: сначала запросить список отделов (1 звонок в HR), потом для каждого отдела отдельно позвонить в HR за зарплатами (N звонков). Итого: 1 + N звонков. Хороший подход: один звонок — «дайте все отделы сразу со списком сотрудников».

## 3. Механика N+1

```java
// Сущности:
@Entity class Order {
    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}

// Репозиторий:
List<Order> orders = orderRepo.findAll();
// SQL: SELECT * FROM orders  ← 1 запрос

for (Order order : orders) {
    int count = order.getItems().size(); // триггер LAZY-загрузки
    // SQL: SELECT * FROM order_items WHERE order_id = ?  ← N запросов
}

// При 100 заказах: 1 + 100 = 101 запрос вместо 1
```

**Почему Hibernate так делает:** по умолчанию `@OneToMany` — LAZY. Hibernate загружает коллекцию только при обращении к ней. Каждое обращение → отдельный SELECT.

## 4. LAZY vs EAGER

### Дефолты в JPA:
| Аннотация | Дефолт | Почему |
|-----------|--------|--------|
| `@OneToMany` | **LAZY** | Коллекция может быть большой |
| `@ManyToMany` | **LAZY** | Аналогично |
| `@ManyToOne` | **EAGER** | Один объект — не страшно |
| `@OneToOne` | **EAGER** | Один объект — не страшно |

### LAZY:
```java
@OneToMany(fetch = FetchType.LAZY)
private List<OrderItem> items;

// items НЕ загружаются при загрузке Order
// Загружаются при первом обращении: order.getItems()
// Плюс: не грузим лишнее
// Минус: нужна открытая сессия в момент обращения
```

### EAGER:
```java
@ManyToOne(fetch = FetchType.EAGER)
private Customer customer;

// customer загружается ВМЕСТЕ с Order всегда
// Hibernate выполняет JOIN или subselect
// Плюс: всегда доступно
// Минус: грузим даже когда не нужно; с @OneToMany → Cartesian product
```

### Опасность EAGER с коллекциями:
```java
@OneToMany(fetch = FetchType.EAGER) // НИКОГДА так не делай
private List<OrderItem> items;

// Hibernate делает JOIN: orders × order_items
// Если у заказа 100 позиций — в ResultSet 100 строк на КАЖДЫЙ заказ
// При 1000 заказов → 100 000 строк в памяти
```
**Правило:** `@OneToMany` — всегда LAZY. EAGER только для `@ManyToOne`/`@OneToOne` и только когда реально нужно всегда.

## 5. LazyInitializationException

### Когда возникает:
```java
// Сессия Hibernate живёт внутри транзакции.
// После закрытия транзакции — сессия закрыта.

@Service
public class OrderService {
    // НЕТ @Transactional — сессия закрывается после findById()
    public Order getOrder(Long id) {
        return orderRepo.findById(id).orElseThrow();
    }
}

@RestController
public class OrderController {
    public ResponseEntity<OrderDto> get(Long id) {
        Order order = service.getOrder(id); // сессия уже закрыта!
        order.getItems().size(); // LazyInitializationException
        // "could not initialize proxy - no Session"
    }
}
```

### Решения:
**1. Добавить `@Transactional` на сервисный метод:**
```java
@Transactional(readOnly = true)
public Order getOrderWithItems(Long id) {
    Order order = orderRepo.findById(id).orElseThrow();
    order.getItems().size(); // сессия жива — загружается нормально
    return order;
}
```

**2. Использовать JOIN FETCH или @EntityGraph — загрузить жадно в одном запросе.**

**3. Возвращать DTO вместо сущности** — тогда LAZY-поля вообще не нужны.

**Антипаттерн — `open-session-in-view`:** Spring Boot включает его по умолчанию (держит сессию до рендеринга view). Это маскирует проблему и провоцирует N+1 в слое представления. Рекомендую отключать: `spring.jpa.open-in-view=false`.

## 6. Решения проблемы N+1

### JOIN FETCH в JPQL:
```java
@Query("SELECT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems();
// SQL: SELECT o.*, i.* FROM orders o JOIN order_items i ON o.id = i.order_id
// Один запрос вместо N+1

// Внимание: дубликаты при коллекциях!
@Query("SELECT DISTINCT o FROM Order o JOIN FETCH o.items")
List<Order> findAllWithItems(); // DISTINCT убирает дубли на уровне Java

// Несколько коллекций одновременно — нельзя:
// JOIN FETCH o.items JOIN FETCH o.payments → MultipleBagFetchException
// Решение: последовательно или @EntityGraph
```

### @EntityGraph:
```java
@EntityGraph(attributePaths = {"items", "customer"})
@Query("SELECT o FROM Order o")
List<Order> findAllWithItemsAndCustomer();
// Hibernate генерирует LEFT JOIN FETCH для указанных атрибутов
// Удобнее для нескольких ассоциаций
```

### Batch fetching (`@BatchSize`):
```java
@Entity class Order {
    @OneToMany(mappedBy = "order")
    @BatchSize(size = 20)   // загружать порциями по 20
    private List<OrderItem> items;
}
// Вместо N запросов: N/20 запросов
// SELECT * FROM order_items WHERE order_id IN (1,2,...,20)
// Компромисс: не один запрос, но значительно лучше N+1

// Глобально в application.properties:
// spring.jpa.properties.hibernate.default_batch_fetch_size=20
```

### DTO проекции:
```java
// Нет сущностей — нет LAZY — нет N+1
@Query("SELECT new com.example.OrderDto(o.id, o.total, i.productName) " +
       "FROM Order o JOIN o.items i")
List<OrderDto> findOrderDtos();
```

## 7. JOIN FETCH vs обычный JOIN

```java
// Обычный JOIN: только для условий фильтрации, данные НЕ загружаются
@Query("SELECT o FROM Order o JOIN o.customer c WHERE c.city = 'Moscow'")
List<Order> findByCity(String city);
// customer в persistence context НЕ загружен
// При обращении к o.getCustomer() — отдельный SELECT (или LIE если сессия закрыта)

// JOIN FETCH: загружает данные В persistence context
@Query("SELECT o FROM Order o JOIN FETCH o.customer c WHERE c.city = 'Moscow'")
List<Order> findByCityWithCustomer(String city);
// customer загружен сразу — обращение к o.getCustomer() без доп. запроса
```

| | JOIN | JOIN FETCH |
|---|---|---|
| **Цель** | Фильтрация | Жадная загрузка |
| **Данные в PC** | Нет | Да |
| **SQL** | Только JOIN условие | SELECT + JOIN данные |
| **Lazy доступ после** | Доп. запрос или LIE | Нет запроса |

## 9. Ответ на собесе (2 минуты)

> "N+1 — классическая проблема ORM. При загрузке коллекции с LAZY-ассоциацией Hibernate делает 1 запрос за родителями и N отдельных запросов за каждой дочерней коллекцией. На 100 заказах — 101 запрос вместо 1.
>
> **Диагностика:** подключаю `spring.jpa.show-sql=true` или Hibernate Statistics и смотрю количество запросов. В тестах использую Datasource-proxy с подсчётом SQL.
>
> **Решения:** JOIN FETCH в JPQL — один запрос с JOIN. Для нескольких коллекций — `@EntityGraph` или `@BatchSize`. DTO-проекции вообще исключают проблему — нет сущностей, нет LAZY.
>
> **LazyInitializationException:** возникает когда обращаешься к LAZY-коллекции после закрытия сессии. Сессия живёт внутри транзакции. Решение: `@Transactional` на сервисном методе или загружать данные жадно. Отключаю `open-session-in-view` — он маскирует проблему и провоцирует N+1 на уровне контроллера.
>
> **EAGER везде** — антипаттерн. С `@OneToMany(EAGER)` Hibernate делает Cartesian product: 1000 заказов × 100 позиций = 100 000 строк в памяти."

## Шпаргалка

| Концепция | Суть | Решение |
|-----------|------|---------|
| **N+1** | 1 + N SQL вместо 1 | JOIN FETCH / @EntityGraph / @BatchSize |
| **LAZY** (дефолт @OneToMany) | Загружается при обращении | Нужна открытая сессия |
| **EAGER** (дефолт @ManyToOne) | Загружается всегда | Осторожно с коллекциями |
| **LazyInitializationException** | Нет сессии при обращении | @Transactional / жадная загрузка |
| **JOIN FETCH** | Загружает в persistence context | DISTINCT при коллекциях |
| **@EntityGraph** | Несколько ассоциаций | Удобнее JOIN FETCH |
| **@BatchSize** | Групповая загрузка | Компромисс: N/size запросов |
| **DTO проекция** | Нет сущностей → нет LAZY | Лучший вариант для read |
| **open-in-view** | Маскирует LIE | Отключить: `open-in-view=false` |
