# Транзакции уровни изоляции

## 1. Проблема — зачем это существует?

Без транзакций операция перевода денег (снять с одного счёта, добавить другому) может прерваться посередине — деньги исчезнут. Без изоляции один пользователь читает данные, которые другой ещё не закоммитил или уже изменил. Транзакции и уровни изоляции — это контракт между приложением и базой данных о том, как она ведёт себя при сбоях и параллельных запросах.

## 2. Аналогия

**ACID** — как банковская операция у кассира. **Атомарность:** кассир либо оформляет всю операцию, либо отказывает — нельзя снять деньги без зачисления. **Согласованность:** баланс никогда не уходит в минус (ограничение). **Изоляция:** очередь: пока один клиент у окошка, другие ждут снаружи и не видят промежуточного состояния. **Долговечность:** операция подписана и не исчезнет, даже если электричество выключат.

## 3. ACID

### Atomicity (Атомарность):
Транзакция — неделимая единица. Все операции выполняются или ни одна.
```sql
BEGIN;
  UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
  UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT; -- оба UPDATE применяются вместе

-- Если второй UPDATE упал — ROLLBACK откатит и первый
```

### Consistency (Согласованность):
Транзакция переводит БД из одного **валидного** состояния в другое. Все ограничения (NOT NULL, UNIQUE, FK, CHECK) соблюдены после коммита.

### Isolation (Изоляция):
Параллельные транзакции не видят незакоммиченных изменений друг друга (в зависимости от уровня). Реализуется через блокировки или MVCC.

### Durability (Долговечность):
Закоммиченные данные сохраняются даже при сбое. Реализуется через WAL (Write-Ahead Log): изменения сначала пишутся в журнал, потом применяются к данным.

## 4. Проблемы параллельного доступа

### Dirty Read (Грязное чтение):
Транзакция A читает **незакоммиченные** данные транзакции B. Если B откатится — A работала с несуществующими данными.
```
T1: UPDATE orders SET status='PAID' WHERE id=1  -- ещё не COMMIT
T2: SELECT status FROM orders WHERE id=1  → 'PAID'  -- читает незакоммиченное
T1: ROLLBACK  -- T2 уже приняла решение на основе несуществующих данных
```

### Non-Repeatable Read (Неповторяемое чтение):
В одной транзакции один и тот же SELECT дважды возвращает разные строки, потому что другая транзакция закоммитила UPDATE между чтениями.
```
T1: SELECT salary FROM emp WHERE id=1  → 50000
T2: UPDATE emp SET salary=60000 WHERE id=1; COMMIT;
T1: SELECT salary FROM emp WHERE id=1  → 60000  -- другой результат!
```

### Phantom Read (Фантомное чтение):
Повторный SELECT возвращает **новые строки** (фантомы), потому что другая транзакция закоммитила INSERT/DELETE.
```
T1: SELECT COUNT(*) FROM orders WHERE amount > 1000  → 5
T2: INSERT INTO orders (amount) VALUES (2000); COMMIT;
T1: SELECT COUNT(*) FROM orders WHERE amount > 1000  → 6  -- появился фантом
```

## 5. Уровни изоляции

| Уровень | Dirty Read | Non-Repeatable | Phantom | Производительность |
|---------|-----------|----------------|---------|-------------------|
| **Read Uncommitted** | ✅ возможно | ✅ | ✅ | Максимальная |
| **Read Committed** | ❌ нет | ✅ возможно | ✅ | Высокая |
| **Repeatable Read** | ❌ | ❌ нет | ✅ (в теории) | Средняя |
| **Serializable** | ❌ | ❌ | ❌ нет | Минимальная |

**Дефолты СУБД:**
- **PostgreSQL**: Read Committed
- **MySQL InnoDB**: Repeatable Read
- **Oracle**: Read Committed

**Read Committed на практике:**
Каждый оператор видит снапшот закоммиченных данных **на момент выполнения этого оператора**. Самый распространённый уровень.

**Repeatable Read:**
Транзакция видит снапшот **на момент своего начала**. В PostgreSQL Repeatable Read также защищает от фантомов (благодаря MVCC).

**Serializable:**
Транзакции выполняются как будто последовательно. Реализуется через предикатные блокировки или SSI (Serializable Snapshot Isolation) в PostgreSQL.

## 6. @Transactional в Spring

```java
@Service
public class OrderService {

    @Transactional  // default: REQUIRED, READ_COMMITTED (зависит от БД)
    public void createOrder(OrderDto dto) {
        Order order = orderRepo.save(new Order(dto));
        paymentService.charge(order); // если бросит RuntimeException — откат всего
    }

    // Откат на checked exception:
    @Transactional(rollbackFor = Exception.class)
    public void processFile() throws IOException { ... }

    // Только чтение — оптимизация (нет dirty-checking, snapshot в некоторых БД)
    @Transactional(readOnly = true)
    public List<Order> findAll() { return orderRepo.findAll(); }
}
```

**Важно:** `@Transactional` работает через AOP-прокси. Вызов метода **внутри того же класса** — прокси обходится, транзакция не стартует.

### Propagation (распространение транзакций):
| Тип | Поведение |
|-----|-----------|
| **REQUIRED** (default) | Использует существующую или создаёт новую |
| **REQUIRES_NEW** | Всегда новая; приостанавливает текущую |
| **SUPPORTS** | Если есть — использует, нет — без транзакции |
| **MANDATORY** | Требует существующей, иначе исключение |
| **NEVER** | Запрещает транзакцию, иначе исключение |
| **NESTED** | Вложенная через savepoint |

```java
// REQUIRES_NEW: логирование должно сохраниться даже при откате основной транзакции
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logAction(String action) { auditRepo.save(new Audit(action)); }
```

## 7. MVCC (Multi-Version Concurrency Control)

PostgreSQL и MySQL InnoDB не блокируют читателей писателями и наоборот. Каждая строка имеет версии (xmin/xmax). Читатель видит снапшот без ожидания блокировок. Это основа высокой конкурентности.

## 9. Ответ на собесе (2 минуты)

> "ACID — четыре гарантии транзакции. **Атомарность:** либо всё, либо ничего — нельзя снять деньги без зачисления. **Согласованность:** все ограничения БД соблюдаются после коммита. **Изоляция:** параллельные транзакции не видят промежуточных состояний друг друга (в зависимости от уровня). **Долговечность:** закоммиченное переживёт любой сбой — благодаря WAL.
>
> **Уровни изоляции** решают разные проблемы. Dirty read — читаешь незакоммиченное, которое может откатиться. Non-repeatable read — два одинаковых SELECT в одной транзакции дают разный результат из-за чужого UPDATE. Phantom — появляются новые строки из-за чужого INSERT.
>
> **Read Committed** — дефолт PostgreSQL, убирает dirty reads. Каждый запрос видит свежий снапшот закоммиченных данных. **Repeatable Read** — снапшот фиксируется на старте транзакции, non-repeatable reads невозможны. **Serializable** — максимальная изоляция, но медленнее.
>
> **В Spring:** `@Transactional` с дефолтным REQUIRED. Важный нюанс — откат только на RuntimeException. Если метод бросает checked, нужно явно указать `rollbackFor = Exception.class`. И нельзя вызывать `@Transactional`-метод из того же класса — AOP-прокси обходится."

## Шпаргалка

| Концепция | Суть | Реализация |
|-----------|------|-----------|
| **Atomicity** | Всё или ничего | ROLLBACK при ошибке |
| **Consistency** | Ограничения соблюдены | FK, CHECK, UNIQUE |
| **Isolation** | Уровни изоляции | Блокировки / MVCC |
| **Durability** | Выживает сбои | WAL (Write-Ahead Log) |
| **Dirty Read** | Читаешь незакоммиченное | Решает Read Committed |
| **Non-Repeatable** | Разный результат двух SELECT | Решает Repeatable Read |
| **Phantom** | Новые строки появились | Решает Serializable |
| **Дефолт PostgreSQL** | Read Committed | — |
| **Дефолт MySQL** | Repeatable Read | — |
| **@Transactional rollback** | Только RuntimeException | `rollbackFor` для checked |
| **Self-invocation** | Прокси обходится | Инжектировать себя или реструктурировать |
