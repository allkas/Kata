# Docker vs VM

## 1. Проблема — зачем это существует?

«На моей машине работает» — классическая проблема разработки. Разные версии Java, разные конфиги, разные OS. Docker упаковывает приложение вместе со всем окружением в контейнер — и оно работает одинаково везде. VM решает похожую задачу, но с гораздо большими затратами ресурсов.

## 2. Аналогия

**VM** — отдельный дом. Своя земля (аппаратные ресурсы), свой фундамент (kernel), свои стены, свои коммуникации. Полная изоляция. Но каждый дом занимает много места и строится долго.

**Docker** — квартира в многоквартирном доме. Общий фундамент и коммуникации (kernel хоста), но отдельная квартира (изолированный процесс). Быстро, дёшево, тысячи квартир в одном здании.

## 3. Как работает

### Виртуальная машина (VM):
```
[Приложение A]  [Приложение B]
[Гостевая ОС A] [Гостевая ОС B]  ← полные копии Linux/Windows
[Гипервизор (VMware, VirtualBox, KVM)]
[Железо / Хост-ОС]
```
- Каждая VM запускает **полную копию ОС** (ядро + userspace)
- Изоляция: полная (отдельное ядро)
- Размер: гигабайты
- Запуск: минуты

### Docker-контейнер:
```
[Приложение A]  [Приложение B]
[Libs/deps A]   [Libs/deps B]   ← только зависимости приложения
[Docker Engine / Container Runtime]
[Хост-ОС (ядро Linux)]          ← одно ядро на всех
[Железо]
```
- Контейнеры **разделяют ядро** хост-ОС (через namespace/cgroups)
- Изоляция: процессов и файловой системы
- Размер: мегабайты
- Запуск: секунды

## 4. Сравнение

| Характеристика | VM | Docker |
|----------------|-----|--------|
| **Гостевая ОС** | Полноценная (kernel + userspace) | Нет (только userspace) |
| **Ядро** | Своё (гостевое) | Ядро хоста |
| **Размер** | Гигабайты | Мегабайты |
| **Запуск** | Минуты | Секунды |
| **Изоляция** | Полная (hardware level) | Процессов (namespace) |
| **Плотность** | Десятки на сервере | Тысячи на сервере |
| **Overhead** | Высокий (двойная ОС) | Минимальный |
| **Безопасность** | Выше | Ниже (shared kernel) |
| **Применение** | Разные ОС, максимальная изоляция | Микросервисы, CI/CD, dev окружение |

## 5. Dockerfile — как строится образ

Dockerfile — инструкция для сборки образа. Каждая инструкция — новый **слой** (layer).

```dockerfile
# Базовый образ — первый слой
FROM openjdk:17-slim

# Слой: рабочая директория
WORKDIR /app

# Слой: зависимости (кэшируются если pom.xml не изменился)
COPY pom.xml .
RUN mvn dependency:go-offline

# Слой: исходный код
COPY src/ ./src/
RUN mvn package -DskipTests

# Слой: точка входа
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "target/app.jar"]
```

**Кэширование слоёв:** Docker кэширует каждый слой. Если слой не изменился — пересборка не происходит. Поэтому зависимости (pom.xml) копируют отдельно от кода — зависимости меняются редко, код — часто.

**Multi-stage build** — уменьшение размера итогового образа:
```dockerfile
# Стадия 1: сборка
FROM maven:3.9-openjdk-17 AS builder
WORKDIR /app
COPY . .
RUN mvn package -DskipTests

# Стадия 2: runtime — только JRE, без Maven
FROM openjdk:17-slim
WORKDIR /app
COPY --from=builder /app/target/app.jar .
ENTRYPOINT ["java", "-jar", "app.jar"]
```
Итоговый образ не содержит Maven, исходников, промежуточных файлов — только JAR и JRE.

## 7. Ответ на собесе (2 минуты)

> "VM создаёт полную виртуальную машину с собственной ОС и ядром — полная изоляция, гигабайты, минуты запуска. Docker разделяет ядро хоста — изоляция на уровне процессов через Linux namespaces и cgroups, мегабайты, секунды запуска.
>
> **Ключевое отличие:** VM изолирует на уровне hardware/hypervisor, Docker — на уровне ОС. Отсюда: Docker быстрее, легче, но изоляция слабее (shared kernel — уязвимость одного контейнера теоретически влияет на хост).
>
> **Dockerfile:** набор инструкций, каждая — слой. Слои кэшируются. Поэтому зависимости (pom.xml) копируем до исходников — зависимости меняются редко, кэш не сбрасывается. Multi-stage build убирает build-инструменты из финального образа."

## Шпаргалка

| Концепция | VM | Docker |
|-----------|-----|--------|
| **Ядро** | Своё | Общее с хостом |
| **Размер** | Гигабайты | Мегабайты |
| **Запуск** | Минуты | Секунды |
| **Изоляция** | Hardware-level | OS-level (namespace) |
| **Dockerfile FROM** | Базовый образ | Первый слой |
| **Слой (layer)** | Одна инструкция | Кэшируется |
| **Multi-stage** | Раздельные стадии build/run | Меньший итоговый образ |
| **EXPOSE** | Объявить порт | Не публикует — только документация |
| **ENTRYPOINT** | Команда при запуске | `java -jar app.jar` |
