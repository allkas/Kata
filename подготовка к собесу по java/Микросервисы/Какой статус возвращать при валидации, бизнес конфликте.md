# Какой статус возвращать при валидации, бизнес-конфликте (400, 409, 422)

## 1. Проблема — зачем это существует?

При ошибке в запросе клиент должен понять, что именно пошло не так и как исправить. Вернуть 400 на любую ошибку — не информативно. HTTP выделяет три разных случая: ошибка формата (400), нарушение бизнес-правил (422), конфликт состояния ресурса (409). Правильный выбор помогает клиенту принять решение — исправить запрос, получить актуальное состояние или повторить.

## 2. Аналогия

Представь, что ты подаёшь документы в госорган:

- **400** — секретарь: *«Я не понимаю этот язык. Это не документ»* — форма не заполнена
- **422** — инспектор: *«Форма заполнена правильно, но вы несовершеннолетний»* — бизнес-правило нарушено
- **409** — регистратор: *«Этот номер паспорта уже зарегистрирован»* — конфликт с текущим состоянием

## 3. Как работает

### 400 Bad Request — ошибка формата

Сервер не может **разобрать** запрос. Проблема в синтаксисе или типах данных.

```http
POST /users
{"name": "Alice", "age": 30    ← невалидный JSON (нет закрывающей скобки)

POST /users
{"name": "Alice", "age": "тридцать"}   ← age должен быть числом

POST /users
{"name": "Alice"}    ← обязательное поле email отсутствует
```

---

### 422 Unprocessable Entity — бизнес-валидация

Запрос корректен по формату, но **нарушает бизнес-правила**. Сервер понял запрос, но не может его исполнить.

```http
POST /users
{"name": "Alice", "email": "alice@mail.com", "age": 16}
→ 422: возраст должен быть >= 18

POST /orders
{"productId": 123, "quantity": -5}
→ 422: количество не может быть отрицательным

POST /users
{"name": "Bob", "email": "not-an-email"}
→ 422: невалидный формат email
```

Ответ по RFC 7807 (Problem Details):
```json
{
  "type": "https://api.example.com/errors/validation-failed",
  "title": "Validation Error",
  "status": 422,
  "errors": {
    "age":   ["Must be at least 18"],
    "email": ["Must be a valid email"]
  }
}
```

---

### 409 Conflict — конфликт состояния ресурса

Запрос правильный, но **конфликтует с текущим состоянием** на сервере. Возможно, повторить после получения актуального состояния.

```http
PUT /users/alice@mail.com
→ 409: пользователь с таким email уже существует

POST /orders/123/cancel
→ 409: заказ уже доставлен, отмена невозможна

PUT /users/123
If-Match: "etag-old"
→ 409: ресурс изменён другим пользователем (optimistic locking)
```

---

### Decision tree — как выбирать

```
Ошибка в запросе
      │
      ├─ Сервер смог разобрать запрос? ──── Нет ──→ 400 Bad Request
      │
      └─ Да
            │
            ├─ Нарушены бизнес-правила? ──── Да ───→ 422 Unprocessable Entity
            │  (данные некорректны сами по себе)
            │
            └─ Конфликт с состоянием ресурса? ─────→ 409 Conflict
               (данные ok, но сейчас нельзя)
```

## 4. Глубже — что важно знать

**Спорный случай: email уже существует — 422 или 409?**

Зависит от контекста:
- `POST /users` (создание, ID назначает сервер) → **422**: нарушено бизнес-правило уникальности
- `PUT /users/alice@mail.com` (клиент задаёт ID) → **409**: ресурс с таким ID уже существует

**400 vs 422 для формата email:**
- Email отсутствует или не строка → **400** (не можем разобрать)
- Email строка, но не соответствует формату → **422** (не прошло бизнес-валидацию)

**Retry-логика:**
| Статус | Повторять? | Как |
|--------|-----------|-----|
| 400 | ❌ Нет | Исправить синтаксис |
| 422 | ❌ Нет | Исправить данные |
| 409 | ✅ Возможно | Получить актуальное состояние, повторить |

**Полная карта клиентских ошибок:**

| Статус | Название | Когда |
|--------|----------|-------|
| 400 | Bad Request | Невалидный синтаксис, неверный формат |
| 401 | Unauthorized | Нет аутентификации |
| 403 | Forbidden | Аутентифицирован, но нет прав |
| 404 | Not Found | Ресурс не найден |
| 409 | Conflict | Optimistic lock, дубль, состояние не позволяет |
| 422 | Unprocessable Entity | Бизнес-валидация не пройдена |
| 429 | Too Many Requests | Rate limiting |

## 6. Ответ на собесе (2 минуты)

Три разных ошибки — три разных статуса. 400 Bad Request — сервер не смог разобрать запрос: невалидный JSON, неверный тип поля, отсутствует обязательное поле. 422 Unprocessable Entity — запрос понятен по формату, но нарушает бизнес-правила: возраст меньше 18, отрицательное количество, невалидный формат email. 409 Conflict — запрос правильный, но конфликтует с текущим состоянием: ресурс уже существует, optimistic lock, состояние не позволяет операцию.

Ключевое отличие 409 от 422: 422 означает «эти данные никогда не пройдут», 409 означает «сейчас нельзя, но после получения актуального состояния — возможно».

Для структурированных ошибок использую RFC 7807 Problem Details: `type`, `title`, `status`, `detail` и поле `errors` с конкретными полями и сообщениями. Так клиент точно знает, что исправить.

## Шпаргалка

| Статус | Образ | Ключевой факт | Retry? |
|--------|-------|---------------|:---:|
| **400** | «Не понимаю этот язык» | Синтаксис, тип данных, missing required | ❌ |
| **422** | «Так нельзя по правилам» | Бизнес-правила при корректном формате | ❌ |
| **409** | «Сейчас невозможно» | Конфликт состояния: дубль, optimistic lock | ✅ |
