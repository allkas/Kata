# Микросервисы vs монолит

## 1. Проблема — зачем это существует?

Когда команда растёт, монолит начинает болеть: 5 команд мешают друг другу при релизах, масштабировать можно только всё целиком, выбор технологий заморожен. Микросервисы решают эти проблемы — но создают новые. Вопрос не в том, что лучше, а в том, когда какой компромисс оправдан.

## 2. Аналогия

Монолит — как ресторан, где один шеф-повар делает всё: принимает заказы, готовит, выносит блюда. Быстро запустить, просто управлять. Но если посетителей много — один человек не справится, и нельзя нанять специалиста только по десертам.

Микросервисы — как ресторан с разными станциями: холодный цех, горячий, кондитерская. Каждый специализируется, можно масштабировать только кондитерскую в праздники. Но нужна координация, общий язык, система передачи заказов между станциями.

## 3. Как работает

### Монолит

Единое приложение: весь код деплоится вместе, общая база данных, одна кодовая база.

**Плюсы монолита:**

| Плюс | Почему важно |
|------|-------------|
| Простота разработки на старте | Один репозиторий, одна среда, не нужно думать о сети |
| Быстрый деплой и отладка | Всё поднимается одной командой, один лог-файл |
| ACID транзакции «из коробки» | Вся бизнес-логика в одной БД — не нужны саги |
| Низкие операционные издержки | Нет 20 сервисов с CI/CD, мониторингом и алертами |

**Минусы монолита (когда начинает болеть):**

| Минус | Почему критично |
|-------|----------------|
| Масштабируется только целиком | Если нагружен один модуль — масштабируешь всё |
| Жёсткая связанность команд | Релиз одной команды ломает другую, мешают друг другу |
| Технологическое замыкание | Выбор стека — решение на годы, нельзя сменить один модуль |
| "Большой комок грязи" со временем | Циклические зависимости, страшно что-то менять |

---

### Микросервисы

Набор независимо деплоимых сервисов, каждый со своей базой данных и зоной ответственности.

**Плюсы микросервисов:**

| Плюс | Почему важно |
|------|-------------|
| Независимое масштабирование | Масштабируем только нагруженный сервис — экономия ресурсов |
| Независимость команд | Каждая команда — свой стек, свой ритм релизов |
| Изоляция отказов | Если упал сервис уведомлений — заказы продолжают работать |
| Технологическое разнообразие | Python для AI, Go для performance-critical, Java для бизнес-логики |
| Постепенное переписывание | Strangler Pattern: заменяем монолит по частям, не останавливая бизнес |

**Минусы микросервисов (цена, которую платим):**

| Минус | Почему критично |
|-------|----------------|
| Распределённые системы | Сетевые задержки, распределённые транзакции, Service Discovery |
| Операционная сложность | 10–20 сервисов × (CI/CD + мониторинг + трейсинг) |
| Сложность тестирования | Интеграционные тесты требуют поднять несколько сервисов |
| Консистентность данных | Нет ACID — нужны саги, итоговая согласованность |

---

### Когда что выбирать

```
Монолит → Модульный монолит → Микросервисы
  (MVP)       (рост команды)    (зрелый продукт)
```

**Выбирать монолит когда:**
- MVP, нужно быстро выйти на рынок
- Команда до 10–15 разработчиков
- Нет предпосылок к highload
- Строгие требования к консистентности (финансы, инвентарь)

**Выбирать микросервисы когда:**
- Несколько команд работают независимо
- Ожидаются пиковые нагрузки на отдельные компоненты
- Есть DevOps-зрелость для поддержки сложности
- Продукт прошёл stage 1, домены понятны

## 4. Глубже — что важно знать

**Strangler Fig Pattern** — безопасная декомпозиция:
1. Новый функционал пишется как отдельный сервис
2. Постепенно вырезаем куски монолита, перенаправляя трафик
3. Монолит «задушён» новыми сервисами, как лиана душит дерево

**Модульный монолит** — золотая середина: единый деплой, но чёткие границы между модулями (пакеты, интерфейсы без циклических зависимостей). При необходимости модуль легко вырезать в отдельный сервис.

**Закон Конвея:** структура системы повторяет структуру команд. В монолите команды конкурируют за общую кодовую базу. В микросервисах — каждая команда владеет своим сервисом.

**Распределённые проблемы, которых нет в монолите:**
- Вызов функции (μs) → HTTP-запрос (ms) → задержка ×1000
- Одна транзакция → сага с компенсирующими шагами
- Один лог → распределённый трейсинг (Jaeger, Zipkin)
- Один config file → Config Server + Service Discovery

## 6. Ответ на собесе (2 минуты)

Начну с того, что нет правильного ответа — монолит и микросервисы это разные компромиссы, и выбор зависит от контекста.

Монолит хорош на старте: один репозиторий, ACID транзакции, простой деплой. Команда из 5 человек выпускает MVP за месяц. Но когда команд становится 5 и каждая хочет релизиться независимо — монолит начинает мешать. Релиз одной команды ломает другую. Масштабировать можно только целиком.

Микросервисы решают это: каждая команда — свой сервис, своя БД, свой ритм релизов. Можно масштабировать только нагруженный сервис. Изоляция отказов через Circuit Breaker. Но за это платим: вместо вызова функции — HTTP через сеть, вместо одной транзакции — сага, вместо одного лога — распределённый трейсинг. Это операционная зрелость, которой нет на старте.

Мой подход: начинаю с модульного монолита с чёткими границами между модулями. Когда появляется явная необходимость — вырезаю модуль в сервис через Strangler Pattern, не останавливая бизнес. Никогда не начинаю с микросервисов на пустом месте.

## Шпаргалка

| Критерий | Монолит | Микросервисы |
|----------|---------|-------------|
| Стадия | MVP, start | Зрелый продукт |
| Команда | До 15 человек | Несколько независимых команд |
| Деплой | Один артефакт | Независимо по сервисам |
| БД | Одна, общая | Своя на каждый сервис |
| Транзакции | ACID | Сага (итоговая согласованность) |
| Масштабирование | Только целиком | Отдельные сервисы |
| Сложность | Простая на старте | Операционная с первого дня |
