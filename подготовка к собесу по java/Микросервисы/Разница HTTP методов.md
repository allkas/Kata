# Разница HTTP методов (GET, POST, PUT, PATCH, DELETE)

## 1. Проблема — зачем это существует?

HTTP-методы описывают намерение запроса — что клиент хочет сделать с ресурсом. Без семантики методов сервер не знает, безопасно ли кэшировать ответ, можно ли повторить запрос при сбое сети, является ли операция идемпотентной. Браузеры, CDN, прокси и клиентские библиотеки используют эту семантику для оптимизаций и безопасных повторов.

## 2. Аналогия

HTTP-методы — как стандартные действия с документом в офисе:
- **GET** — прочитать документ (никаких изменений)
- **POST** — создать новый документ
- **PUT** — полностью заменить документ другим
- **PATCH** — исправить отдельные строки в документе
- **DELETE** — уничтожить документ

## 3. Как работает

### Таблица всех методов

| Метод       | Действие           | Идемпотентный | Безопасный | Тело запроса |
| ----------- | ------------------ | :-----------: | :--------: | :----------: |
| **GET**     | Получить ресурс    |       ✅       |     ✅      |      —       |
| **POST**    | Создать ресурс     |       ❌       |     ❌      |      ✅       |
| **PUT**     | Полностью заменить |       ✅       |     ❌      |      ✅       |
| **PATCH**   | Частично обновить  |      ⚠️       |     ❌      |      ✅       |
| **DELETE**  | Удалить ресурс     |       ✅       |     ❌      |      —       |
| **HEAD**    | Получить заголовки |       ✅       |     ✅      |      —       |
| **OPTIONS** | Узнать возможности |       ✅       |     ✅      |      —       |

> **Безопасный** = не изменяет состояние сервера (GET, HEAD, OPTIONS).
> **Идемпотентный** = повторный вызов с теми же данными даёт тот же результат.

---

### GET — получение данных

```http
GET /users/123
GET /products?category=books&page=2
GET /search?q=kafka
```

- Данные передаются в **URL** (query string) — видны в логах, истории браузера, referer
- Кэшируется браузером, CDN, прокси
- **Никогда не передавать чувствительные данные** — пароль в URL попадёт в логи Nginx, историю, может уйти в Referer header при переходе на другой сайт

---

### POST — создание ресурса

```http
POST /users
Content-Type: application/json
{"name": "Alice", "email": "alice@mail.com"}
```

- Данные в **теле запроса** — не попадают в стандартные логи, не кэшируются
- Сервер назначает ID → `201 Created` + `Location: /users/456`
- **Не идемпотентен**: два одинаковых POST создадут двух пользователей

**Почему чувствительные данные — только через POST:**

```
GET /login?password=Secret → в логах Nginx: "GET /login?password=Secret HTTP/1.1 200"
POST /login {password: Secret} → в логах: "POST /login HTTP/1.1 200"  (тела нет)
```

Пять мест, куда утечёт пароль через GET: история браузера, логи сервера, логи балансировщика, Referer header при переходе, корпоративный прокси.

---

### PUT — полная замена

```http
PUT /users/123
{"name": "Alice Smith", "email": "alice@mail.com", "age": 30}
```

- Заменяет ресурс целиком: поля, не переданные в теле, **удаляются**
- Идемпотентен: второй PUT с теми же данными — тот же результат
- Если ресурс не существует — создаёт (`201`); если существует — заменяет (`200` или `204`)

---

### PATCH — частичное обновление

```http
PATCH /users/123
{"age": 31}          ← только изменённые поля
```

- Незатронутые поля **сохраняются**
- Идемпотентность **опциональна**: `{"age": 31}` — идемпотентно; `{"op": "increment"}` — нет

---

### DELETE — удаление

```http
DELETE /users/123    → 204 No Content (первый вызов)
DELETE /users/123    → 404 Not Found (повторный вызов)
```

Идемпотентен: конечное состояние одинаково (ресурса нет). Статус-код может отличаться (204 vs 404) — это нормально.

## 4. Глубже — что важно знать

**PUT vs POST для создания:**

| Сценарий | Метод | Пример |
|----------|-------|--------|
| ID назначает сервер | POST | `POST /users` |
| ID известен клиенту | PUT | `PUT /users/alice@mail.com` |
| Нужна идемпотентность создания | PUT | повторный PUT не создаст дубль |

**HTTPS не спасает GET:** HTTPS шифрует трафик в сети, но после расшифровки на сервере URL-параметры оказываются в логах. Разница не в шифровании, а в том, куда попадают данные после обработки запроса.

**Частая ошибка:** использовать POST для получения данных (например, поиск с большим фильтром). Это ломает кэширование и идемпотентность. Лучше передавать фильтры через GET query string или использовать специальный endpoint.

## 6. Ответ на собесе (2 минуты)

HTTP-методы описывают намерение: GET — читаем, POST — создаём, PUT — полностью заменяем, PATCH — частично обновляем, DELETE — удаляем.

Ключевые свойства — идемпотентность и безопасность. GET, PUT, DELETE идемпотентны: повторный вызов даёт тот же результат. GET и HEAD ещё и безопасны: не меняют состояние сервера. POST — ни то, ни другое. PATCH — идемпотентен только при правильной реализации.

Почему чувствительные данные через POST, а не GET? GET передаёт данные в URL — они попадают в логи Nginx, историю браузера, Referer header при переходе. HTTPS здесь не помогает: оно шифрует трафик в сети, но после расшифровки URL оказывается в логах.

PUT vs PATCH: PUT заменяет ресурс целиком, поля которые не передал — удаляются. PATCH обновляет только то, что указал. Поэтому для обновления одного поля — PATCH, для полной замены — PUT.
