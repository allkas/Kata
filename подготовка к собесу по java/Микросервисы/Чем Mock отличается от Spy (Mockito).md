# Чем Mock отличается от Spy (Mockito)

## 1. Проблема — зачем это существует?

При unit-тестировании нужно изолировать тестируемый класс от зависимостей. Но зависимости бывают разные: одни нужно полностью заменить заглушкой (база данных, HTTP-клиент), другие — протестировать почти реально, подменив лишь один метод. Mock и Spy решают эти два сценария: полная изоляция и частичный контроль.

## 2. Аналогия

Съёмки фильма: нужно протестировать сцену с опасным трюком.

**Mock** — нанять дублёра вместо актёра. Дублёр делает ровно то, что ты скажешь. Он не знает оригинальных реплик, не имеет истории персонажа. Полный контроль — ты решаешь каждое действие.

**Spy** — попросить реального актёра сыграть себя, но переснять одну конкретную фразу. Все остальные сцены — настоящие. Только одна реплика подменена.

## 3. Как работает

### Mock — полностью поддельный объект

```java
// Создание
UserService mock = mock(UserService.class);   // программно
@Mock UserService mock;                        // через аннотацию

// Все методы — заглушки, возвращают дефолтные значения:
// null для объектов, 0 для чисел, false для boolean, пустой список для List
mock.findUser(1);   // вернёт null без вызова реального кода

// Настройка поведения
when(mock.findUser(1)).thenReturn(new User("Alice"));
```

**Характеристики Mock:**

| | |
|---|---|
| Реальное поведение | Нет — всё замоканo |
| Изоляция | Полная |
| Синтаксис настройки | `when(mock.method()).thenReturn(value)` |
| Когда использовать | Внешние зависимости, БД, HTTP-клиенты, сложные объекты |

---

### Spy — обёртка над реальным объектом

```java
// Создание — требует реальный объект!
UserService spy = spy(new UserService());     // программно
@Spy UserService spy = new UserService();     // через аннотацию

// По умолчанию — всё реальное
spy.findUser(1);   // РЕАЛЬНО вызовет findUser из UserService

// Переопределить только один метод
doReturn(new User("Alice")).when(spy).findUser(1);   // findUser НЕ вызовется
```

**Характеристики Spy:**

| | |
|---|---|
| Реальное поведение | Да — по умолчанию вызывает реальные методы |
| Изоляция | Частичная |
| Синтаксис настройки | `doReturn(value).when(spy).method()` |
| Когда использовать | Легаси код, частичный контроль, проверка вызовов на реальном объекте |

---

### Критичный нюанс: `when()` vs `doReturn()` для Spy

```java
// ОПАСНО с Spy: findUser(1) ВЫЗОВЕТСЯ реально перед моком!
when(spy.findUser(1)).thenReturn(user);   // реальный вызов происходит здесь ↑

// БЕЗОПАСНО: метод не вызывается
doReturn(user).when(spy).findUser(1);    // ← правильный синтаксис для Spy
```

`when(spy.method())` сначала вызывает реальный метод для получения значения, потом перехватывает. Если реальный метод бросает исключение или имеет side effects — тест сломается. `doReturn().when()` не вызывает реальный метод совсем.

## 4. Глубже — что важно знать

**Сравнительная таблица:**

| Характеристика | Mock | Spy |
|---|---|---|
| Создаётся из | Класса или интерфейса | Реального объекта (требует `new`) |
| Реальное поведение | Нет | Да (если не переопределено) |
| Изоляция | Полная | Частичная |
| Синтаксис настройки | `when().thenReturn()` | `doReturn().when()` |
| Риск | Переизоляция (тест не отражает реальность) | Побочные эффекты реальных методов |
| Предпочтение | Чаще всего | Только когда реально нужно |

**Когда Mock:**
- Тестируемый класс зависит от БД, HTTP-клиента, файловой системы
- Реальный объект сложно создать (10 зависимостей в конструкторе)
- Нужна полная предсказуемость поведения

**Когда Spy:**
- Тестируем реальную логику класса, но один метод нужно подменить (например, `sendEmail()`)
- Тестирование легаси кода — не хочется переписывать, нужно добавить тесты
- Проверяем, что конкретный метод реально вызвался на реальном объекте

**Правило большого пальца:** предпочитать Mock. Spy использовать только когда реальное поведение действительно нужно. Spy в тестах — сигнал, что код, возможно, плохо структурирован и зависимости не вынесены через интерфейсы.

## 6. Ответ на собесе (2 минуты)

Mock и Spy — два типа тестовых двойников в Mockito. Разница в степени изоляции.

Mock — полностью поддельный объект. Все методы — заглушки, возвращают null/0/false по умолчанию. Я полностью контролирую поведение через `when().thenReturn()`. Использую для зависимостей, которые нужно изолировать: базы данных, HTTP-клиенты, внешние сервисы.

Spy — обёртка над реальным объектом. По умолчанию все методы работают реально. Можно переопределить отдельные методы. Важный нюанс с синтаксисом: для Spy нужно использовать `doReturn().when(spy).method()`, а не `when(spy.method()).thenReturn()` — иначе реальный метод вызовется до того, как Mockito перехватит вызов.

Когда использую Spy: тестирую реальную логику класса, но хочу заглушить один метод — например, `sendEmail()`, чтобы не слать реальные письма. Или тестирование легаси-кода, где зависимости не вынесены через интерфейсы.

Моё правило: предпочитаю Mock. Если тест требует Spy — это часто сигнал, что код плохо структурирован.

## Шпаргалка

| | Mock | Spy |
|---|---|---|
| **Аналогия** | Дублёр актёра | Актёр с правкой одной реплики |
| **Создание** | `mock(Class.class)` | `spy(new Real())` |
| **Реальное поведение** | Нет | Да (по умолчанию) |
| **Синтаксис** | `when().thenReturn()` | `doReturn().when()` |
| **Изоляция** | Полная | Частичная |
| **Когда** | Всегда по умолчанию | Легаси, частичная заглушка |
